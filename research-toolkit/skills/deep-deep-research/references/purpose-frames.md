# Purpose Frames

> **Decision-frame catalog, routed by question type — the input for the
> purpose-fit lever's *multi-frame* mode.** Use this *only* when no single
> decision purpose is confidently inferable. Pick the matching question type,
> then present **that type's frames separately and evenly** — one decisive
> set per frame. **Never merge the frames into one union list:** merging
> re-creates the 360° dump that this lever exists to escape (the reader is
> back to re-ranking a balanced pile themselves). When a purpose *is*
> confidently inferable, skip this catalog and foreground that one frame's
> decisive findings (consolidated mode).

## How to use

1. **Route.** Classify the question's type → pick the frame family below.
   If it isn't covered, use the **Generic fallback**.
2. **Present each frame separately, evenly.** For each frame, surface the
   findings that are *decisive* for that frame's decision (see each frame's
   decisive-finding kind). Keep frames as distinct sub-sections.
3. **Anti-collapse guard (N ≤ 4).** Each type lists **at most 4** frames.
   More than ~4 degenerates back into a 360° dump — if a fifth feels needed,
   fold it into an existing frame rather than adding a slot.

Each frame has the shape: `name — decision it serves — what finding is
*decisive* for it`.

---

## Equity / single-stock (4 frames)

Route here for a single-name equity decision. Present these four frames
separately; do not blend them into one list.

- **Entry / growth** — *initiate or add a position for upside.* Decisive: valuation vs intrinsic value (margin of safety), moat durability, length and credibility of the growth runway.
- **Trim / exit a winner** — *protect gains on a position that has already run.* Decisive: monitorable degradation signals (the thesis starting to break), valuation stretch vs history/peers, durability of the demand that drove the run.
- **Income / hold-for-dividend** — *preserve income and capital from a held position.* Decisive: dividend payout coverage (FCF / earnings), solvency and capital buffer, specific threats to the dividend (cut risk, covenant pressure).
- **Position-size / risk** — *decide how much to hold, not whether to hold.* Decisive: idiosyncratic / concentration risk, tail and variance exposure, correlation with the rest of the book.

## Software adoption / migration (4 frames)

- **Velocity / ship-faster** — *will adopting this speed up delivery?* Decisive: evidence it actually shortens the delivery loop; whether the real bottleneck is process/organization rather than the architecture this would change.
- **Scale / performance** — *will it hold up under real load?* Decisive: real load thresholds (not synthetic benchmarks); where the constraint actually binds (CPU / IO / network / coordination).
- **Cost / maintenance-burden** — *what does it cost to run and keep running?* Decisive: ongoing ops / infra burden, the team-size threshold below which it isn't worth it, who carries the operational load.
- **Lock-in / reversibility** — *how hard is this to undo?* Decisive: migration cost in and out, how reversible the decision is (one-way vs two-way door), depth of platform / vendor dependency created.

## Generic fallback (use when the question type isn't covered above)

- **Upside / opportunity** — *what is the best-case gain and how attainable is it?* Decisive: size and credibility of the upside, what has to go right.
- **Downside / risk** — *what is the worst plausible outcome?* Decisive: the dominant failure modes, their severity, and any tail that dwarfs the central case.
- **Feasibility / execution** — *can this actually be done with what we have?* Decisive: hard prerequisites, resource/skill gaps, who or what could veto or block it.
- **Reversibility / commitment** — *how locked-in are we once we act?* Decisive: cost to undo, one-way vs two-way door, sunk commitment created.
