# 4DX Meta — Team XPS Evaluation

[English](README.md) | [日本語](README.ja.md) | **繁體中文**

> 用書本標準的 XPS（四個 component × max 1.0 = max 4.0）稽核團隊 4DX 執行品質，判定 band、點出最弱 component、給一條 fix priority。

## 何時觸發這個 skill

- 「稽核我們團隊的 4DX 執行品質」「幫我們團隊打 XPS 分數」
- 「我們 4DX 跑得怎麼樣？」「我們是真的在跑 4DX 還是裝的？」
- 跑了 6-12 週後的季中 / 季末 health check
- WIG 沒進度，想在判定 WIG 錯之前先稽核執行
- 外部教練接手「在做 4DX」的團隊，要先有 baseline

## 它做什麼（protocol 摘要）

Auditor / consultant 聲音 — 只給事實、分數、fix priority；不教練、不打氣：

1. **確認稽核對象** — 一次一個團隊，WIG 已述、上線週數已知（XPS 不跨團隊加總）
2. **Component 1 — Establishing a Cadence（0.0-1.0）** — held/expected × meeting-discipline multiplier；書標準 100%
3. **Component 2 — Fulfilling High-Impact Commitments（0.0-1.0）** — fulfillment × specificity multiplier；書標準 90%+
4. **Component 3 — Optimizing Lead-Measures（0.0-1.0）** — 五個 optimizing 問題 + scoreboard 健康度；書標準 90%+ weekly target
5. **Component 4 — Achieving Lag-Measure Results（0.0-1.0）** — actual / required，上限 1.0
6. **加總 + 判 band** — 3.6-4.0 excellence / 3.2-3.59 good / 2.5-3.19 fair / 0-2.49 significant concern
7. **點出 fix priority（1-3 條動作）** — 最低 component 先修；同分用 C1 > C2 > C3 > C4
8. **輸出 XPS report** — `Components → Total → Band → Weakest → Fix priority`

## 不要在這些情境使用

| 情境 | 改去 |
|---|---|
| solo / personal coach 模式 | `4dx-sustain-momentum-rescue` |
| 要跑這週的 WIG session（不是稽核過往） | `4dx-d4-cadence` |
| 「我們到底要不要用 4DX？」 | `4dx-meta-strategy-triage` |
| 跨團隊頭對頭排名 | XPS 是 intra-team self-diagnostic，不是 leaderboard |
| User 沒權限 / 沒意圖跟進結果 | 不要做 audit-without-action |
