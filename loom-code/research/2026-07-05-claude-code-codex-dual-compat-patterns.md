# How the industry ships one skill/plugin package for both Claude Code and Codex CLI

**Date:** 2026-07-05
**Trigger:** loom-code v0.23.1 (PR #496) fixed a Claude-Code-only gotcha (naming an `Agent()` dispatch call turns it into a mailbox-semantics background teammate). Confirmed via OpenAI's own Codex manual that Codex's subagent architecture has no equivalent mechanism. This research asks: how does the rest of the ecosystem actually solve dual Claude Code + Codex compatibility, especially for subagent dispatch?
**Method:** 5-angle web research fan-out (5 parallel subagents, each searching + fetching primary sources), cross-corroboration across angles, one direct WebFetch re-verification by the orchestrator of the single most load-bearing claim. Not a full 25-claim/3-vote mechanical adversarial quorum — given multiple independent agents converged on the same primary-source claims from different search angles, and the highest-stakes claim was independently re-verified, a full quorum pass was judged disproportionate for this conversational research task. This scoping choice is disclosed, not hidden.

## Bottom line

There **is** a real, named, cross-vendor standard for skill *packaging* — the **Agent Skills** format (SKILL.md), which Anthropic released openly on 2025-12-18 and which OpenAI's Codex CLI docs explicitly say Codex builds on too. But that standard **explicitly does not cover subagent/task dispatch** — every project that needs cross-host subagent dispatch solves it itself, and **loom-code's own existing architecture (one shared SKILL.md + a `codex-tools.md` host-specific tool-mapping reference) is exactly the pattern the most relevant prior art (`obra/superpowers`) already uses.** The one concrete, actionable gap: our `codex-tools.md`'s "Subagent dispatch" section is stale — it says "Assumed, not exercised," but Codex's real primitive is a native, stable `multi_agent` feature exposing `spawn_agent` / `wait_agent` / `close_agent`, enabled via `~/.codex/config.toml`'s `[features] multi_agent = true`.

## 1. The packaging layer is standardized; the dispatch layer is not

| Layer | Standardized? | Evidence |
|---|---|---|
| Skill file format (SKILL.md: YAML frontmatter + Markdown body) | **Yes** — "Agent Skills," agentskills.io / github.com/agentskills/agentskills | Anthropic released it as an open standard 2025-12-18 ([the-decoder.com](https://the-decoder.com/anthropic-publishes-agent-skills-as-an-open-standard-for-ai-platforms/)); Claude Code's own docs say skills "follow the Agent Skills open standard, which works across multiple AI tools" ([code.claude.com/docs/en/skills](https://code.claude.com/docs/en/skills)); OpenAI's Codex docs independently say Codex skills "build on the open agent skills standard (agentskills.io)" ([developers.openai.com/codex/skills](https://developers.openai.com/codex/skills)). Adopted by 20+ tools per the spec site (Cursor, Gemini CLI, GitHub Copilot, Goose, etc.) |
| Subagent/task dispatch | **No** | The spec itself: *"There is no mention of task delegation, subagent invocation, or how skills coordinate across distributed hosts... [this] remains an open design space"* ([agentskills.my/specification](https://agentskills.my/specification/)) |
| Plugin manifest (`plugin.json`) | **No official cross-gen tool** | Claude Code (`.claude-plugin/plugin.json`) and Codex (`.codex-plugin/plugin.json`) have overlapping-but-different schemas (Codex adds a marketing `interface` block; Claude Code has no such block). No Anthropic/OpenAI tool converts one to the other — every dual-manifest project hand-authors both. |
| Hooks | **Partially — 10 of ~29 events shared** | Codex has an official hooks system (`/hooks`, `hooks.json`) with ~10 lifecycle events; Claude Code has ~29. Shared subset: `SessionStart`, `PreToolUse`, `PostToolUse`, `PermissionRequest`, `PreCompact`, `PostCompact`, `UserPromptSubmit`, `SubagentStart`, `SubagentStop`, `Stop`. |
| Project instructions | **No native fallback; community convention exists** | Claude Code reads only `CLAUDE.md`; Codex reads only `AGENTS.md`; neither auto-loads the other ([open Anthropic GitHub issue #34235](https://github.com/anthropics/claude-code/issues/34235), unresolved). Community convention: put shared instructions in `AGENTS.md`, make `CLAUDE.md` a one-line `@AGENTS.md` import. |

## 2. How `obra/superpowers` — our own closest prior art — actually handles it

This is the most directly relevant finding, because loom-code's own `codex-tools.md` was already modeled on superpowers.

- **One shared, host-neutral `SKILL.md`.** `dispatching-parallel-agents/SKILL.md` never says "Claude," "Codex," "Task tool," or any host-specific noun — it uses generic prose ("Subagent (general-purpose)," "dispatch") ([raw file](https://raw.githubusercontent.com/obra/superpowers/main/skills/dispatching-parallel-agents/SKILL.md)).
- **A per-harness tool-mapping reference file — exactly our `codex-tools.md` pattern.** `skills/using-superpowers/references/codex-tools.md` translates the host-neutral prose into Codex's real primitives. Directly re-verified by fetching the raw file myself: *"Add to your Codex config (`~/.codex/config.toml`): `[features] multi_agent = true`. This enables `spawn_agent`, `wait_agent`, and `close_agent` for skills like dispatching-parallel-agents and subagent-driven-development."* Same pattern exists for `copilot-tools.md`, `gemini` mapping, etc.
- **It evolved as Codex's own capabilities evolved.** In Oct 2025 (before Codex had any subagent primitive), the author's own blog documents a cruder stopgap: *"Task tool with subagents → Tell the user that subagents aren't available in Codex yet and you'll do the work the subagent would do"* ([blog.fsck.com](https://blog.fsck.com/2025/10/27/skills-for-openai-codex/)). By mid-2026, once Codex shipped native `multi_agent`, the mapping file was rewritten to point at the real primitives (confirmed via RELEASE-NOTES.md v6.0.0: *"A tool reference per harness at skills/using-superpowers/references/, covering Claude Code, Codex, Copilot, Gemini, Pi, and Antigravity"*).
- **Codex's own dispatch verbs are `spawn_agent` / `wait_agent` / `close_agent`** — not a Claude-Code-style single `Agent()` call with an optional `name:` that silently changes blocking behavior. This structurally confirms our earlier finding: Codex has no "mailbox semantics" pitfall because its model is explicit-verb-based (spawn, then explicitly wait, then explicitly close) rather than one overloaded call whose behavior depends on an easy-to-miss extra parameter.

## 3. Other real dual-host projects (confirms this isn't a one-off pattern)

| Project | Pattern | Confidence |
|---|---|---|
| [`duyet/codex-claude-plugins`](https://github.com/duyet/codex-claude-plugins) | Literal dual-manifest: `.codex-plugin/plugin.json` beside `.claude-plugin/plugin.json` per plugin; two separate marketplace catalogs (`marketplace.json` for Claude, `.agents/plugins/marketplace.json` for Codex) pointing at the same plugin folder | High |
| [`avivsinai/skills-marketplace`](https://github.com/avivsinai/skills-marketplace) | Same dual-manifest requirement, generated from one `registry/plugins.json` source of truth | High |
| [`wshobson/agents`](https://github.com/wshobson/agents) | Single canonical `plugins/` source, compiled via `make generate` into **harness-native** artifacts for 5 hosts (Claude Code, Codex CLI, Cursor, Gemini CLI, GitHub Copilot) — explicitly *not* lowest-common-denominator. Its own `docs/harnesses.md` states the exact gap we found: *"Codex requires naming an agent in prose to delegate"* vs. Claude Code's native Task/Agent spawn tool | High |
| [`shinpr/sub-agents-skills`](https://github.com/shinpr/sub-agents-skills) | Agents defined once in Markdown with a `run-agent:` frontmatter field naming the target CLI; an external Python wrapper (`run_subagent.py`) resolves and translates to the actual host invocation (3-tier: explicit flag > frontmatter > auto-detect) | Medium |
| [`alirezarezvani/claude-skills`](https://github.com/alirezarezvani/claude-skills) | 345 skills authored once, a `convert.sh --tool all` script regenerates native formats for 9 hosts — but scoped to skill-*file-format* conversion only, does not address dispatch/orchestration semantics | Medium |
| [`openai/codex-plugin-cc`](https://github.com/openai/codex-plugin-cc) | Not symmetric — a Claude Code plugin that shells out to a locally-installed Codex CLI for review/delegation. Evidence that even OpenAI's own cross-tool work treats the two CLIs as separate processes calling each other, not as one package running natively inside both | High |

## 4. What this means for loom-code, concretely

1. **Our architecture choice (shared SKILL.md + `codex-tools.md` mapping file) is already the industry's leading pattern**, not something we improvised — it matches superpowers, the most mature prior art in this exact space.
2. **`codex-tools.md`'s "Subagent dispatch" section is now confirmed stale.** It currently says: *"Assumed (not exercised in this test)... most likely shape: Codex `Task`/`Agent` tool equivalent to Claude Code's `Agent(subagent_type, prompt)`."* That guess is wrong — Codex's real primitive is `multi_agent` (`spawn_agent`/`wait_agent`/`close_agent`), config-gated via `~/.codex/config.toml`. This section can now be updated to a **verified** entry instead of an assumed one.
3. **§A1 (the named-Agent gotcha) is correctly scoped as Claude-Code-only** — confirmed independently again here: Codex's explicit-verb model (spawn/wait/close) structurally can't have Claude Code's "one overloaded call, one easy-to-miss parameter flips blocking behavior" failure mode.
4. **loom-code's `agents/*.md` role-prompt files still have no confirmed Codex equivalent.** None of the 6 dual-host projects surveyed document shipping reusable custom-agent *definitions* as part of a plugin bundle the way loom-code does — Codex's own custom-agent mechanism (`~/.codex/agents/*.toml`) is a separate, user/session-level concept outside the plugin manifest schema. This remains an open gap, not addressed by anything found in this research.
5. **AGENTS.md**: not currently a live issue for loom-code (a plugin's `CLAUDE.md` isn't loaded as context anyway, per `code.claude.com/docs/en/plugins-reference` — plugins deliver instructions via skills, not CLAUDE.md), but worth knowing if loom-code ever ships a project-root convention doc.

## Caveats

- One claim (an alleged `apply_patch` ↔ `Edit`/`Write` alias mechanism in Codex) could not be verified — the cited GitHub file 404'd and the claim only surfaced via an AI search summary. Not included above; flagged as unconfirmed.
- "27+ agents adopted the standard" figures trace partly to SEO/content-mill sites amplifying the same claim without independent citation — the *existence* of the standard and its adoption by Claude Code + Codex specifically is independently confirmed by primary vendor docs; the exact adoption count is not verified beyond that.
- No official, joint Anthropic/OpenAI governance body for the Agent Skills spec was found — it's Apache-2.0/CC-BY on GitHub, "open to contributions," but no visible cross-company steering committee.
