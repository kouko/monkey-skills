# stakeholder-and-team-thinking

複数ステークホルダーの因果ループ図オーバーレイ（外向き）＋ メンタルモデル調和とリーダーシップ・エナジーポンピング（内向き）。同じメンタルモデル surfacing 機構を共有しつつ、目的が正反対の二つのプロトコル。

## こんなときに使う

- **ステークホルダー対立**：2〜4 グループが繰り返し衝突している（Engineering vs Product、Sales vs Operations、ベンダー vs 顧客）。妥協では解決しなかった場合。
- **チームのメンタルモデル乖離**：個人は優秀なのに集合的アウトプットが劣化。意思決定が遅くなり、コードレビューが険悪になる。
- **M&A 後のアライメント**：買収側と被買収側がロードマップや文化で噛み合わない。外向き → 内向きを順に実施する。

## 起動方法

```
/systems-thinking-toolkit:stakeholder
```

## 得られるもの

- すべてのステークホルダーの因果ループ図を有益に改善する **straddle ポリシー**（妥協の中間点ではなく、どちら側も当初提案しなかった第三の選択肢）。
- 名前付きの **メンタルモデル調整成果物**：メンバーごとの因果ループ図、S/O ラベルの食い違いと欠落ノードの記録、根本価値での収束交渉結果。
- **リーダーシップ・エナジー継続投入プラン**：週次以上で持続する実践（アクティブ・リスニング、価値の反復強化、共有体験の蓄積）。一回限りのオフサイトではない。

## 適用外（境界）

- **心理的安全性の代替にはならない**。発言を恐れる状態でメンタルモデルを surfacing しても、本物の表出ではなく形だけの同意が出るだけ。先に安全性を修復（Edmondson の研究を参照）。
- **ゼロサムゲームには不適**。固定予算の取り合い、レイオフ枠の配分、M&A の重複整理にはストラドルが必要とする余白がない。「賢明な政策」という言葉が権力行使を糊塗してしまう。
- 他に不適：単一意思決定者の問題、主要ステークホルダーが場にいない、短期タスクフォース、リモート／非同期ファーストのチーム（2002 年のエナジーポンプ比喩は共在を前提）、認知多様性を設計上求めるチーム（レッドチーム、監査）— ここでは摩擦が目的。
