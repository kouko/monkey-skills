# InnovAction!™ — Past Application (cases)

Verbatim worked applications from Sherwood, Chapter 10. Read before
running a perturbation pass; the chess example calibrates the
Martian-test density bar that the rest of the method depends on.

## Case 1 — Chess as the worked Martian-test example (Chapter 10)

Sherwood demonstrates the completeness bar by enumerating chess
features at the level a Martian could use to distinguish chess from
football, cards, or a board meeting:

- two players, alternating moves
- 8×8 board
- six piece types with distinct movement rules
- capture by displacement
- win by checkmate
- no random element

Density check: a context-free outsider can tell chess apart from its
nearest game-neighbors *and* from non-games (board meeting) using
this list alone. Once at this density, a perturbation like "what if
capture were probabilistic?" generates a substantively different game
(probably closer to backgammon or a wargame with hit-rolls). A
shallower list — "a game, with pieces, on a board" — does not
support useful perturbation: every perturbation lands on something
already present in some other board game.

**Lesson**: the Martian-test density bar is not literary completeness;
it is the threshold below which perturbation produces trivial
variants. Sherwood's heuristic: 20-40 features, each independently
checkable.

## Case 2 — Scenario-planning future generation (Chapter 10, f15 application)

Sherwood explicitly invokes InnovAction!™ as the generative engine
*for* scenario planning. The procedure:

1. Compile a Martian-test-density description of today's business
   context: regulations, demographics, supply chain, capital cost,
   customer behavior, distribution channels, switching cost,
   competitive landscape.
2. For each scenario candidate, perturb ONE feature dramatically.
   "Customer is now a regulator paying for prevention" is a
   dramatic perturbation of who-pays-whom. "GDP +5%" is not.
3. Elaborate the consequence chain: which other features must then
   change? Which lever settings does the new world require?
4. Stop at 3-4 plausible (not probable) futures. These become the
   canvas for reverse-engineering required lever settings via
   `strategy-lever-and-cascade` (sk08) 3×N scenario table.

**Lesson**: the discipline that distinguishes scenarios from
forecasts is the one-feature-at-a-time perturbation rule. Multi-
feature perturbation collapses back to blank-page brainstorming.

## Case 3 — Product/service innovation workshops (Chapter 10, framework f16)

Sherwood positions the method as suited to product innovation as
well as scenario work — list features of a known product/service,
perturb one feature, elaborate. The book does **not** name a
specific worked case (e.g. a toaster-or-microwave perturbation
example is not present in the verified units); the cross-context
claim rests on the chess-vs-scenarios pairing inside Chapter 10
plus Sherwood's own assertion that the method generalizes.

**Lesson + caveat**: the perturbation logic transfers from scenario
work to product work *in principle*, but the empirical track record
in the book is scenario-heavy. Treat product-perturbation
applications as "consistent with the method" rather than
"validated by Sherwood case evidence." Where TRIZ or morphological
analysis is already in place for product work, prefer those — see
the Boundary section of SKILL.md for prior-art credit.
