# Triage + 延續性大腦(prioritization-framework)

> 這份文件是 daily-brief 的**判斷核心**:把 fan-out 撈到的原始項目轉成「重要度排序」、決定「誰升上晨報焦點」、推論「哪些在等你」、標好信心、並跨日算出「自上次以來」的 delta。
> 排版原則:**能用表/規則講清楚的不寫成散文**。每條硬原則都附 WHY——因為這些規則違反時的後果(誤報、幻覺滾雪球、硬湊),正是這個 skill 存在要防的失敗模式。

---

## 0) 這份文件的自由度地圖(先讀)

不同段落該被當成不同剛性的規則來執行——這決定了你能不能「自己判斷」:

| 段落 | 自由度 | 為什麼 |
|---|---|---|
| §1 緊急度×相關性評分 | **中**(rubric + 判斷) | 訊號清楚,但加權與排序需要 case-by-case 判斷 |
| §2 動態焦點門檻 | **中高**(curation 判斷) | 「夠不夠格當焦點」是專家取捨,不是查表;但下限 0 / 軟上限不截斷是硬框 |
| §3 需回覆偵測 | **中**(啟發式 + 強制標信心) | 推論本身有彈性,但「必標信心」是硬規則 |
| §4 信心模型 | **中**(分級 + 同源判斷) | 分級清楚,但同源判定要識別「副本」需要判斷 |
| §5 continuity-diff（ID-join） | **低**(逐步精確) | JOIN key 對齊、四狀態判定、live 重驗順序是低自由度精確操作——錯一步整段延續性失真 |
| §6 量化 caveat | **低**（查表照標） | 固定陷阱清單,逐項照標,不發明 |

> 一句話:**焦點要「狠」要判斷(§2),延續性 diff 要「準」要照步驟(§5)。** 兩者自由度相反,別搞混。

## 1) 緊急度 × 相關性評分

產出物:對 fan-out 撈到的**每一筆**算出一個「重要度」,用於①晨報焦點候選排序、②完整事項表降序排序。

### 緊急度訊號(時間壓力 — 不處理會怎樣)

| 訊號 | 強度 | 判讀 |
|---|---|---|
| 今天就是 deadline / 到期日 | 🔴 高 | 今天不動就 miss |
| 已逾期(到期日 < 今天且未結) | 🔴 高 | 已經欠著,優先補 |
| 會卡住別人(別人的下一步等你這步) | 🔴 高 | 你的延遲在放大成別人的延遲 |
| CI 紅 / build 壞 / 部署被擋 | 🔴 高 | 阻塞整條線,且通常你是唯一能解的人 |
| 被 @ 直接提問且我未回 | 🟡 中高 | 有人在等回應,時鐘在走 |
| 未來 N 天內到期 | 🟡 中 | 需要排程準備,非今日必動 |
| 無明確時間壓力 | 🟢 低 | 知會性質 |

### 相關性訊號(這關我什麼事 — 我的涉入程度)

| 訊號 | 強度 | 判讀 |
|---|---|---|
| 我是 owner / assignee | 🔴 高 | 責任在我,沒人會替我做 |
| 被點名 / 被 @ / 被要求 review | 🔴 高 | 明確指向我 |
| 我是作者 / 發起人 | 🟡 中高 | 我推動的事,進度與我相關 |
| 我在收件群但非主要對象(cc / 頻道成員) | 🟢 低 | 知會,通常不需我動 |
| 自動通知 / 電子報 / 系統訊息 | ⬜ 排除候選 | 見 §6 灌水過濾 |

### 重要度合成(rubric,非公式)

- **重要度 ≈ 緊急度 × 相關性**,但**不是**做兩個分數相乘求一個小數——那是假精確。用象限直覺:

```
        相關性高 ──────────────────► 相關性低
緊急 高  │ ① 立刻(焦點候選)      │ ② 替別人留意(通常不升焦點)
緊急 低  │ ③ 排程/準備(你的事)   │ ④ 知會/可略(完整表收底)
```

- **象限①** = 焦點門檻的主要來源(見 §2)。
- **象限③** 進「進行中專案 / 未來 N 天」分段,不佔今日焦點(除非有今日後果)。
- **象限②④** 進完整事項表,但通常不升晨報焦點。
- 排序時:同象限內,先逾期 > 今天到期 > 卡住別人 > 高槓桿決策。

> **不發明數字**:不要產出「重要度 8.5/10」這種看似客觀的分數——緊急度與相關性都是**判讀**,給假小數會讓使用者誤以為有量測。用象限 + 序位即可。

## 2) 動態焦點門檻（核心 expert rule）

**焦點 = 跨過門檻的事,不是 top-N。** 這是整個 skill 最容易被做錯的地方,單獨拉出來講。

### 納入條件(任一成立才算焦點)

| 條件 | 說明 |
|---|---|
| 今天是 deadline / 到期日 | 今天不動就 miss |
| 會卡住別人(**今天/現在**) | 你不處理,別人**今天**停在原地。**要有「現在就卡」的證據**(對方明說在等、有今日交期)——只是「一個 open 的 review request / 指派,沒期限」**不算**焦點,歸「進行中專案」(避免把無期限的等待硬升成焦點 = padding)。 |
| 不處理今天有後果 | 錯過會議、過了視窗、機會消失 |
| 高槓桿決策 | 你今天拍板,後面一串才動得了。**例**:你不今天定資料口徑,團隊明天三張報表都卡著開不了工。 |

**只要任一成立 → 焦點。四個都不成立 → 不是焦點(下推到分段)。**

### 窗緣高重要度項目(別被 N 天硬切掉)

未來窗預設 `{FUTURE_N}=7` 天,但**重要的事不會剛好卡在第 7 天結束**。一件落在 `{FUTURE_N}+1 ~ +3` 天、但**需要提前準備**的高重要度項目(送件 deadline、績效面談、外部 demo、release),如果現在不開始準備就會趕,**仍應納入**,放進晨報「🔭 未來」段並標「(窗緣 ~MM-DD)」。

```
判定:落在 [{TODAY}+{FUTURE_N}+1, {TODAY}+{FUTURE_N}+3] 且「現在不準備會趕」
  → 納入未來段,標「窗緣」
純資訊性的遠期項(無需提前準備)→ 不必硬拉進來,留給完整表收底即可
```

WHY:硬切 N 天會把「第 8 天的送件、但今天就該動手」這種事漏掉——時間窗是搜尋邊界,不是「重要度」的邊界。但也別無限外擴(只給緊鄰窗緣 + 需準備的),否則未來段會被遠期雜訊稀釋。

### 數量規則(這是反直覺的部分)

```
焦點數 = 跨過門檻的件數(由事實決定,不由「想列幾件」決定)

下限 0  ┌─ 今天真的沒有跨門檻的事
        │  → 明說「今日無關鍵焦點,可專注深度工作」
        └─ 嚴禁從象限③④硬撈兩三件來「湊」一個焦點段

軟上限 ~5 ┌─ 跨門檻的事 > 5 件
          │  → 加橫幅「⚠️ 今日高負載:N 件」
          └─ 全部保留、不靜默截斷;低於門檻的才下推到分段
```

### 為什麼這樣設計(WHY — 雙產物化解張力)

President's Daily Brief 的紀律是「狠取捨」——客戶買的是分析不是功課,寧可少列也不灌水。但本 skill 的使用者要「零省略」——怕漏掉任何待辦。這兩者**直接衝突**。

化解靠**雙產物**,不是折衷:

| 產物 | 紀律 | 角色 |
|---|---|---|
| 晨報焦點段 | **狠** — 只放跨門檻的 | 60 秒讀回今天該動什麼 |
| 完整事項表 | **全** — 零省略,每筆都在 | 安全網,確認沒漏 |

所以焦點段可以放心地狠:被刷下焦點的事**沒有消失**,它在完整表裡。**焦點段的省略 ≠ 丟棄**,過濾只決定「誰被升上來」,不決定「誰被丟掉」。理解這點,才敢把焦點段做狠。

### 反模式(這段的 NEVER,附 WHY)

- 🚫 **硬湊焦點**:今天沒有跨門檻的事,卻從低優先項撈幾件填滿焦點段。
  → WHY:焦點段一旦摻水,使用者就學會「焦點段不可信,還是得自己看完整表」——直接廢掉晨報的價值(60 秒讀回今天)。**0 件是合法輸出,而且是誠實的輸出。**
- 🚫 **靜默截斷**:跨門檻的事有 8 件,只留「最重要的 5 件」悄悄砍掉 3 件。
  → WHY:被砍的那 3 件**也跨了門檻**(今天有後果),漏掉任何一件就是這個 skill 的核心失敗。高負載時要**標出來**(橫幅),不是藏起來。截斷與標示是相反的動作。
- 🚫 **把 top-N 當焦點**:固定「每天列 3 件焦點」。
  → WHY:今天可能 1 件、可能 6 件,事實量是浮動的。固定 N 在低負載日硬湊(踩第一個反模式)、在高負載日截斷(踩第二個)——兩頭都錯。

## 3) 「需回覆 / 需處理」偵測啟發式

判斷「這件在等我回應」——這**永遠是推論,不是事實**(平台沒有「正在等本人」這個欄位)。

### 啟發式訊號

| 平台 | 「在等你」的訊號 | 推論強度 |
|---|---|---|
| Slack | 對我直接提問的 @mention,且 thread 中我之後沒發言 | 中(可能已私下回) |
| Gmail | thread 最後一封來自他人、收件人含我、且問句/請求語氣 | 中 |
| Asana | 指派給我 + 到期日近/已過 + 未完成 | 中高(有結構訊號) |
| GitHub | PR 的 requested reviewers 含我 + 狀態 open + 我未 submit review | 高(平台有明確 review-request 狀態) |
| Gmail/Slack | 法務/合約/用印類「流程下一棒可能是我」 | 低(推論,常無直接訊號) |

### 硬原則:必標信心,絕不宣稱確定

```
每一筆「需回覆/需處理」項目都要帶信心標(§4):
  ✅ / ⚠️ / 推論

凡是靠「語氣/流程/沒看到我回覆」推出來的 → 一律「推論」,不可標 ✅。
晨報用詞:寫「推測等你」「可能在等你」,不寫「David 在等你回」這種斷言句。
```

### thread / 訊息串讀取失敗時的降級規則(實測踩過)

「最後一則不是我發的 → 還沒回」這個判定**需要可靠讀到完整 thread**。但實務上會踩到:Slack Group DM 的 `read_channel` / `read_thread` 回 **Internal Server Error**、Gmail thread 過長被截斷、權限不足讀不到全部回覆。讀不到完整串時:

```
能讀到完整 thread → 照「最後一則是誰發的」判,信心可到 ⚠️(單一來源事實)。
讀不到完整 thread → 降級:
  1. 用「搜尋結果的時序」推斷(你之後是否還有發言出現在結果裡)
  2. 信心**只能掛「推論」**,不可標 ⚠️/✅
  3. 在該筆的一句話 / caveat 標明「未逐則翻完整 thread,依時序推斷」
```

WHY:把「讀不到所以猜你沒回」當成事實報上去,正是 §誤報傳染性 要防的——讀取失敗是**你的盲區**,不是「對方在等你」的證據。降級成推論 + 標明依據,使用者才知道這條要自己再確認。

### 為什麼(WHY — 誤報的傳染性)

一筆誤報(把「其實已經回了/根本不需我動」的事報成「在等你」)的代價,不是這一筆錯——是**使用者開始不信任整份簡報**。一旦他發現晨報叫他回一封他早就回過的信,他下次就會把每一條都再查一遍,簡報就從「行動入口」退化成「又一個要查的地方」。

所以寧可標弱(「推論:可能在等你」)讓使用者自己判斷,也不要標強(「在等你回」)賭它對。**信心標是在管理信任,不是在裝飾。**

## 4) 信心模型

信心分級如下,加上一個 expert 破洞修補。

| 標記 | 條件 |
|---|---|
| ✅ 高 | 同一事在 **2+ 平台交叉印證**(真·獨立來源) |
| ⚠️ 中 | 僅單一平台來源(內容雖完整也要標) |
| 推論 | 由情境/語氣/流程推得(§3 的需回覆判斷多屬此類),須明講推論依據 |

### 同源獨立性破洞(expert anti-pattern,附 WHY)

「2+ 平台 = ✅ 高信心」這條,有個**會誤判的破洞**:跨平台不等於跨來源。

典型的「同一事實副本」鏈:

```
GitHub PR 開了
   └─► CI bot 自動發到 Slack #dev 頻道
          └─► Asana task 自動建立/連動
   結果:GitHub + Slack + Asana「三個平台都看到」
   但這是 同一個事實 的 3 份副本,不是 3 個獨立印證
```

其他常見同源鏈:
- Google Calendar 邀請 → Gmail 自動收到同一場會議的邀請信(1 件事,2 平台)。
- Asana 指派 → Email/Slack 通知(1 件事,2 平台)。
- 同一 PR 的 commit、review request、CI 狀態散在 GitHub 不同 API(同平台內也會重複計)。

**硬原則**:

```
判信心前,先問:這幾個「來源」是不是同一事實的自動副本?

是同源 → 當作「單一來源(⚠️)」,不因為「出現在 3 個地方」就灌成 ✅。
真獨立 → 才標 ✅(例:David 在 email 問合約 + 他在 Slack 另外催,
                  是兩個人為動作,才算交叉印證)。
```

**WHY**:把「平台數」當獨立性的 proxy 會系統性高估信心——而且偏偏高估在**自動化最密的事**上(PR、會議邀請這類最會 fan-out 到多平台的)。使用者最常依賴 ✅ 標記跳過自己查證,一個被同源灌出來的假 ✅,等於請他把沒驗過的事當已驗過。**識別副本、收斂成一筆、按真實獨立來源數標信心。**

### 預期管理:✅ 在這種生態下本來就稀有(別誤判成「撈得不夠」)

在重度自動化的工作環境裡,跨平台的「重複出現」**幾乎都是同源回聲**(Calendar↔Gmail 邀請、Asana/Slack↔Gmail 通知信、PR→Slack bot),依上面的硬原則收斂後,**一份晨報常態就是滿江紅 ⚠️、極少 ✅**。這是**正確且誠實的結果**,不是 skill 撈得不夠或漏了交叉印證。

```
真·✅ 需要「兩個獨立的人為動作」指向同一件事:
  例:David 在 email 問合約 + David 另外在 Slack 私訊催 → 兩個獨立人為觸發 = ✅
不是 ✅:同一封邀請信同時出現在 Calendar 與 Gmail(系統自動複製)。
```

實務建議:若整份幾乎全 ⚠️,**在晨報底部加一句**「本份多為單一來源:跨平台重複多屬同源回聲,已收斂、未灌 ✅」,主動向使用者解釋,避免他誤以為要再去別處找佐證。

## 5) 延續性分析（continuity-diff）— 跨日核心

這是 daily-brief 跨日的命脈:回答「跟昨天比,什麼新發生了、什麼還在等你、什麼結了」。分兩步:fan-out **前** load、triage **後** diff。

### 5a) continuity-load（fan-out 前）

```
1. 在輸出資料夾,定位「日期 < 今天的最近一份」 *_完整事項.csv
   （扁平日期前綴佈局 → 排序檔名取 < 今天的最大者）
2. 讀該 CSV 的「唯一識別碼」欄(= JOIN key)+ 焦點段 / 未來段內容
3. 分支:
   ├─ 找不到任何 prior  → 標「首份簡報,無對照」,跳過延續段(§5b 不執行)
   ├─ 最近一份是昨天     → 正常 diff,標「對照 06-02,1 天前」
   └─ 中間跳了天(週末/休假) → 讀最近一份,標「距上次 N 天」,照常 diff
```

> continuity-load 只讀**最近一份**,不累讀整個歷史台帳——昨日 CSV 已經是「截至昨天的狀態快照」,再往前讀是浪費且會引入更舊的幻覺。

### 5b) continuity-diff（triage 後）四狀態

今日撈到的項目，用**唯一識別碼 JOIN 昨日 CSV**,逐筆判定:

| 狀態 | 定義(昨 vs 今) | 晨報呈現 |
|---|---|---|
| 🆕 新發生 | 今有、昨 CSV 無此 ID | 「🆕 新發生」 |
| ⏳ 仍在等你 | 昨有、今 live 重驗仍 open/未回 | 「⏳ 仍在等你(已 N 天)」 |
| ✅ 已結 | 昨有、今 live 已 closed/已回/已消失 | 「✅ 已結」 |
| 🔄 狀態變化 | 昨有、今 live 狀態移動了(如 in-progress→review) | 「🔄 狀態變化:X→Y」 |

「已 N 天」= 今天 − 該 ID 第一次出現的日期(可從更早的台帳回推,或保守標「≥昨天」)。

### 硬原則(這份文件最重要的一條):延續狀態以 live 平台重驗為準,不信昨日文字

```
昨日 CSV 的角色 = 只提供「今天要去追哪些 ID」(JOIN key 清單)。
昨日 CSV 的「標題/狀態/一句話」文字 = 不可當事實採用。

每一筆延續項目的「現在狀態」,一律拿唯一識別碼去 live 平台重撈:
  PR#482 → 去 GitHub 查現在是不是 merged
  Asana task GID → 去 Asana 查現在 completed 沒
  email thread → 去 Gmail 查最後一封是不是我回的
然後用「重撈到的事實」判 🆕/⏳/✅/🔄,不是用昨日 CSV 寫了什麼判。
```

### 為什麼(WHY — 幻覺 carry-forward 會滾雪球)

如果相信昨日文字:**昨天若有一筆是幻覺(或當時判錯),今天會原封不動 carry-forward,明天再 carry-forward**——一筆錯誤會沿著台帳一路滾下去、越滾越「看起來確定」(因為它「連續出現了好幾天」,反而被 §4 誤判成高信心)。

ID-join 重驗切斷這條鏈:**昨日只告訴你「追這些 ID」,事實一律今天重新問 live 平台。** 昨天的幻覺項今天一重驗就會發現「live 平台根本沒這東西」→ 自動掉出,不會傳染到明天。

> 對照 §4 同源破洞:那個是「同一天內跨平台」的假印證;這個是「跨天」的假印證(連續出現)。兩個都是把「重複」誤當「證據」,解法同源——**回到真實來源重驗,不信轉述。**

### 反模式(這段的 NEVER)

- 🚫 **信昨日文字**:直接把昨日 CSV 的「⏳ 仍在等你」抄進今天。
  → WHY:見上,幻覺/過時狀態會滾雪球。昨日 CSV 是 ID 清單,不是事實來源。
- 🚫 **用 ID 以外的東西 JOIN**:用標題/摘要做模糊比對來配對昨今項目。
  → WHY:標題會被人改、會被你自己的摘要措辭影響,模糊比對會錯配(把 A 的更新算到 B 頭上)。**只用唯一識別碼**(PR URL、task GID、thread ID、message ts)精確 JOIN。
- 🚫 **首份/跳天當常態處理**:沒有 prior 卻硬產一個空的延續段,或跳天不標天數直接 diff。
  → WHY:首份無對照,延續段會是假的(把所有今日項都標🆕);跳天不標,使用者會誤以為只隔一天而看漏中間累積。**首份跳過、跳天明標 N 天。**

## 6) 量化 caveat 速查

常見量化陷阱母體,精簡為晨報情境最常踩的。凡在簡報裡出現數字,照表標 caveat——**不發明新陷阱,也不省略該標的**。

| 數字 | 陷阱 | 正確讀法 |
|---|---|---|
| 行事曆行程數 | 行程數 ≠ 實際出席;重疊時段加總 ≠ 實際時數 | 標「排程數」,重疊處點出需取捨 |
| 專案總任務數 | 全員任務數 ≠ 我的任務數 | 只 tally 指派給我的 |
| commit 數 | commit 數 ≠ 貢獻量(一個大重構可能 1 commit,瑣碎改動可能 20) | 不拿 commit 數當產出量化 |
| review / approval 量 | 量大但單件輕——審核是「一個動作」的記錄單位 | 與「高槓桿少量產出」分開講,別混為一談 |
| 「近期很多 X」 | 搜尋引擎近期/相關性偏誤,近幾天被過度代表 | Slack/搜尋類來源,近期權重要打折 |
| 收件匣項目數 | newsletter / 自動通知灌水 | 先過濾系統寄件者(見 §1 排除候選)再數 |

---

> 速記:**評分用象限不用假小數(§1)・焦點靠門檻不靠 N、0 件合法不硬湊(§2)・需回覆永遠標信心不斷言(§3)・同源副本不灌 ✅(§4)・延續性 ID-join + live 重驗、不信昨日文字(§5)・數字照表標 caveat(§6)。**
