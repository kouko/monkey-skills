## 個資法 §5 (蒐集原則)

蒐集 / 處理 / 利用 個資 必須符合 **特定目的** 且不得 **逾越** 必要範圍，
方法應與目的具 **正當合理關聯**。是 個資法 全法的 **比例原則** 條款 —
事實涉及 個資的「過度蒐集」/「目的外利用」/「fishing 蒐集」皆從 §5
切入。

| 構成要件 element | 1-line 白話 | 典型 反例 | 典型 carve-out |
|---|---|---|---|
| (a) 特定目的 | 蒐集前已 identify 一個明確的個資使用目的（不能「先蒐再說」） | 為「未來可能的行銷需求」蒐集 / 目的欄位填「業務所需」一句話 | §6 特種敏感個資例外（醫療 / 基因 / 性生活 / 健康檢查 / 犯罪前科 — 仍適用更嚴格的 §6 規則）/ 公務機關依法定職務 |
| (b) 必要範圍 | 蒐集的 欄位 / 數量 / 對象 應為達成 (a) 目的最低限度 | 為登入功能蒐集 GPS 定位 / 為門市寄 DM 蒐集 身分證字號 + 健保卡號 | 法律 / 命令 明文要求（e.g. 洗錢防制法的 KYC 欄位）/ 當事人 自主提供且符合目的 |
| (c) 不得逾越 (a)+(b) 範圍 | 後續 處理 / 利用 不得超出原始蒐集範圍 | 把行銷蒐集的 email 改作 信評查詢 / 把售後服務通訊紀錄賣給第三方 | §20 第 1 項 各款例外（法律明文 / 公共利益 / 學術研究 / 當事人同意 等）— 但「同意」需符合 §7 同意要件 |
| (d) 與蒐集目的 具正當合理關聯 | 蒐集 方法 / 來源 / 對象 不能與目的脫節 | 為招募蒐集 應徵者 社群帳號全公開內容 / 透過第三方 data broker 蒐集無 §9 告知關係的個資 | 公開可得且與目的相關（e.g. 公司負責人之 商業登記資訊）/ 當事人 自行公開 |

**典型 反例**（fact pattern 對應）：
- 「為提供登入功能蒐集 GPS」 — (b) 不該當（GPS 與 登入功能 無必要關聯）
- 「先把 email + 電話 + 地址蒐起來，未來可能會用到」 — (a) 不該當（無 特定目的）
- 「行銷活動蒐集的 email 拿來做 風險評分」 — (c) 不該當（逾越原始目的）

**典型 carve-out**：
- §6 特種敏感個資 — 不是 §5 的 carve-out，而是 **更嚴格** 的 lex specialis；事實涉及 §6 列舉欄位時優先 涵攝 §6
- §16 / §20 公務機關 / 非公務機關目的內利用 vs 目的外利用 — 「目的內」自動符合 §5 (c)；「目的外」必須符合 §16 / §20 各款例外
- 「當事人同意」(§7) — 是 §5 (c) 逾越限制的最常見 carve-out，但同意必須符合 §7 的「告知後同意」+ 「書面 / 電子書面」+「可隨時撤回」三要件，否則同意無效，§5 (c) 仍 不該當

## 個資法 §8 (告知義務)

非公務機關 **蒐集 個資前** 對當事人應主動告知；告知內容 mandatory 6+ 項
（v.s. 9 項列為 法務部 / PDPC 函釋擴充）。是「線上服務 / APP / 網站
隱私權政策」最常涵攝的條款。**注意**：§8 是「自當事人本人蒐集」的告知；
非自當事人本人蒐集 走 §9（告知時點 + 例外不同）。

| 構成要件 element | 1-line 白話 | 典型 反例 | 典型 carve-out |
|---|---|---|---|
| (a) 蒐集前 告知 | 告知時點必須在「蒐集」這個動作之前；事後補告知不符合 | 註冊完成後才彈出隱私權政策 / 在使用條款最後一句寫「點此即代表同意蒐集」 | §8 第 2 項 4 款例外（見 carve-out 區）— 例外時 免告知 而非「事後告知」 |
| (b) 告知 蒐集者 身分 | 告知 蒐集 個資的法人 / 自然人 名稱 + 聯絡方式 | 隱私權政策只寫「本平台」未寫法人全名 / 委託處理時只寫委託人未寫受託者 | 受託者：§4「視同委託人」— 受託者本身對當事人 免單獨告知（但委託人需在 §8 告知中揭露受託者範圍） |
| (c) 告知 蒐集 目的 | 告知一個或多個 特定目的（對應 §5(a)）；多目的須逐項列出 | 寫「為提供服務」一句話 / 用 法務部 公告之 個資特定目的代碼 但 未文字化 | 法律明文要求（依 §8 第 2 項 第 1 款）/ 公務機關 依職務 必要 |
| (d) 告知 個資 類別 | 告知 蒐集 哪幾類個資（identifying / contact / 特種 / 行為 等） | 只寫「必要個資」未列舉欄位 / 漏未告知 §6 特種敏感個資 蒐集範圍 | §8 第 2 項 第 2 款（基於公共利益）/ 第 3 款（法定義務）/ 第 4 款（當事人 自行公開）— 上述 4 款為 §8 第 2 項法定免告知例外 |
| (e) 告知 利用期間 / 地區 / 對象 / 方式 | 告知 4W：how long / where / who / how | 寫「依本公司營業需要」/ 漏未告知 跨境傳輸地區（涉 §21） | 同上 §8 第 2 項 4 款例外 |
| (f) 告知 當事人權利 | 告知 §3 權利：查詢 / 閱覽 / 製給複製本 / 補正 / 刪除 / 停止 蒐集處理利用 | 只寫「依法行使權利」未列舉 §3 7 項權利 / 漏未告知行使方式 + 聯絡窗口 | 同上 §8 第 2 項 4 款例外 |
| (g) 告知 拒絕之效果 | 告知 當事人 拒絕提供個資 對其權益之影響 | 全包式寫「拒絕提供將無法使用任何服務」/ 強迫綁定非必要欄位 | 同上 §8 第 2 項 4 款例外 |

**典型 反例**（fact pattern 對應）：
- 「註冊完成後才跳出隱私權政策請使用者同意」 — (a) 不該當（事後補告知）
- 「隱私權政策只寫『本平台依法蒐集您的個資』」 — (c)+(d)+(e) 全部 不該當（無 特定目的 / 類別 / 4W）
- 「拒絕提供 任一欄位 = 無法註冊」+ 「無 拒絕之具體影響說明」— (g) 不該當（且涉 §7 同意自願性瑕疵）

**典型 carve-out**（§8 第 2 項 4 款 法定免告知例外）：
- 第 1 款：依 法律 明文規定
- 第 2 款：基於 公共利益 為統計或學術研究
- 第 3 款：個資 取自 一般可得來源（已公開）
- 第 4 款：對當事人權益無侵害
- 注意：實務上 第 4 款 是 法務部 / PDPC 函釋密集區域 — 「無侵害」標準 fact-sensitive，⚠️ 涵攝結論偏 ⚠️ + research

## 個資法 §9 (告知例外，非自當事人蒐集時的告知時點調整)

當 個資 **非自當事人本人** 蒐集（e.g. 從 第三方 data broker / 公開
資料庫 / 委託他人蒐集），告知時點調整為「**處理或利用前**」（而非
§8 的「蒐集前」），並有 6 款例外免告知。**勿與 §8 混淆** — 涵攝時
需先判斷 蒐集來源 才能選對條款。

| 構成要件 element | 1-line 白話 | 典型 反例 | 典型 carve-out |
|---|---|---|---|
| (a) 自 當事人 以外蒐集 | 蒐集來源 不是 當事人 本人 提供（含 第三方 / 公開資料庫 / data broker / 委託他人蒐集） | 從 LinkedIn 公開資料 蒐集應徵者 / 從 dun & bradstreet 買 BD lead list / 從 客戶 介紹的「也可以聯絡」名單 蒐集 | 自 當事人 本人蒐集走 §8 而非 §9（注意是「適用 §8」而非「§9 不適用」） |
| (b) 利用 前 告知 | 告知時點 = 處理 / 利用 動作之前（晚於 §8 的「蒐集前」） | 已寄出 第一封行銷 email 後才告知 / 已將 第三方蒐集 個資匯入 CRM 才告知 | §9 第 2 項 6 款例外（見 carve-out 區）/ 已適用 §8 例外的 4 款 sub-set |
| (c) 告知 內容 比照 §8 | 告知項目 與 §8 (b)~(g) 相同 | 告知 但 缺 §8 (e) 利用期間 / 地區 / 對象 / 方式 / 缺 §8 (f) 當事人權利 | 同 §9 第 2 項 6 款例外 |

**典型 反例**：
- 「從 LinkedIn 公開資料 抓應徵者，未告知就 直接寄招募信」 — (b) 不該當（利用前未告知）
- 「客戶介紹的潛客 lead list，匯入 CRM 後 一週才寄告知信」 — (b) 不該當（已開始處理）

**典型 carve-out**（§9 第 2 項 6 款 法定免告知例外）：
- 第 1 款：依 法律 明文規定
- 第 2 款：基於 公共利益 為統計或 學術研究
- 第 3 款：個資 取自 一般可得來源（**注意**：「可得」≠「易得」— 公開且 任何人可合法取得；社群媒體公開貼文是否該當 fact-sensitive，⚠️）
- 第 4 款：當事人 自行公開
- 第 5 款：對當事人權益無侵害
- 第 6 款：當事人 不利之必要保護（e.g. 失蹤協尋 / 緊急醫療）
- 注意：第 3 款的「取自一般可得來源」是 §9 vs §8 carve-out 最大灰區 — LinkedIn / GitHub / 公開政府網站 公開資料是否該當「一般可得來源」需 PDPC 函釋 / 個案 fact 評估，多數情況 ⚠️

## 個資法 §27 (適當安全措施)

非公務機關 蒐集 / 處理 / 利用 個資 應採行 **適當之安全措施**，防止
個資 被竊取 / 竄改 / 毀損 / 滅失 / 洩漏。是個資外洩 incident response
+ 預防性 compliance 的 核心條款。**注意**：「適當」是 不確定法律概念
— PDPC 函釋 + 主管機關 guideline + 同業 ISMS 標準 共同形塑門檻；
涵攝結論 多數 ⚠️ 並 handoff 至 research。

| 構成要件 element | 1-line 白話 | 典型 反例 | 典型 carve-out |
|---|---|---|---|
| (a) 蒐集 / 處理 / 利用 個資 者 | 義務人 = 任何 蒐集 / 處理 / 利用 個資 的非公務機關（含 受託者 per §4） | 主張「公司只是 把資料丟給 SaaS，不是 蒐集者」— 不能規避（仍是 利用者） | 純 個人 / 家庭 活動之蒐集（§51 第 1 項 第 1 款）/ 純非自動化 + 未列入 §6 之蒐集（爭議性 carve-out） |
| (b) 採行 (動作要件) | 須有 具體實際 落實 的安全措施，非僅紙上 policy | 寫了 ISMS policy 但 從未實施 / 口頭承諾「資安做得很好」/ 取得 ISO 27001 證書但 控制項未對應 個資場景 | 不可抗力（天災 / 駭客 unprecedented zero-day）— 但 仍須舉證 已採行符合 同業基準 之措施 |
| (c) 適當 (品質要件) ⚠️ | 「適當」= 與 個資 規模 / 敏感度 / 風險 相當的 **技術 + 組織** 措施 | 純技術措施無組織配套（e.g. 有 加密但 無 access control 規範）/ 純口頭規範無技術落實 / 措施程度 顯不相當（e.g. 大量 §6 特種個資 + 未加密儲存） | ⚠️ 「適當」門檻 待 PDPC 函釋 + 主管機關 guideline 細化（個資法 施行細則 §12 列舉 11 款建議措施 — 採行 11 款全部 ≠ 一定適當；未全採 ≠ 一定不適當）。`(2025/11 待 PDPC 子法)` |

**典型 反例**：
- 「ISMS policy 文件齊全 但 員工 從未受訓 / 從未做 access review」 — (b) 不該當（紙上 policy）
- 「客戶大量資料以 plain text 存於 公開可達 之 S3 bucket」 — (c) 不該當（顯不相當）
- 「每月一次 phishing 演練，但 incident response plan 內未含 §12 即時通報流程」 — (c) ⚠️（部分 該當 / 部分缺漏）

**典型 carve-out**：
- §4 委託處理：**委託人 仍負 §27 全責**；受託者 同樣受 §27 拘束（雙重義務），但對當事人 損害賠償 由委託人首先承擔（§4「受託者 視為委託人」）— 委託人 不能用 「我們已委託專業 SaaS」 抗辯免除自己 §27 義務
- 不可抗力：天災 / 戰爭 / unprecedented 0-day — 須舉證 已採行 符合 同業基準 之 預防措施 + 事件發生 確 不可預見
- §51 第 1 項 個人 / 家庭 活動：純 私人通訊錄 不適用 §27（不是 carve-out，是 §27 不適用範圍）
- ⚠️ 「適當」門檻 涵攝 多數情形 ⚠️：建議 同步 emit `/legal-research --query="個資法 §27 適當安全措施 PDPC 函釋 同業基準"` handoff

## 個資法 §29 (損害賠償，非公務機關)

非公務機關 違反個資法 致 當事人 受損害 應負損害賠償；採 **過失推定**
責任（行為人應 證明 自己無故意或過失 方得免責，與 民法 §184 通常侵權
舉證責任相反）。**注意**：公務機關 損害賠償 走 §28（無過失責任 + 國家
賠償），不要混用條號。

| 構成要件 element | 1-line 白話 | 典型 反例 | 典型 carve-out |
|---|---|---|---|
| (a) 違反 個資法 | 行為違反 個資法 任一條款（§5 / §8 / §9 / §11 / §27 / §20 / §21 等皆可） | 「我們只是 政策寫錯，沒有實際 蒐集 / 利用」— 仍可能 違反 §8 告知義務 | 行為 屬 §8 第 2 項 / §9 第 2 項 / §20 第 1 項 各款 法定例外 — 不構成「違反」 |
| (b) 致 當事人 損害 | 損害包含 財產 損害 + 非財產 損害 (慰撫金，§29 第 2 項 + §28 第 2 項) | 「資料外洩了 但 當事人 還沒收到 詐騙簡訊」— 仍可能 主張 慰撫金（§28 準用）但 金額 fact-sensitive | 當事人 自行公開 該筆個資（§29 第 3 項 → §28 第 3 項免責）/ 損害 非由 個資 違法行為 引起 |
| (c) 因果關係 | 違法行為 與 損害 須有 相當因果關係 | 「外洩 → 當事人 之後收到 詐騙簡訊」— ⚠️ 因果關係 fact-sensitive（垃圾簡訊來源 多元，需證明 確由 本案外洩 流出）| 介入因素切斷因果（e.g. 當事人 在外洩前 已將 個資 公開於社群）/ 當事人 故意自損 |
| (d) 過失 推定（與 民法 §184 不同） | 受害人 不需 證明 行為人 有故意或過失；行為人 須 主動 證明 已盡相當注意義務 方免責 | 行為人 主張「我們不知道 個資外洩，沒有過失」— 不能單以 不知 抗辯，須證明 已採 §27 適當安全措施 + 監測機制 | 行為人 證明 已盡 §27 適當安全措施 + 損害 由 不可抗力 / 第三人故意 / 當事人 自身行為 引起 — §29 第 1 項但書（已盡相當注意義務不負賠償） |

**典型 反例**：
- 「外洩了，當事人 提告 民法 §184，主張我們有過失」— ⚠️ 應 涵攝 §29（特別法 優先 適用）+ 注意 過失推定 反轉舉證
- 「當事人 自己把 信用卡號 發在 公開 IG 後 被盜刷」— (b)+(c) 不該當（自行公開 + 因果切斷）
- 「我們已通報 §12，所以 §29 不適用」— ❌ 通報義務 ≠ 損害賠償 抗辯（§12 通報 + §29 賠償 為 獨立義務）

**典型 carve-out**：
- §29 第 1 項 但書：行為人 證明 其 無故意或過失 不負損害賠償 — 須積極舉證 已盡 §27 + 員工訓練 + 內部稽核 + 監測機制（被動「不知」不夠）
- §29 第 3 項 → §28 第 3 項：當事人 故意 / 過失 為 損害發生原因 — 法院 得減輕或免除賠償
- §28 vs §29 區別：公務機關 走 §28（無過失責任）/ 非公務機關 走 §29（過失推定）— 不要混用條號
- **公務機關 §28 第 4 項 民眾不負損害賠償義務** — 公務機關 違法 蒐集 / 利用 個資 致當事人 受損害，民眾（即 當事人）對 國家 不負反向賠償義務（這是 國家 對 民眾 的 單向責任）；事實涉及 公務機關 時 須改用 §28 而非 §29
- 團體訴訟（§32-§40）：當 同一原因事實 致 多數 當事人 受害，財團法人 / 公益社團 得受讓請求權提起團體訴訟 — 不是 carve-out，是 救濟管道

## 跨條文 涵攝注意事項

### §5 ↔ §8/§9 互動
- §5 (a) 特定目的 = §8 (c) / §9 (c) 告知 蒐集目的 — 兩條 必須一致；告知目的 與 實際蒐集目的 不一致 同時 違反 §5 + §8/§9
- 涵攝時 須 雙向檢核：(1) 隱私權政策 寫的目的 是否 符合 §5 (a) 必要性 / (2) 實際使用 是否 在 (a)~(c) 範圍內

### §27 ↔ §29 互動
- §27 義務違反 → §29 過失推定難以推翻（行為人 已主動違反 注意義務）
- 反向：已盡 §27 + 已通報 §12 → §29 第 1 項但書 抗辯 較有利

### §4 委託處理 ↔ §27 / §29 連動
- 委託人 對 當事人 全責（§4「受託者 視為 委託人」）— 委託人 不能 用「已委託專業 受託者」抗辯
- 受託者 對 委託人 之契約責任（民法）+ 對 當事人 之 §27 義務 並存
- 涵攝時 注意 不要把 委託人 / 受託者 套用 GDPR controller / processor 框架（責任分配規則 不同）

### 2025/11 修法 影響範圍（標記 ⚠️）
- §12 通報義務 已修但 通報時限 / 通報範圍 / 通報入口 待 PDPC 子法 — 涵攝 通報是否「即時」採 施行細則 §22「即時」safe default；不要 inline 24hr / 72hr
- §20-1 內部 稽核框架 promulgated 但 施行日期未定 — 涵攝 公司是否「定期 稽核」時 暫不適用 §20-1，但可參照 §27 適當安全措施 之 organisational 措施
- §21 跨境傳輸 子規則 未發布 — 涵攝 跨境 是否合法 須 case-by-case；handoff 至 research 確認 主管機關 是否已 公告 限制
- §47-§48 新行政罰 tier 施行日期未定 — 涵攝 罰則風險 採 pre-2025 tier safe default
- ⚠️ 任何 涉及 上述 4 處 主張 SHOULD 標記 `(2025/11 amendment: 待 PDPC 子法)` 並 emit research handoff

### §5 ↔ §6 特種敏感個資 lex specialis
- 事實涉及 §6 列舉欄位（醫療 / 基因 / 性生活 / 健康檢查 / 犯罪前科 / DNA / 病歷）— 優先 涵攝 §6（更嚴格 同意要件 + 例外更窄），§5 退為 一般原則
- 此 reference 未 詳細展開 §6 構成要件 — 事實涉及 §6 時 emit handoff `/legal-research --query="個資法 §6 特種敏感個資 [具體欄位] 蒐集要件"`
