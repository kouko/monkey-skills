# A/B Harness Protocol

How `skill-tasting` Phase 3 presents variants to a human and
captures their preference.

## Design constraints

The harness must:

1. **Blind label** — the user must not know which output is the
   baseline (current skill) vs which is a candidate variant.
   Position bias is real; revealing identity defeats the data.
2. **Compact display** — 3+ variants × multi-paragraph outputs is
   too much for terminal display. Truncate / summarize / paginate
   appropriately.
3. **Capture decision atomicity** — user picks once; reveal only
   after pick is locked. No second-chance "wait actually I meant".
4. **Preserve full data** — even if user picks A, the B/C outputs
   must be logged for future analysis (which dimensions failed?).

## Variant generation rules

For each round:

- **Generate 2-3 variants** (cap at 3; user judgment fatigue
  rises sharply beyond N=3 in side-by-side comparison literature)
- Each variant must explore a **named dimension** — explicitly
  state what's different (tone / structure / detail-level / etc.)
- Variants must be **distinguishable in output** — cosmetic
  rewords that produce near-identical outputs waste a round
- Variants generated by spawning an editor subagent with a feature-
  hat prompt; see SKILL.md §Phase 1

If the editor agent generates variants whose outputs are too
similar to baseline (similarity > 0.85 by Jaccard), regenerate
with explicit instruction "vary more aggressively along the
named dimension". Don't burden the user with non-distinct A/B.

## Random label assignment

For each test prompt's A/B harness:

```python
labels = ['A', 'B', 'C', 'D'][:len(variants)+1]  # +1 for baseline
shuffled = random.shuffle([baseline, *variants])
labeled = dict(zip(labels, shuffled))
# Now: which letter = baseline is random per prompt
```

Save the assignment to a side log (not shown to user) so the
preference log can correctly identify which letter the user picked
in terms of "baseline" vs "variant_N":

```jsonl
{"round": N, "prompt": M, "label_assignment": {"A": "variant_1", "B": "baseline", "C": "variant_2"}}
```

## Side-by-side display

For each test prompt, render all variants:

```
─────────────────────────────────────────────────────────────────
TEST PROMPT (#M of #N)
─────────────────────────────────────────────────────────────────
<the user prompt verbatim>

─────────────────────────────────────────────────────────────────
OUTPUT A
─────────────────────────────────────────────────────────────────
<output content; truncate if >500 lines, append "[truncated; full
output saved to <path>]">

─────────────────────────────────────────────────────────────────
OUTPUT B
─────────────────────────────────────────────────────────────────
<...>

─────────────────────────────────────────────────────────────────
OUTPUT C
─────────────────────────────────────────────────────────────────
<...>

─────────────────────────────────────────────────────────────────
PICK ONE:
  [a] Variant A
  [b] Variant B
  [c] Variant C
  [m] Multiple (specify which two: e.g., "a+c")
  [n] None — all worse than current state
  [r] Refine — different test prompt or different variants needed
─────────────────────────────────────────────────────────────────
```

For multi-prompt sessions, repeat for each prompt; capture per-
prompt picks; aggregate at end of session.

## Multi-evaluator extension (optional)

If 2+ humans review the same A/B independently:

1. Each evaluator is shown the same blind labels (same random
   assignment)
2. Each evaluator picks independently; do not show them others'
   picks until all have picked
3. Aggregate:
   - Unanimous (all pick same letter) → that's the verdict
   - Majority (≥⅔ pick same letter) → that's the verdict, log as
     "majority"
   - Tie or wide spread (<⅔ agreement) → ESCALATE — this is a
     human-to-human resolution, not a tasting decision

For 1-evaluator workflow (the common case), skip aggregation;
their pick is the verdict.

## Decision options (the 4 + 1)

The user has 5 possible responses. Each is meaningful:

| Pick | What it means | Verdict |
|---|---|---|
| **A** (or B / C, single) | Clear preference for that variant | ADOPT (or DROP if pick = baseline) |
| **Multiple** ("A+C") | More than one variant is preferred over baseline; no single winner | DEFER — record both as candidates for future rounds |
| **None** | All variants are worse than baseline | DROP — current state preserved; signal that variant generation may be too narrow |
| **Refine** | User wants different test prompt OR different variants generated | REFINE — re-run; no decision recorded |

The "Refine" option exists because not every round produces a
useful A/B. If the test prompt is wrong-shape or the variants are
all off-axis, force-picking would corrupt the preference log.

## Truncation rules

Long outputs (>500 lines, >5000 chars) are truncated for display:

- Show first ~200 lines
- Append "[... truncated; full output: <path>]"
- User can request to view full output before picking

Truncation must be **uniform across variants** — same truncation
length, same boundary (don't truncate A at 200 lines but B at 300).
Asymmetric truncation introduces bias.

## Atomicity / lock-in

Once the user submits a pick:

1. Save pick to preference log immediately (`preference_log.py`
   append operation)
2. Reveal which letter was the baseline / which were variants
3. Apply the picked variant's SKILL.md edit (if pick ≠ baseline)
4. `git commit` the change

If the user changes their mind after seeing the reveal, that's a
new round, not a retroactive edit. This is to keep the preference
log as honest signal — "what did the user pick *blind*", not
"what did the user pick *after seeing the trick*".

## Per-session log entry

After the session (all prompts × all variants × all evaluators
processed), the harness writes one session-level summary to the
preference log alongside the per-pick entries:

```jsonl
{"timestamp": "...", "skill": "...", "session_id": "...", "round": N, "prompts_count": M, "variants_count": K, "constitutional_rejected_count": R, "verdicts": {"adopt": A, "drop": D, "defer": F, "refine": E}}
```

This summary is the human-time-cost record — useful when
evaluating whether tasting investments are worth it.

## When the harness fails

| Failure | Cause | Mitigation |
|---|---|---|
| User can identify baseline by style alone | Variants too cosmetic | Re-generate with stronger dimension; widen |
| All variants picked equally over baseline | Baseline is genuinely worse on all axes user cares about | ADOPT the first picked; future rounds against new baseline |
| All variants worse on every prompt | Variant generation prompt is wrong | Refine generation prompt; or hand off to skill-creator-advance |
| User fatigue (round N+1 user "doesn't care") | Too many rounds in one session | Cap at 3-5 rounds per session; encourage breaks |
| Inter-evaluator disagreement >30% | Genuine taste split among evaluators | ESCALATE; resolve human-to-human; if irreconcilable, fork the skill (different versions for different evaluator preferences) |
