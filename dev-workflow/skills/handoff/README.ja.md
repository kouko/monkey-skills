# Handoff

[English](README.md) | **日本語** | [繁體中文](README.zh-TW.md)

> セッション終了時に 10 ブロックの HANDOFF ファイルを `.claude/handoffs/` に書き出す。
> 次回セッション開始時にそのファイルをロードし、すべての検証コマンドを
> 逐語実行してから再開する。コールド AI スタート、ゼロコンテキストロス。

---

## 概要 — このスキルは何をするか

クロスセッション引き継ぎスキル。作業セッションを終えて、次の AI エージェントが
チャット履歴やメモリに頼らず正確に続きから再開できるよう状態を保存したい場合に
使う。構造化 10 ブロック HANDOFF ファイルを `.claude/handoffs/` に書き出す。

2 つのモード：

- **Prepare モード（準備）** — セッション終了時。エージェントが Bash で
  `git rev-parse HEAD`、`git status --short`、`git log --oneline -5`、
  `claude --version` を実行し、その出力をすべて含む
  `.claude/handoffs/HANDOFF-YYYY-MM-DD-HHMMSS-<slug>.md` を Write ツールで
  作成する。`references/handoff-schema.md` に定義された 10 ブロックすべてを
  記述する。出力は精度優先——正確なファイルパス、完全なエラーメッセージ、
  ユーザーターンの逐語記録、実行可能な検証コマンド。

- **Resume モード（再開）** — セッション開始時。エージェントが
  `ls -t .claude/handoffs/ | head -1` で最新ファイルを特定し、全体を
  Read して、Block 9 の検証コマンドをすべて Bash で実行し、期待値と実際の
  出力を逐語で報告する。不一致があれば継続を拒否する。全検証がパスした後、
  状況・未完タスク・次のステップを 3〜5 箇条でまとめ（Synthesis-check）、
  ユーザーの確認を得てから作業に入る。

HANDOFF ファイルの想定読者は、前回の文脈をゼロから出発するコールド AI。
可読性より精度を優先する（技術パス、正確な識別子、完全なコマンド出力）。

## 使い分け — 姉妹スキル `recap` との違い

どちらも再オリエンテーションのためのスキルだが、対象状況が異なる：

| 状況 | 使うもの |
|---|---|
| **会話の途中**で流れを見失った（まだ同じセッション内） | `dev-workflow:recap`（L3 セッション内、チャット出力のみ） |
| **セッションを終わる**；後でコールド AI に再開させたい | `dev-workflow:handoff` prepare モード |
| **新しいセッションを始める**；前回の状態をロードしたい | `dev-workflow:handoff` resume モード |
| 離席後に開いたセッションに戻ってきた（組み込み） | Claude Code 組み込み `/recap`（away summary） |
| 並行エージェントへのタスク委譲 | OpenAI Agents SDK 等（別の問題） |

`dev-workflow:recap` は L3 — ファイルへの書き込みなし、セッション内の
ウォームな人間読者向け、7 ブロックのチャット出力。

`dev-workflow:handoff` は L2 — `.claude/handoffs/` にファイルを書き出し、
将来セッションのコールド AI 読者向け、再開前に実行可能な検証を強制する。

両スキルは共存し、補完関係にある。

## 呼び出しフレーズの例

### Prepare モード — セッション終了時に以下のいずれかを言う

- 「今日はここまで」
- 「引き継ぎ」
- 「状態を保存して」
- 「セッションを終わる」
- 「ここで終わりにする」
- 「進捗を記録して」
- 「引き継ぎファイルを書いて」
- 「今日の作業を保存」

### Resume モード — セッション開始時に以下のいずれかを言う

- 「前回の続きから」
- 「引き継ぎを読んで」
- 「再開して」
- 「前回どこまでやったか確認して」
- 「引き継ぎファイルを読んで再開」
- 「昨日の続きをやろう」

## `.gitignore` 推奨設定

`dev-workflow:handoff` は常に `.claude/handoffs/` に HANDOFF ファイルを
書き出す。それを git でコミットするかどうかはユーザーのポリシー選択による：

**オプトイン（コミットしない）** — `.gitignore` に追加：
```
.claude/handoffs/
```
HANDOFF ファイルに認証トークン・内部パス・プライベートなタスク詳細が
含まれる場合、または git 履歴にセッション成果物を残したくない場合に使う。

**オプトアウト（git 履歴に残す）** — `.claude/handoffs/` を `.gitignore`
に追加しない。HANDOFF ファイルはブランチとともに移動し、別マシンや
チームメンバーが `git checkout` して既知の状態から再開できる。

v0.1 はパス固定式：スキルは常に `.claude/handoffs/` に書き出し、
resume モードの検索コマンド（`ls -t .claude/handoffs/ | head -1`）も
そのパスに依存する。git トラッキングはユーザー中立——スキルは
`.gitignore` を追加・変更しない。

## v0.2 に保留していること

v0.1 から意図的に外したもの：

- **強制引き継ぎトリガー検出** — コンテキスト圧迫・長い無音・「じゃあ」
  などの暗黙的セッション終了シグナルを検知する機能。v0.1 は明示的な
  呼び出しが必要。
- **割り込みスナップショットブロック** — タスク中断（セッション終了でない）
  専用の Block 11。v0.1 は全ケースに既存の 10 ブロックスキーマを使う。
- **ツール別 init-prompt バリアント** — Codex / Cursor / Gemini は
  書き込みパスが異なる。v0.1 は Claude Code の Write ツールのみ対応。
  HANDOFF ファイルの読み込み（resume モード）は Markdown を読めるツールなら
  クロスツール動作する。
- **`test-prompts.json` + `trigger-eval.json` スキャフォールディング** —
  トリガー精度とスキーマ完全性の構造的評価基盤。実セッションでの
  ドッグフードでベースラインが出てから追加予定。
