# 改善（Kaizen）

## Core Philosophy

改善とは、大きな変革を一度に行うのではなく、小さな改善を
継続的に積み重ねることで、長期的に大きな成果を生む手法である。

「今日の仕事を昨日より少しだけ良くする」。
この姿勢が改善の本質であり、完璧を目指すのではなく、
昨日よりも良い状態を毎日つくることに価値がある。

プロセス駆動型のスキルであり、エージェントはユーザーと共に
6つのステップを順に進め、最小の改善を特定し実行する。

## When to Use

- ユーザーが「なんとなく非効率だ」と感じている
- 既存のプロセスに摩擦やムダがある
- 「大きく変えたい」と言いつつ、何から始めればいいか分からない
- 小さな改善を習慣化したい
- 開発フロー、会議運営、デプロイプロセスなどの最適化

使わない場面：
- ゼロから新しいものを作りたい場合（第一原理を使う）
- 本質的な目的を問い直したい場合（ikigai を使う）
- 選択肢を比較検討したい場合（弁証法を使う）
- 思考の前提を疑いたい場合（ソクラテス式問答を使う）
- 既存プロセスが根本的に破綻している場合（改善ではなく再設計が必要）

## Topic Discovery

**ユーザー入力：** スキルが起動される（具体的なテーマあり / なし）
**エージェントの行動：** 改善対象のプロセスを確定する。

- ユーザーの入力に具体的なプロセスがある → Method へ進む
- 入力が曖昧だが会話にコンテキストがある →
  「[プロセス]について話されていますね。これを改善の
  6ステップで分析してみましょうか？」
- コンテキストがない →
  「どのプロセスを改善しますか？日々の業務で
  摩擦やムダを感じている場面を教えてください。」

**制約：プロセスを確認する。解決策を先取りしない。**

## Method: 改善の6ステップ

対象プロセスを6つのステップで順に改善する。
各ステップで質問し、ユーザーの回答を受けて次に進む。

### ステップ1：現状把握（Grasp the Current State）

判断を交えず、今の状態をそのまま観察する。

質問例：
- 「[プロセス]の現在の流れを、ステップごとに教えてください」
- 「このプロセスにかかっている時間・コスト・人数は？」
- 「このプロセスが最後に変更されたのはいつですか？」

観察のポイント：
- 実際の作業フローを可視化する（理想ではなく現実を）
- 数値化できるものは数値化する（時間、頻度、回数）
- 「当たり前」と思っていることも記録する

### ステップ2：問題発見（Find the Problem）

摩擦やムダが発生している箇所を特定する。

質問例：
- 「このプロセスで一番ストレスを感じるのはどこですか？」
- 「手戻りや待ち時間が発生する箇所はありますか？」
- 「このプロセスについて、チームから不満の声はありますか？」

ムダの7分類（トヨタ生産方式より）：
1. **作りすぎのムダ** — 必要以上に作る
2. **待ちのムダ** — 承認待ち、レビュー待ち
3. **運搬のムダ** — 不要なデータ移動、ツール間の転記
4. **加工のムダ** — 過剰な手順、不要な作業
5. **在庫のムダ** — 未処理のバックログ、未読の通知
6. **動作のムダ** — コンテキストスイッチ、ツールの切り替え
7. **不良のムダ** — バグ、手戻り、やり直し

### ステップ3：根本原因（Root Cause）

なぜこの問題が存在するのかを掘り下げる。

質問例：
- 「なぜこの問題が起きていますか？」
- 「その原因のさらに原因は何ですか？」（5回繰り返す）

**5 Whys の適用：**

```
問題：コードレビューに平均3日かかる
なぜ？→ レビュアーが忙しくて後回しにする
なぜ？→ レビュー依頼が一括で来るため、まとまった時間が必要
なぜ？→ PR が大きすぎて1回で見れない
なぜ？→ 機能を全部実装してからPRを出す文化がある
なぜ？→ 「小さなPR」の基準がチームで合意されていない
根本原因：PRサイズの基準が未定義
```

注意：5回が目安であり、3回で根本に到達することもあれば、
7回必要なこともある。「もうこれ以上分解できない」が目安。

### ステップ4：改善案（Improvement Proposal）

根本原因に対する最小の改善を提案する。

質問例：
- 「この根本原因を解消するための、最も小さな変更は何ですか？」
- 「明日から始められる一歩は何ですか？」

改善案の原則：
- **最小**：一度に変えるのは1つだけ
- **可逆**：失敗しても元に戻せること
- **計測可能**：改善の効果を数値で確認できること
- **即時**：「いつかやる」ではなく「今週やる」

反例（改善ではなく変革になっている）：
- NG：「CI/CDパイプラインを全面刷新する」
- OK：「テストの実行時間を計測し、最も遅いテストを1つ特定する」

### ステップ5：実行と検証（Execute and Verify）

改善を実行し、結果を計測する。

質問例：
- 「改善を試した結果、何が変わりましたか？」
- 「数値で比較すると、ビフォー・アフターはどうですか？」
- 「予想外の副作用はありましたか？」

検証のフレームワーク：

| 項目 | ビフォー | アフター | 変化 |
|------|---------|---------|------|
| 所要時間 | ? | ? | ? |
| エラー率 | ? | ? | ? |
| 満足度 | ? | ? | ? |

結果に応じたアクション：
- **改善した** → ステップ6（標準化）へ
- **変化なし** → 根本原因の再分析（ステップ3に戻る）
- **悪化した** → 改善を元に戻し、別のアプローチを試す

### ステップ6：標準化（Standardize）

効果が確認された改善を新しいデフォルトにする。

質問例：
- 「この改善をチームの標準にするには何が必要ですか？」
- 「ドキュメント化、自動化、ルール化のどれが適切ですか？」
- 「次の改善サイクルで取り組みたい課題は何ですか？」

標準化の方法：
- **ドキュメント化**：手順書、チェックリスト、Wiki
- **自動化**：スクリプト、CI/CD、テンプレート
- **ルール化**：チーム規約、コーディング規約、レビュー基準
- **習慣化**：ふりかえり（retrospective）で定期的に確認

標準化した後は、次のサイクルへ。
改善は一度きりではなく、継続的に回すものである。

## Response Format

各ステップの分析時：
1. **ステップ名** — 明示的にステップ名を宣言する
2. **要約** — 前のステップで分かったことをまとめる（ステップ2以降）
3. **質問** — ユーザーのプロセスに適応した1〜2の質問
4. **待機** — ユーザーの回答を待つ（先に改善案を出さない）

ステップ4（改善案）では：
1. **根本原因の確認** — ステップ3の結果を要約する
2. **改善案** — 最小の改善を1〜3個提示する
3. **比較** — 「改善」と「変革」の違いを明示する
4. **選択** — ユーザーに選んでもらう


**Stop after each phase. Never advance without user input.**

## Critical Constraints

### 絶対にしないこと：
1. **大きな変革を提案しない** — 「全面刷新」は改善ではない
2. **ステップを飛ばさない** — 特にステップ1（現状把握）を省略しない
3. **根本原因の前に解決策を出さない** — 症状への対処療法を避ける
4. **完璧を求めない** — 「少しだけ良くなる」で十分
5. **ユーザーの現状を否定しない** — 現在のプロセスにも理由がある

### 必ずすること：
1. **小ささを維持する** — 改善は常に最小単位で
2. **数値で語る** — 定性的な「良くなった」ではなく定量的に
3. **サイクルを意識する** — 1回で終わらず、次の改善を示唆する
4. **現場の声を重視する** — プロセスを実際に使っている人の感覚を尊重する
5. **可逆性を確認する** — 失敗しても元に戻せることを確認する

## Safety Measures

- ユーザーが「全部壊してやり直したい」と感情的になっている場合、
  まず感情を受け止めてから、「全部壊す」と「一箇所ずつ直す」の
  メリット・デメリットを提示する
- プロセスの問題が人間関係に起因する場合（「あの人がボトルネック」）、
  個人攻撃にならないよう、プロセスの構造的な問題として再フレーミングする
- ユーザーが改善疲れを感じている場合、「改善しなくてもよい」という
  選択肢も提示する。現状維持も意図的な判断として尊重する
- 組織の権限構造上、ユーザーが変えられないプロセスがある場合、
  影響範囲内で可能な改善に集中する

## Examples

### Example 1: 開発ワークフローの改善

```
ユーザー：「PRレビューが遅くて開発が止まる」

エージェント：「改善の6ステップで分析しましょう。
ステップ1 — 現状把握。PRレビューの現在の流れを
教えてください。PRを出してからマージまで、
通常どのくらいかかりますか？」

ユーザー：「PRを出す → Slackでレビュー依頼 → レビュアーが
確認 → コメント → 修正 → 再レビュー → マージ。
平均3日、長いと1週間かかる。」

エージェント：「3日〜1週間。ステップ2 — 問題発見。
この流れの中で、一番待ち時間が長いのはどこですか？」

ユーザー：「PRを出してからレビュアーが見始めるまで。
忙しいと2-3日放置されることもある。」

エージェント：「レビュー開始までの待ち時間が最大のムダですね。
ステップ3 — 根本原因。なぜレビュアーが見始めるのが
遅いのですか？」

ユーザー：「レビュアーが2人しかいない。しかも他の作業で
忙しい。PRも大きくて1時間くらいかかる。」

エージェント：「もう一段掘ります。なぜPRが大きいのですか？」

ユーザー：「機能を全部作り終えてからPRを出す文化がある。
途中で出すと『まだ完成してないの？』と言われる。」

エージェント：「根本原因が見えてきました。
・レビュアーが少ない
・PRが大きい（全部作ってから出す文化）
・レビュー時間が確保されていない

ステップ4 — 改善案。最小の一歩として：
1. PRサイズの上限を設ける（例：変更300行以内）
2. 毎日30分の「レビュータイム」をチームで確保する
3. ドラフトPRを活用して早い段階でフィードバックを得る

全面刷新（レビュープロセスを根本から変える）ではなく、
まず1つだけ試してみませんか？どれが一番始めやすいですか？」

ユーザー：「PRサイズの上限が一番簡単そう。300行ルールで
やってみる。」

エージェント：「ステップ5 — 実行と検証。1〜2週間試して
結果を見ましょう。確認する数値：
- レビュー開始までの平均待ち時間（現在：2-3日）
- PR1件あたりのレビュー時間
- PRの平均サイズ（行数）

これらのビフォー・アフターを記録してください。
結果が出たら、ステップ6（標準化）に進みましょう。」
```

### Example 2: 会議の効率化

```
ユーザー：「週次ミーティングが長すぎて生産性が下がっている」

エージェント：「ステップ1 — 現状把握。現在の週次ミーティングの
構成を教えてください。時間、参加人数、議題は？」

ユーザー：「毎週月曜10:00-11:30、90分。チーム全員8人参加。
議題は各自の進捗報告、相談事項、お知らせ。」

エージェント：「90分 × 8人 = 12人時/週。ステップ2 — 問題発見。
90分のうち、一番ムダだと感じるのはどこですか？」

ユーザー：「進捗報告。8人が順番に話すだけで45分使う。
しかも、自分に関係ない報告を聞いている時間が長い。」

エージェント：「進捗報告に45分（全体の50%）。聞いている側は
受動的。ステップ3 — 根本原因。なぜ進捗報告を口頭で
やっていますか？」

ユーザー：「昔からそうだった。テキストで共有する仕組みがない。」

エージェント：「習慣の惰性ですね。ステップ4 — 改善案。
最小の改善として：

会議前にSlackで進捗を非同期共有する。
フォーマット：
- やったこと（3行以内）
- 困っていること（あれば）
- 来週やること（3行以内）

会議では「困っていること」だけを議論する。
報告は読めばわかるので口頭不要。

予想効果：90分 → 45分。
まずは来週1回だけ試してみませんか？」

ユーザー：「それならすぐ始められる。やってみる。」

エージェント：「ステップ5で確認する数値：
- 会議の実時間（現在：90分）
- 参加者の満足度（5段階で聞く）
- 「有意義だった」と感じた議題の数

来週の会議後に振り返りましょう。」
```

追加の事例は `references/kaizen-cases.md` を参照。

## Implementation Checklist

毎回のレスポンス前に確認：
- [ ] 現在のステップに集中しているか？先のステップに飛んでいないか？
- [ ] 改善案は「最小」か？大きな変革になっていないか？
- [ ] ユーザーの現状を否定していないか？
- [ ] 数値での計測を提案しているか？
- [ ] サイクルの継続を示唆しているか？（一回で終わらせない）
- [ ] 可逆性を確認しているか？（失敗しても戻せるか？）
