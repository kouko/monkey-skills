# 改善 — 追加の改善ケース

改善サイクルの参考として使用する。ユーザーのドメインに応じて適宜参照すること。

## デプロイメントパイプラインの改善

**現状把握：**
- デプロイ頻度：週1回（金曜日）
- デプロイ作業：手動で約2時間、担当者は1人（属人化）
- ロールバック：手動。過去3ヶ月で2回失敗し、深夜対応が発生

**問題発見：**
- ムダの種類：待ちのムダ（週1回に集約）、不良のムダ（ロールバック失敗）
- 最大の摩擦：金曜デプロイが失敗すると週末対応になる

**根本原因（5 Whys）：**
```
問題：デプロイが失敗する
なぜ？→ ステージング環境でのテストが不十分
なぜ？→ ステージング環境が本番と乖離している
なぜ？→ ステージング環境の更新が手動で、後回しにされる
なぜ？→ 環境構築の手順がドキュメント化されていない
なぜ？→ 最初に構築した人が退職し、知識が失われた
根本原因：環境構築手順の暗黙知化
```

**改善案（最小の一歩）：**
- 現在のステージング環境の構成を、そのまま README に書き出す（1日で完了）
- 全面的な IaC 化は改善ではなく変革。まずは記録することから始める

**検証指標：**
| 項目 | ビフォー | 目標 |
|------|---------|------|
| デプロイ失敗率 | 月2回 | 月1回以下 |
| ロールバック時間 | 45分 | 30分以下 |
| 属人化度 | 1人だけ実行可能 | 2人以上が実行可能 |

**標準化：**
README をチームの Wiki に移動し、デプロイ手順のチェックリストを作成。
次の改善サイクルでは、チェックリストの自動化を検討する。

## ドキュメント作成プロセスの改善

**現状把握：**
- 技術ドキュメントの作成に平均3日かかる
- レビュー → 修正 → 再レビューのサイクルが2-3回
- 「書き方が分からない」という声が多い

**問題発見：**
- ムダの種類：加工のムダ（毎回ゼロから書く）、不良のムダ（レビューでの大幅修正）
- 最大の摩擦：何をどの粒度で書けばいいか分からない

**根本原因（5 Whys）：**
```
問題：ドキュメント作成に3日かかる
なぜ？→ 何を書くか決めるのに時間がかかる
なぜ？→ ドキュメントの構成が毎回バラバラ
なぜ？→ テンプレートがない
なぜ？→ 「テンプレートを作ろう」と言いつつ誰も着手しない
なぜ？→ 理想的なテンプレートを作ろうとして、ハードルが上がっている
根本原因：完璧なテンプレートを目指して着手できない
```

**改善案（最小の一歩）：**
- 過去の「良かったドキュメント」を1つ選び、その構成をそのままテンプレートにする
- 「完璧なテンプレート」ではなく「とりあえずのテンプレート」で良い
- 30分で作れるレベルのものを今日中に

**検証指標：**
| 項目 | ビフォー | 目標 |
|------|---------|------|
| 作成時間 | 3日 | 2日以下 |
| レビュー修正回数 | 2-3回 | 1-2回 |
| 「書き方が分からない」の発生 | 毎回 | テンプレート参照で解決 |

**標準化：**
テンプレートを Git リポジトリの `docs/templates/` に配置。
新規ドキュメント作成時に必ず参照する運用を README に明記。
3ヶ月後にテンプレート自体を改善する（メタ改善）。

## オンボーディングプロセスの改善

**現状把握：**
- 新メンバーが最初のPRを出すまで平均2週間
- 環境構築に1-2日、ドメイン理解に3-5日
- メンター（先輩社員）が付くが、メンターの負荷が高い

**問題発見：**
- ムダの種類：待ちのムダ（メンターの空き時間待ち）、動作のムダ（環境構築の試行錯誤）
- 最大の摩擦：環境構築で詰まる。README が古い

**根本原因（5 Whys）：**
```
問題：環境構築に1-2日かかる
なぜ？→ README の手順通りにやってもエラーが出る
なぜ？→ README が最後に更新されたのが6ヶ月前
なぜ？→ 環境に変更があっても README を更新する習慣がない
なぜ？→ 更新の責任者が決まっていない
なぜ？→ 「みんなの仕事」は「誰の仕事でもない」
根本原因：README 更新の責任の所在が不明確
```

**改善案（最小の一歩）：**
- 次にオンボーディングする新メンバー自身に、環境構築しながら
  README を更新してもらう（「新人の目」が最も正確）
- 「環境構築 README の更新」をオンボーディングタスクの1つに正式に組み込む

**検証指標：**
| 項目 | ビフォー | 目標 |
|------|---------|------|
| 環境構築時間 | 1-2日 | 半日以内 |
| README の鮮度 | 6ヶ月前 | 最終オンボーディング時に更新済み |
| メンターへの環境構築質問 | 5-10回 | 2-3回 |

**標準化：**
オンボーディングチェックリストに「README 更新」を追加。
PR テンプレートに「環境構築手順に影響する変更ですか？」チェックボックスを追加。

## 改善サイクルの連鎖パターン

一つの改善が次の改善を呼ぶ。典型的な連鎖：

```
サイクル1：PRサイズの上限を300行に設定
  → レビュー待ち時間が3日→1日に短縮
  
サイクル2：小さなPRが増えたことで、レビューの質が向上
  → コメントが具体的になり、修正が1回で済むように
  
サイクル3：レビューが早くなったことで、デプロイ頻度が上がる
  → 週1回→週3回に増加
  
サイクル4：デプロイ頻度が上がり、1回のリスクが下がる
  → ロールバックが不要に近づく
```

**学び：** 最初の改善は小さくても、連鎖的に大きな変化を生む。
だからこそ「最小の改善」にこだわる価値がある。

## 改善が機能しないケース

改善アプローチが適さない状況も認識すること：

| 状況 | なぜ改善では不十分か | 代替アプローチ |
|------|-------------------|-------------|
| プロセス自体が根本的に間違っている | 最適化しても間違った方向に速く進むだけ | 第一原理（根本から再設計） |
| 目的が不明確 | 何を改善すべきか判断できない | 生き甲斐（目的の再定義） |
| 二者択一の決断が必要 | 漸進的改善では解決しない | 弁証法（対立の統合） |
| 組織全体の構造的問題 | 個人やチームレベルの改善では限界がある | 経営層への提言が必要 |
