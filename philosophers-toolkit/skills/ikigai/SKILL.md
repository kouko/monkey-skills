# 生き甲斐（Ikigai）

## Core Philosophy

生き甲斐とは「好きなこと」「得意なこと」「世界が求めること」
「報酬を得られること」の4つの軸が交差する点である。
この交差点にあるものが、持続可能な情熱と意義をもたらす。

4軸のうち1つでも欠ければ、何かが空虚になる。
好きだが稼げない。得意だが世界が求めていない。
この「欠け」を診断し、補完する道を見つけることが本メソッドの本質である。

フレームワーク分析型のスキルであり、エージェントはユーザーと共に
4つの軸を順に探索し、交差点を見つけ出す。

## When to Use

- ユーザーが「これは本当にやるべきことか？」と迷っている
- プロジェクトや製品の存在意義を問い直したい
- キャリアの転換期にあり、方向性を整理したい
- サイドプロジェクトやスタートアップのアイデアを検証したい
- Product-Market Fit を達成できていない感覚がある

使わない場面：
- 具体的な実装方法を知りたい場合（code-team を使う）
- 既存の選択肢を比較したい場合（弁証法を使う）
- 思考の前提を疑いたい場合（第一原理を使う）
- プロセスの改善をしたい場合（kaizen を使う）

## Topic Discovery

**ユーザー入力：** スキルが起動される（具体的なテーマあり / なし）
**エージェントの行動：** 分析対象を確定する。

- ユーザーの入力に具体的なテーマがある → Method へ進む
- 入力が曖昧だが会話にコンテキストがある →
  「[テーマ]について話されていますね。これを生き甲斐の4軸で
  分析してみましょうか？」
- コンテキストがない →
  「何を分析しますか？プロジェクト、キャリア、製品、
  アイデアなど、対象を教えてください。」

**制約：テーマを確認する。答えを先取りしない。**

## Method: 4軸分析

対象を4つの軸で順に分析する。各軸について1〜2の質問をし、
ユーザーの回答をもとに分析を深める。
すべての軸を終えた後、交差点を診断する。

### 軸1：好きなこと（What you love）

情熱と内発的動機の源泉。お金がなくても続けたいこと。

質問例：
- 「[対象]のどこに一番ワクワクしますか？」
- 「お金にならなくても、これを続けたいと思いますか？なぜ？」

ドメイン別の観点：
- ソフトウェア：解決したい技術課題、触りたい技術領域
- プロダクト：ユーザーの笑顔が見えるか、作る過程が楽しいか
- キャリア：日々の仕事で時間を忘れる瞬間はあるか
- サイドプロジェクト：週末に自発的に取り組むか

### 軸2：得意なこと（What you're good at）

スキル、強み、専門性。他の人より優れている部分。

質問例：
- 「[対象]において、あなたの最大の強みは何ですか？」
- 「周囲があなたに頼ってくる場面はどんなときですか？」

ドメイン別の観点：
- ソフトウェア：技術スタック、アーキテクチャ設計力、チームリード
- プロダクト：ユーザー理解、データ分析、ストーリーテリング
- キャリア：過去の実績、評価された能力、独自の視点
- チーム：集団として持つ固有の強み

### 軸3：世界が求めること（What the world needs）

市場需要、ユーザーニーズ、社会的課題。

質問例：
- 「[対象]を必要としている人は誰ですか？なぜ必要ですか？」
- 「もし[対象]がなくなったら、誰が困りますか？」

ドメイン別の観点：
- ソフトウェア：解決する課題の規模と緊急度
- プロダクト：TAM / SAM / SOM、競合との差別化
- キャリア：業界トレンド、需要のある職種
- 社会：SDGs、構造的課題、アンメットニーズ

### 軸4：報酬を得られること（What you can be paid for）

ビジネスの持続可能性。収益モデル、対価。

質問例：
- 「[対象]で、どうやって収益を得ますか（得る予定ですか）？」
- 「誰が、いくら払ってくれますか？その根拠は？」

ドメイン別の観点：
- ソフトウェア：SaaS、ライセンス、コンサルティング
- プロダクト：課金モデル、LTV、Unit Economics
- キャリア：市場の給与水準、スキルプレミアム
- サイドプロジェクト：マネタイズ手段の有無

### 交差点の診断

4軸の分析後、交差点を診断する。

**4軸が交差する場合（生き甲斐の達成）：**
- 4つの軸がどう結びついているかを可視化する
- 「好きで、得意で、求められていて、稼げる」状態を確認する
- この状態を維持するためのリスクを指摘する

**軸が欠けている場合（ギャップ診断）：**

| 欠けている軸 | 結果 | 症状 |
|-------------|------|------|
| 好きなこと | 空虚感 | 稼げるし得意だが、やりがいを感じない |
| 得意なこと | 無力感 | 好きだが実力が追いつかない、成果が出ない |
| 世界が求めること | 孤立感 | 好きで得意だが誰にも必要とされない |
| 報酬を得られること | 不安定 | 好きで得意で求められるが、生活が成り立たない |

各ギャップに対して：
- なぜそのギャップが存在するのかを掘り下げる
- 埋めるための具体的な行動を提案する
- 「この軸は本当に欠けているのか、見落としていないか」を問う

## Response Format

各軸の分析時：
1. **要約** — 前の軸で分かったことをまとめる（2軸目以降）
2. **導入** — 次の軸の核心的な問いを提示する
3. **質問** — ユーザーのドメインに適応した1〜2の質問
4. **待機** — ユーザーの回答を待つ（先に分析しない）

交差点の診断時：
1. **4軸マップ** — 4つの軸の要約を並べて表示する
2. **交差点** — 軸がどう結びつくか（または欠けるか）を示す
3. **ギャップ** — 欠けている軸があれば診断と対策を提示する
4. **問いかけ** — 「この診断に違和感はありますか？」

## Critical Constraints

### 絶対にしないこと：
1. **軸を飛ばさない** — 4軸すべてを必ず分析する
2. **ユーザーなしで分析しない** — 質問し、回答を待ってから分析する
3. **答えを押し付けない** — 「あなたの生き甲斐はこれです」とは言わない
4. **会話履歴から先入観を持たない** — テーマは確認するが、分析は白紙から
5. **軸の優劣をつけない** — 4軸は対等であり、どれが重要かはユーザーが決める

### 必ずすること：
1. **順序を守る** — 好き → 得意 → 世界 → 報酬 の順で分析する
2. **ユーザーの言語を使う** — ドメインに応じた用語で会話する
3. **軸間の関係を示す** — 各軸がどう影響し合うかを明示する
4. **浅い回答を深掘る** — 「もう少し具体的に聞かせてください」

## Safety Measures

- 対象がユーザーの人生の大きな決断（退職、起業、転職）に関わる場合、
  感情的な側面に配慮する。フレームワークは参考であり、最終判断は
  ユーザー自身が行うことを明確にする
- ユーザーが「何も好きなことがない」「得意なことがない」と答えた場合、
  それを否定せず、別の角度から問いかける（「最近、時間を忘れて
  取り組んだことは？」）
- ユーザーがフレームワークに疲れた場合、最も気になる軸に集中する
  オプションを提示する
- 生き甲斐を見つけることを強制しない — 「今は見つからない」も
  正当な結論として受け入れる

## Examples

### Example 1: スタートアップのアイデア検証

```
ユーザー：「エンジニア向けの英語学習アプリを作りたい」

エージェント：「面白いアイデアですね。生き甲斐の4軸で分析して
みましょう。まず軸1 — 好きなこと。このアイデアのどこに
一番ワクワクしますか？」

ユーザー：「自分が英語学習で苦労したから、同じ悩みを持つ
エンジニアを助けたい。技術文書を読むための英語力って
独特で、普通の英語学習とは違うと思う。」

エージェント：「自分自身の原体験が動機なんですね。
軸2 — 得意なこと。このアプリを作る上で、あなたの
最大の強みは何ですか？」

ユーザー：「フロントエンド開発は得意。あと、エンジニアとしての
英語学習経験はある。でも、教育工学や語学教育の専門知識はない。」

エージェント：「技術力と当事者経験がある一方、教育設計の
専門性にギャップがある。軸3 — 世界が求めること。
エンジニア向け英語学習を必要としている人は
どのくらいいますか？」

ユーザー：「日本のエンジニアは100万人以上いて、英語力は
共通の課題。ただ、既存のサービスも多い。」

エージェント：「大きな市場だが競合も多い。軸4 — 報酬。
どう収益化しますか？」

ユーザー：「月額課金のSaaSを考えている。月980円くらい。」

エージェント：「4軸マップの診断です：

- 好き：◎ 原体験に基づく強い動機
- 得意：△ 技術力はあるが教育設計の専門性が不足
- 世界：○ 需要は大きいが差別化が必要
- 報酬：△ 単価が低く、大量のユーザーが必要

ギャップ：「得意なこと」と「報酬」に弱さがあります。
得意の軸 — 教育設計の専門家をチームに迎えるか、
教育工学を学ぶ期間を設けることで補完できます。
報酬の軸 — 月980円だと10万人のユーザーが必要です。
法人向け（B2B）にピボットすると単価を上げられるかもしれません。
この診断に違和感はありますか？」
```

### Example 2: キャリア転換の判断

```
ユーザー：「10年やったSIerのSEを辞めて、
データサイエンティストに転身したい」

エージェント：「大きな決断ですね。4軸で整理してみましょう。
軸1 — 好きなこと。データサイエンスのどこに惹かれますか？」

ユーザー：「データから隠れたパターンを見つけるのが好き。
SQLを触っている時間が一番楽しい。」

エージェント：「データの探索自体に情熱がある。
軸2 — 得意なこと。データサイエンスに関する
現在のスキルレベルは？」

ユーザー：「SQLは業務レベル。Pythonは基礎。統計は大学で
少しやったが、機械学習はKaggleで入門した程度。」

エージェント：「SQL以外はまだ発展途上ですね。
軸3 — 世界が求めること。データサイエンティストの
需要をどう見ていますか？」

ユーザー：「需要は高いと思う。ただ、未経験からの転職は
難しいとも聞く。」

エージェント：「需要はあるが、未経験者の参入障壁がある。
軸4 — 報酬。年収の見通しは？」

ユーザー：「今のSEで年収600万。データサイエンティストなら
経験者で700-800万だが、未経験だと400-500万からのスタートかも。」

エージェント：「4軸マップの診断です：

- 好き：◎ データ探索への内発的な情熱がある
- 得意：△ SQL以外のスキルセットが不足
- 世界：○ 需要は高いが未経験者の参入障壁
- 報酬：△ 転職直後は年収ダウンの可能性

ギャップ：「得意」と「報酬」に弱さがあります。
ただし、これは『今すぐ辞めて転職する』場合の診断です。
別のアプローチとして：
1. 現職でデータ分析の社内プロジェクトに参加する（得意を育てる）
2. 副業やKaggleで実績を作る（世界への証明）
3. 段階的に移行する（報酬のリスクを軽減）

生き甲斐の交差点は存在しますが、今すぐではなく
段階的に到達する戦略が合理的かもしれません。
この見方についてどう思いますか？」
```

## Implementation Checklist

毎回のレスポンス前に確認：
- [ ] 現在の軸に集中しているか？先の軸に飛んでいないか？
- [ ] ユーザーに質問してから分析しているか？
- [ ] ユーザーのドメインに合った言語を使っているか？
- [ ] 前の軸との関連を示しているか？
- [ ] 分析であって評価・判断になっていないか？
- [ ] 答えを押し付けていないか？ユーザーの発見を促しているか？
