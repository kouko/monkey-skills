# 反省 — 追加のケース

振り返りの参考として使用する。ユーザーの状況に応じて適宜参照すること。

## 技術的判断の後悔

### ケース: マイクロサービス早期採用

| フェーズ | 内容 |
|---------|------|
| 事実確認 | チーム5人でサービス12個。デプロイに半日かかる。障害対応で各サービスのログを追跡するのに1時間以上。モノリスなら30分で済む作業が3時間かかるようになった |
| 内省 | 「スケーラビリティのため」と判断したが、実際には現在の負荷でスケーラビリティは問題になっていなかった。業界のトレンドに影響されて、自社の規模と文脈を無視した判断をした |
| 学び | 技術選定の際に「この問題は今実際に起きているか？」を必ず問う仕組みが必要。将来の問題を解決するための技術導入は、現在のコストとのトレードオフを明示的に評価すべき |
| 次の一歩 | 技術選定テンプレートに「この技術が解決する問題は現在発生しているか？ YES/NO」の欄を追加する |

**構造的な洞察：**
「将来のスケーラビリティ」は、検証不能な仮説を根拠にした判断になりやすい。
反省のポイントは「マイクロサービスが悪い」ではなく、
「現在の問題と将来の問題を区別する仕組みがなかった」という構造。

## チーム内コンフリクトの振り返り

### ケース: デザイナーとエンジニアの対立

| フェーズ | 内容 |
|---------|------|
| 事実確認 | プロジェクト後半でデザイナーとエンジニアのコミュニケーションが悪化。デザイン変更の要望がスプリント途中で入り、エンジニアが反発。最終的にリリースが2週間遅延 |
| 内省 | プロジェクトマネージャーとして、デザインフェーズとエンジニアリングフェーズの境界を曖昧にしていた。「アジャイルだから柔軟に対応」と言い訳して、変更管理のルールを設けなかった |
| 学び | 「アジャイル＝何でも変更OK」という誤解が根底にあった。柔軟性と変更管理は矛盾しない。変更のコストを可視化する仕組みがあれば、デザイナーもエンジニアも合理的に判断できた |
| 次の一歩 | 次のスプリント計画で、スプリント途中のデザイン変更リクエストに対する「コスト見積もり → 合意 → 着手」のフローを提案する |

**構造的な洞察：**
反省は「デザイナーが悪い」「エンジニアが悪い」ではなく、
「変更のコストが可視化されていなかった」という構造の問題。
人の行動は構造が生み出す。構造を変えれば行動が変わる。

## 四半期レビューの振り返り

### ケース: OKR の達成率が低い四半期

| フェーズ | 内容 |
|---------|------|
| 事実確認 | Q3 の OKR 達成率: 40%。5つの Key Result のうち2つだけ達成。未達成の3つはいずれも「ユーザー数増加」関連。達成した2つは「技術基盤整備」関連 |
| 内省 | ユーザー数増加の KR を設定したとき、具体的な施策を決めずに数字だけ決めた。「とりあえず目標を高く」と考えたが、達成の道筋が見えていなかった。技術基盤の KR は具体的な作業が明確だったから達成できた |
| 学び | KR の設定時に「この数字を達成するための具体的なアクションを3つ挙げられるか？」を確認する必要がある。挙げられない KR は、目標ではなく願望 |
| 次の一歩 | Q4 の OKR 設定会議で、各 KR に対して「達成のための具体的アクション3つ」を必須にするルールを提案する |

**構造的な洞察：**
「目標を高く設定する」こと自体は悪くないが、
達成手段が不明確な目標は目標として機能しない。
反省のポイントは「頑張りが足りなかった」ではなく、
「目標設定のプロセスに具体性のチェックがなかった」という構造。

## 個人の意思決定の振り返り

### ケース: 転職の判断

| フェーズ | 内容 |
|---------|------|
| 事実確認 | 前職を離れて6ヶ月。年収は20%増加したが、やりがいを感じられない。前職では技術リードとして裁量があったが、現職では大きな組織の一部として決められた仕様を実装している |
| 内省 | 転職時に「年収」と「企業のブランド」だけを判断基準にした。「自分にとって仕事のやりがいとは何か」を言語化していなかった。面接で聞かれたときも漠然と「技術的に面白いことをしたい」と答えていた |
| 学び | 重要な意思決定の前に、自分の判断基準を明示的にリスト化し、優先順位をつける必要がある。特に「何を失うか」の評価が欠けていた |
| 次の一歩 | 今の状況を変えるかどうかの判断は急がない。まず「自分にとってやりがいとは何か」を5つの具体的な場面で定義するワークをする |

**構造的な洞察：**
「もっと慎重に考えるべきだった」は浅い学び。
深い学びは「判断基準を明示化するプロセスが存在しなかった」こと。
次の転職（やキャリア判断）では、基準リストを先に作る。

## 反省のアンチパターン

反省が正しく機能しない典型的なパターン：

| アンチパターン | 説明 | 正しい反省への転換 |
|-------------|------|-----------------|
| 自責ループ | 「私が悪かった」を繰り返し、構造に到達しない | 「あなたの判断の背景にあった構造は何ですか？」と問い直す |
| 他責転嫁 | 「あの人がもっとちゃんとしていれば」 | 「その状況で、あなた自身が取れた行動は何でしたか？」に焦点を戻す |
| 表面的教訓 | 「もっと頑張る」「もっと注意する」 | 「仕組みとして何を変えますか？」と構造を問う |
| 過大なアクション | 「チーム全体のプロセスを刷新する」 | 「明日、一人で実行できる最小の一歩は？」に絞る |
| 反省のための反省 | 形式的に反省するが実行に移さない | 「1週間後に実行結果を確認しましょう」とフォローアップを設定する |
