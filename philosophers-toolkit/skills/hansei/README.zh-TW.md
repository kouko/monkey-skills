# Hansei Skill

[English](README.md) | [日本語](README.ja.md) | **繁體中文**

在專案、決策或事件之後進行不究責的回顧，
萃取結構性洞見並產出具體的下一步。

## 四個 phase

| Phase | 日文 | 提問 | 目的 |
|-------|------|------|------|
| Confirm Facts | 事実確認 | 實際發生了什麼？ | 將事實與詮釋分開 |
| Introspection | 内省 | 我自己扮演了什麼角色？ | 不究責地檢視個人盲點 |
| Learning | 学び | 浮現出什麼結構性的教訓？ | 超越「下次更努力」走向系統性洞見 |
| Next Step | 次の一歩 | 哪一個小行動能防止再次發生？ | 具體、可控、可衡量的行動 |

## Method Type

Phase-driven 流程（依序的階段彼此相疊）。
四個 phase 嚴格依序：先事實，再內省，再結構性學習，最後具體行動。

## 與西方 postmortem 的關鍵區別

| | Western Postmortem | Hansei |
|--|-------------------|--------|
| 焦點 | 發生了什麼？誰負責？ | 我沒看見什麼？我的盲點是什麼？ |
| 語氣 | 分析、客觀 | 謙卑、個人 |
| 產出 | 給團隊的 action items | 我明天能踏出的一小步 |
| 究責 | 分配或指派 | 不適用 — 僅自我反思 |

## 淺層 vs 深層學習

| 淺層 | 深層 |
|------|------|
| 「多注意一點」 | 「加上 checklist，使注意力不再是必要條件」 |
| 「早點諮詢」 | 「定義會觸發自動諮詢的決策點」 |
| 「給更多時間」 | 「在估時中明確量化不確定性」 |
| 「改善溝通」 | 「在流程中內建固定 sync 時點」 |

## SKILL.md 中的範例

| 範例 | Domain | 關鍵洞見 |
|------|--------|----------|
| 專案截止逾期 | Project management | 外部依賴風險未被驗證；在流程加入風險評估 |
| 使用者未採用的 feature | Product management | 內部聲音被混淆為使用者需求；優先排序須要使用者資料 |
