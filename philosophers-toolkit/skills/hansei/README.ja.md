# Hansei Skill

[English](README.md) | **日本語** | [繁體中文](README.zh-TW.md)

プロジェクト、決定、出来事のあとに責めのない振り返りを行い、
構造的な洞察と具体的な次の一歩を引き出す。

## 4 つの phase

| Phase | 日本語 | 問い | 目的 |
|-------|------|------|------|
| Confirm Facts | 事実確認 | 実際に何が起きたか？ | 事実と解釈を分離する |
| Introspection | 内省 | 自分はどう関わったか？ | 責めず個人の盲点を点検する |
| Learning | 学び | どんな構造的教訓が浮かぶか？ | 「次はもっと頑張る」を超えて系統的な洞察へ |
| Next Step | 次の一歩 | 再発を防ぐ小さな行動ひとつは？ | 具体的・制御可能・計測可能な行動 |

## Method Type

Phase-driven プロセス（順次的な段階が積み重なる）。
4 phase の順序は厳密：まず事実、次に内省、次に構造的学び、最後に具体的行動。

## 西洋の postmortem との違い

| | Western Postmortem | Hansei |
|--|-------------------|--------|
| 焦点 | 何が起きたか？誰が責任者か？ | 自分は何が見えていなかったか？自分の盲点は何か？ |
| トーン | 分析的・客観的 | 謙虚・個人的 |
| 出力 | チームの action items | 明日自分が踏み出せる小さな一歩 |
| 責め | 分散または割り当て | 該当しない — 自己反省のみ |

## 浅い学び vs 深い学び

| 浅い | 深い |
|------|------|
| 「もっと注意する」 | 「checklist を加えて注意に依存しないようにする」 |
| 「早めに相談する」 | 「自動的に相談を起こす意思決定 point を定義する」 |
| 「もっと時間を取る」 | 「見積もりに不確実性を明示的に数値化する」 |
| 「コミュニケーションを改善する」 | 「定期的な sync point をプロセスに組み込む」 |

## SKILL.md 中の例

| 例 | Domain | 主な洞察 |
|------|--------|----------|
| プロジェクト納期超過 | Project management | 外部依存のリスクが未検証；プロセスにリスク評価を追加 |
| ユーザーに使われなかった feature | Product management | 社内の声をユーザーニーズと混同；優先順位付けにはユーザーデータが必要 |
