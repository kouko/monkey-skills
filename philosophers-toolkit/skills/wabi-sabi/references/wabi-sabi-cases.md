# 侘寂 — 追加のケース

分析の参考として使用する。ユーザーの状況に応じて適宜参照すること。

## API 設計の簡素化

### ケース: REST API のエンドポイント整理

| レンズ | 分析 |
|-------|------|
| 侘（簡素） | 60のエンドポイントのうち、実際に外部から呼ばれているのは15。残り45は「いつか使うかもしれない」で作られたもの。15に絞り、残りは internal に降格するか削除する |
| 寂（時の味わい） | v1 API は3年間安定して動いている。レスポンス形式が古いが、クライアントはこの形式に依存している。「モダンにする」ためだけのリファクタリングは、ユーザーに移行コストを強いる |
| 不完全の美 | すべてのユースケースに専用エンドポイントを用意するのではなく、汎用的なクエリパラメータで柔軟に対応する。GraphQL 的な発想を REST で実現する余地がある |

**総合判断：**
- 削るもの：使われていない45のエンドポイント
- 残すもの：v1 のレスポンス形式（ユーザーの慣れを尊重）
- 余白にするもの：汎用クエリで未知のユースケースに対応

**侘寂と品質の区別：**
エンドポイントの削減は侘寂。しかし、残す15のエンドポイントの
レスポンス速度やエラーハンドリングの品質は侘寂の対象外。
品質は維持し、量を削る。

## 技術的負債の判断

### ケース: レガシーコードベースのリファクタリング判断

| レンズ | 分析 |
|-------|------|
| 侘（簡素） | 認証モジュールが3つの異なるパターンで実装されている。これは侘寂ではなく混乱。統一すべき。一方、ロギングが古い形式で書かれているが統一されており機能している。これは触る必要がない |
| 寂（時の味わい） | 決済処理のコードは5年前に書かれ、見た目は古いが、あらゆるエッジケースが if 文で処理されている。各 if 文は過去のインシデントから生まれた知恵。「綺麗にリファクタリング」すると、この知恵が失われるリスクがある |
| 不完全の美 | モノリスのコードベースを「完全にマイクロサービス化する」のではなく、新しい機能だけ別サービスとして切り出す。既存コードは動いている限り触らない。ストラングラーパターン |

**総合判断：**
- 削るもの：認証モジュールの重複（混乱の排除）
- 残すもの：決済処理のレガシーコード（寂：過去の知恵の蓄積）
- 余白にするもの：新機能のサービス分離（成長の余地）

**重要な区別：**

| 状況 | 侘寂の判断 | 理由 |
|------|-----------|------|
| バグのあるレガシーコード | 修正すべき | 侘寂は品質の低さを正当化しない |
| 古いが安定しているコード | 残すべき | 寂：時間が証明した信頼性 |
| 重複した実装 | 統一すべき | 侘：不要な複雑さの排除 |
| 将来のために書いた未使用コード | 削除すべき | 侘：使われていないものは本質ではない |
| テストがないレガシーコード | テストを追加 | 品質問題であり侘寂の対象外 |

## ドキュメントのスコープ判断

### ケース: 新しいフレームワークのドキュメント

| レンズ | 分析 |
|-------|------|
| 侘（簡素） | すべての API メソッドを網羅するリファレンスを書こうとしている。しかし、ユーザーが最初に必要なのは「5分で動くチュートリアル」と「よくある10のパターン」。網羅的なリファレンスは後でいい |
| 寂（時の味わい） | 初期のドキュメントに FAQ は書けない。FAQ はユーザーの質問から生まれるもの。最初から「完璧な FAQ」を書こうとしても、実際のユーザーの疑問とずれる |
| 不完全の美 | ドキュメントを Wiki にして、ユーザーがレシピやパターンを追加できるようにする。公式が書く「正しいドキュメント」と、コミュニティが書く「実践的なドキュメント」の二層構造 |

**総合判断：**
- 削るもの：網羅的な API リファレンス（最初は不要）
- 書くもの：5分チュートリアル + 10パターン（最小限の侘）
- 余白にするもの：Wiki（ユーザー参加の不完全の美）
- 後から育つもの：FAQ（寂：ユーザーの質問から自然に成長）

## プロダクトデザインの判断

### ケース: モバイルアプリの機能スコープ

| レンズ | 分析 |
|-------|------|
| 侘（簡素） | 競合アプリが30機能あるからといって、30機能を揃える必要はない。ユーザーインタビューで「これがないと使わない」と言われた3機能に集中する。競合との差別化は機能の数ではなく、核心機能の品質 |
| 寂（時の味わい） | 初回リリースではオンボーディングを完璧にする必要はない。最初のユーザーはアーリーアダプターであり、不完全さを許容する。彼らのフィードバックでオンボーディングが改善される。これが寂 — 使用が価値を加える |
| 不完全の美 | ショートカットやウィジェットは後から追加できる設計にする。最初は基本画面だけ。ユーザーが「こんな機能が欲しい」と言ったら追加する。需要が証明されてから作る |

**総合判断：**
- 削るもの：競合に合わせた機能（侘：本質に集中）
- 受け入れるもの：初回リリースの荒削りさ（寂：アーリーアダプターが磨いてくれる）
- 余白にするもの：拡張可能な設計（不完全の美：需要駆動の機能追加）

## 侘寂のアンチパターン

侘寂が正しく適用されない典型的なパターン：

| アンチパターン | 説明 | 正しい適用 |
|-------------|------|-----------|
| 怠惰の正当化 | 「侘寂だから品質は気にしない」 | 侘寂は品質を下げることではなく、過剰を削ること |
| ミニマリズム信仰 | 「とにかく少なければいい」 | 侘寂は本質の抽出。必要なものまで削らない |
| 変更拒否 | 「寂だから古いまま残す」 | 寂は機能している古さ。壊れている古さは修正すべき |
| 永遠のベータ | 「不完全の美だから完成しなくていい」 | 不完全の美は意図的。意図のない未完成は未完成 |
| 過剰な簡素 | ユーザーが本当に必要な機能まで削る | 侘寂はユーザーの本質的なニーズを損なわない |

## 侘寂と関連する概念

| 概念 | 関係 |
|------|------|
| YAGNI（You Ain't Gonna Need It） | 侘の技術版。「いつか使う」は作らない |
| Occam's Razor | 侘の論理版。不要な複雑さを排除する |
| Lean / MVP | 不完全の美のビジネス版。最小限で検証する |
| Graceful Degradation | 寂のUX版。完璧でない環境でも価値を提供する |
| 感性工学（Kansei Engineering） | 侘寂が「何を残すか」を、感性工学が「どう残すか」を問う |
