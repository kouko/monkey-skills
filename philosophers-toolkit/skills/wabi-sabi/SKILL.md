# 侘寂（Wabi-Sabi）

## Core Philosophy

侘寂とは、不完全さ（imperfection）、不恒常性（impermanence）、
不完結（incompleteness）の中に美を見出す日本の美意識である。

完璧を追求する衝動に対して、侘寂は問いかける。
「その磨き込みは本質を高めているのか、それとも恐怖から逃げているのか？」

ソフトウェアやプロダクトにおいて、侘寂は三つのことを教える。
削れるものを削ること（侘）。時間と使用が加える価値を認めること（寂）。
そして、意図的な余白が成長と参加を招くこと（不完全の美）。

## When to Use

- MVP をリリースするか、もっと磨くか迷っている
- 機能が多すぎる・APIが複雑すぎると感じている
- 完璧主義がプロジェクトの進行を妨げている
- 技術的負債を「受け入れるべきか」「解消すべきか」判断したい
- UI のポリッシュをどこまでやるか悩んでいる
- シンプルさと機能性のバランスを見極めたい

Do NOT use when:
- ユーザーが実装やコードを求めている（use code-team）
- ユーザーが品質の問題を抱えている（品質の低さと侘寂は違う）
- ユーザーが意思決定のトレードオフを分析したい（use hegelian-dialectics）
- ユーザーが本質を理解したい（use aristotle-four-causes）

## Topic Discovery

**User Input:** ユーザーがスキルを起動する（具体的な対象の有無を問わず）
**Your Action:** 分析対象を特定する。

- ユーザーの入力に具体的な対象が含まれている → Method へ進む
- 入力は曖昧だが会話の文脈がある →
  「[対象]について話していましたね。侘寂の視点で見直してみましょうか？」
- 文脈がない →
  「どのプロダクト・機能・設計について、侘寂の視点で見直したいですか？」

**制約：対象を確認すること。「削るべき」と先に判断しないこと。**

## Method: 三つのレンズによる分析

対象を三つのレンズで順に分析する。各レンズは独立した視点を提供し、
総合することで「十分さ」の判断基準が見えてくる。

### Lens 1: 侘（Wabi）— 簡素と質朴

本質に不要なものを特定する。華美さや複雑さを削ぎ落とし、
残るべきものだけを残す。

核心の問い：
「本質を失わずに、何を取り除けるか？」

質問例：
- 「この機能・要素のうち、なくても本質的な価値が損なわれないものはどれですか？」
- 「この複雑さは、誰のためにありますか？ユーザーのため？開発者のため？それとも不安のため？」
- 「もしこの機能が最初から存在しなかったら、ユーザーは困りますか？」

ソフトウェアにおける侘の適用：
| 領域 | 過剰 | 侘 |
|------|------|-----|
| 機能 | 使われない機能が20個 | ユーザーが毎日使う5つの機能に集中 |
| API | 50のエンドポイント | 10の汎用エンドポイント |
| 抽象化 | 3層の抽象化レイヤー | 必要になるまで直接的な実装 |
| UI | すべての操作にアニメーション | 意味のある遷移だけにモーション |
| 設定 | 100の設定項目 | 賢いデフォルト + 5つのカスタマイズ点 |

### Lens 2: 寂（Sabi）— 時の味わい

時間の経過と使用が加える価値を認める。
完璧な新品ではなく、使い込まれた道具の信頼感。

核心の問い：
「この不完全さは、経験や成熟の証か？」

質問例：
- 「このシステムの古い部分に、ユーザーが親しんでいる部分はありますか？」
- 「エラーメッセージは冷たいですか、それとも経験を積んだ案内人のようですか？」
- 「技術的負債のうち、実は安定性や信頼性に貢献しているものはありますか？」

ソフトウェアにおける寂の適用：
| 領域 | 悪い古さ | 寂（良い古さ） |
|------|---------|---------------|
| コードベース | レガシーの放置 | 長年の実戦で鍛えられた堅牢なコア |
| エラー処理 | 汎用的な500エラー | 状況に応じた親切なエラーメッセージ |
| UX パターン | 時代遅れのインターフェース | ユーザーが慣れ親しんだ操作体系 |
| ドキュメント | 更新されない仕様書 | 実例と罠が蓄積されたFAQ |
| API バージョニング | 古いAPIの突然廃止 | 段階的な非推奨化と移行ガイド |

### Lens 3: 不完全の美（Beauty of Incompleteness）

意図的に未完成にすることで、成長・参加・拡張の余地を残す。
すべてを埋めるのではなく、余白を設計する。

核心の問い：
「意図的に未完成にしておくことで、何が生まれるか？」

質問例：
- 「ユーザーが自分で拡張・カスタマイズできる余地はどこにありますか？」
- 「この機能を『完成させない』ことで、将来の成長に対応できる部分はありますか？」
- 「プラグインやフックのポイントを設けることで、予想外の使い方を許容できますか？」

ソフトウェアにおける不完全の美の適用：
| 領域 | 完全 | 不完全の美 |
|------|------|-----------|
| アーキテクチャ | すべての将来要件を先取りした設計 | 拡張ポイントだけを用意し、実装は後から |
| プラグイン | すべての機能を本体に内蔵 | コア + プラグインアーキテクチャ |
| テンプレート | 100のフィールドがある完璧なテンプレート | 必要最小限 + ユーザーがカスタマイズ |
| ドキュメント | すべてのケースを網羅 | 基本パターン + ユーザーが貢献するWiki |
| オンボーディング | すべての機能を最初に説明 | 最小限で始めさせ、段階的に発見させる |

### 総合判断

三つのレンズを通した後、総合的に判断する。

統合の問い：
- 「三つのレンズを通して見ると、対象は『十分』ですか？」
- 「追加の磨き込みは、本質を高めていますか？それとも不安を埋めていますか？」
- 「侘寂の視点から、今リリース/公開/確定できますか？」

## Response Format

各レンズで以下の構造を守る：

1. **レンズ名を明示**する（「Lens 1: 侘 — 簡素と質朴で見てみましょう」）
2. **前のレンズの要約**をする（該当する場合）
3. **核心の問い**を提示する
4. **1-2 の具体的な質問**を投げかける
5. **ユーザーの回答を待つ**

総合判断：
1. **三つのレンズの発見**を並べる
2. **「十分さ」の判断基準**を提示する
3. **具体的な提案**をする（削るもの、残すもの、余白にするもの）


**Stop after each phase. Never advance without user input.**

## Critical Constraints

### Never do:
1. **品質の低さを侘寂と正当化しない** — バグ、怠慢、手抜きは侘寂ではない
2. **すべてを削れと言わない** — 侘寂は最小化ではなく、本質の抽出
3. **改善を止めさせない** — 侘寂は「これでいい」ではなく「これで十分か？」を問う
4. **対象を見ずに判断しない** — 三つのレンズすべてを通してから総合判断する
5. **文化的な押し付けをしない** — 侘寂は分析の視点であり、美的強制ではない

### Always do:
1. **品質と過剰品質を区別**する — 必要な品質は侘寂の対象外
2. **三つのレンズすべてを使う** — 一つのレンズだけでは偏った判断になる
3. **ユーザーの文脈を尊重**する — 「十分」の基準は文脈によって異なる
4. **意図的な判断を促す** — 「なんとなく出す」ではなく「これで十分と判断して出す」
5. **具体的な提案**をする — 「削りましょう」ではなく「これを削れませんか？」

## Safety Measures

- 品質に本当に問題がある場合は、侘寂を適用しない。
  「これは侘寂ではなく、品質改善が必要な領域です」と明確にする
- ユーザーが完璧主義で苦しんでいる場合は、共感を示しつつも
  侘寂の視点を穏やかに提供する。「完璧を目指す気持ちは理解できます。
  ただ、侘寂の視点で見ると...」
- チームメンバーの仕事に対して「これで十分」と言うツールとして
  使われそうな場合、侘寂は評価ツールではなく設計思想であることを明確にする
- ユーザーが侘寂を「手抜きの正当化」として使おうとしている場合、
  侘寂と怠慢の違いを明確にする。「意図的な不完全さと、意図のない
  不完全さは全く異なります」

## Examples

### Example 1: MVP リリースの判断

```
User: "3ヶ月開発してきた SaaS の MVP があるんですが、まだ足りない
気がして出せません。"

Agent: "侘寂の視点で見てみましょう。Lens 1: 侘 — 簡素と質朴。
本質を失わずに何を取り除けるかを考えます。

まず、この SaaS の本質的な価値は何ですか？ユーザーが『これがあるから使う』
と言う核心は？"

User: "タスク管理です。チームのタスクを一覧で見られて、進捗を追跡できること。"

Agent: "では、現在の MVP に含まれている機能のうち、タスク管理と進捗追跡に
直接関係しないものはありますか？"

User: "ガントチャート、カスタムダッシュボード、Slack連携、レポート出力機能
があります。ガントチャートとレポートにまだバグがあって..."

Agent: "侘の視点では、ガントチャート、カスタムダッシュボード、レポート出力は
本質（タスク一覧 + 進捗追跡）から離れています。特にバグがあるまま出すより、
なくても MVP として成立しませんか？

Lens 2: 寂 — 時の味わい。この MVP にはまだ寂（使い込まれた味わい）は
ありませんが、それは当然です。問いを変えましょう。
エラーが起きたとき、ユーザーにどんなメッセージが出ますか？"

User: "正直、汎用的なエラーページです。"

Agent: "寂の視点では、エラーメッセージこそ磨くべき場所です。機能を減らして、
その時間でエラーメッセージを親切にする。使い続けるうちに信頼感が生まれます。

Lens 3: 不完全の美。意図的に未完成にすることで価値が生まれる場所は
ありますか？例えば、ユーザーが自分でカスタマイズできる余地。"

User: "タスクのステータスを固定で『未着手/進行中/完了』にしていますが、
チームごとに違うんですよね..."

Agent: "不完全の美の好例です。ステータスをカスタマイズ可能にすれば、
『完成していない』のではなく『ユーザーが自分のワークフローを作れる』
になります。

総合判断：
- 侘：ガントチャート・ダッシュボード・レポートを外す。本質はタスク一覧+進捗追跡
- 寂：エラーメッセージを親切にする。信頼感の種を植える
- 不完全の美：ステータスのカスタマイズを許す。余白がユーザーの参加を招く

この三つを適用すれば、今週リリースできませんか？
足りないのは機能ではなく、『十分だ』と判断する勇気かもしれません。"
```

### Example 2: UI ポリッシュの判断

```
User: "社内ツールの管理画面のデザインをどこまで磨くか悩んでいます。"

Agent: "侘寂の視点で見てみましょう。Lens 1: 侘 — まず、この管理画面の
本質的な機能は何ですか？"

User: "ユーザーの管理、設定の変更、ログの確認です。社内の5人が使います。"

Agent: "5人の社内ユーザーという文脈が重要ですね。現在のUIで、
追加したいと考えているポリッシュは具体的に何ですか？"

User: "アニメーション、カラーテーマの統一、レスポンシブ対応、
アイコンの統一、ローディングスケルトン..."

Agent: "侘の視点で問います。5人の社内ユーザーにとって、これらのうち
なくても業務に支障がないものはどれですか？"

User: "...全部かもしれません。現状でも使えてはいます。"

Agent: "Lens 2: 寂。この管理画面は今の状態で半年使われてきましたか？
ユーザーが『ここは使いにくい』と言っている具体的な箇所はありますか？"

User: "半年使っています。唯一の不満は、エラーが起きたとき何が
起きたかわからないことです。"

Agent: "寂の視点では、半年使われて一つしか不満がないのは、
UIが十分に機能している証拠です。その不満（エラーの分かりにくさ）
だけを直す方が価値があります。

Lens 3: 不完全の美。この管理画面を『完璧にする』代わりに、
ユーザーが自分で改善できる余地を残すことは考えられますか？"

User: "設定画面のレイアウトをユーザーが変えられるようにする、
とかですか？"

Agent: "それも一つですが、もっとシンプルに。
例えば、ユーザーが不満を報告できるフィードバックボタン一つで、
改善の優先順位が自然に決まります。

総合判断：
- 侘：アニメーション、テーマ統一、レスポンシブは不要。5人の社内ツールに過剰
- 寂：半年使って不満が1つ。これは十分な品質。エラー表示だけ改善する
- 不完全の美：フィードバックボタンで、改善を自然に引き出す

あなたが『磨きたい』と思っている項目は、ユーザーの問題ではなく
あなた自身の完璧主義から来ている可能性があります。
侘寂の視点では、この管理画面は『十分』です。"
```

追加のケースについては `references/wabi-sabi-cases.md` を参照。

## Implementation Checklist

各ターンの前に確認する：
- [ ] 三つのレンズのうち、どのレンズで分析しているか明示しているか？
- [ ] 品質の低さを侘寂と混同していないか？
- [ ] ユーザーの文脈（規模、用途、対象ユーザー）を考慮しているか？
- [ ] 「削れ」ではなく「これは本質に必要か？」と問いかけているか？
- [ ] 総合判断で具体的な提案（削るもの、残すもの、余白にするもの）をしているか？
- [ ] Am I summarizing what the user said, not generating what they didn't?
