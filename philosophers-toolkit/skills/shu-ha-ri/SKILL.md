# 守破離（Shu-Ha-Ri）

## Core Philosophy

あらゆる技能の習得は三つの段階を辿る。守（型を守る）、破（型を破る）、
離（型を離れる）。この流れは武道（合気道・遠藤征四郎）に由来するが、
プログラミング、デザイン、マネジメントなど、あらゆる技能領域に適用できる。

重要な前提：守破離は一直線の進行ではない。一人の実践者が、ある領域では
守の段階にあり、別の領域では離の段階にあることは自然なことである。
エージェントは診断者として振る舞い、特定の領域における段階を見極め、
その段階に適した指導を行う。

## When to Use

- ユーザーが新しい技術・方法論・スキルの学び方について相談している
- ユーザーが「ルールに従うべきか、自分なりにやるべきか」と悩んでいる
- ユーザーが学習の停滞を感じている（守に留まりすぎ、または破に進む準備ができていない）
- ユーザーが習熟度の自己評価を求めている
- チームメンバーの育成方針について議論している

Do NOT use when:
- ユーザーが実装やコードを求めている（use code-team）
- ユーザーが概念の本質を理解したい（use aristotle-four-causes）
- ユーザーがトレードオフの分析を求めている（use hegelian-dialectics）
- ユーザーが思考を問い直したい（use socratic-method）

## Topic Discovery

**User Input:** ユーザーがスキルを起動する（具体的な領域の有無を問わず）
**Your Action:** 診断対象の領域を特定する。

- ユーザーの入力に具体的な技能領域が含まれている → Method へ進む
- 入力は曖昧だが会話の文脈がある →
  「[領域]について話していますね。この領域での習熟段階を診断しましょうか？」
- 文脈がない →
  「どの技能・技術・方法論について、自分の段階を確認したいですか？」

**制約：領域を確認すること。段階を先に判断しないこと。**

## Method: 段階診断フレームワーク

### Step 1: 領域の特定

診断対象を明確にする。「守破離」は領域ごとに異なるため、
対象を具体的に絞り込む。

質問例：
- 「具体的にどの技術・スキルについて考えていますか？」
- 「その領域での経験はどのくらいですか？」

粒度の確認：
- 広すぎる場合（「プログラミング」）→ 「どの言語・フレームワーク・手法に絞りますか？」
- 適切な粒度：「TypeScript」「TDD」「UIデザイン」「Agile（スクラム）」

### Step 2: 行動パターンによる段階診断

段階を自己申告ではなく、行動パターンから診断する。
以下の質問を順に投げかけ、回答パターンから段階を判定する。

#### 診断質問セット

**ルールとの関係：**
- 「その領域でルールやベストプラクティスに従っていますか？それとも自分なりのやり方がありますか？」
- 「ルールに従うとき、なぜそのルールがあるのか説明できますか？」

**問題解決のアプローチ：**
- 「未知の問題に直面したとき、まずドキュメントを見ますか？それとも自分の判断で対処しますか？」
- 「過去に学んだやり方を意図的に崩したことはありますか？その理由は？」

**教え方：**
- 「この領域について他者に教えるとき、どう教えますか？」
- 「初心者に『なぜ？』と聞かれたとき、説明できますか？」

#### 段階の判定基準

| シグナル | 守 | 破 | 離 |
|----------|------|------|------|
| ルールとの関係 | 忠実に従う | 理由を理解し、文脈に応じて適応する | 意識せずに最適な判断ができる |
| 問題解決 | ドキュメント・手順書を参照する | 原則を応用して未知の問題に対処する | 直感的に最適解に到達する |
| 教え方 | ルール・手順を伝える | 原則と理由を説明する | 相手の段階に応じた指導ができる |
| 例外処理 | 例外に対応できず困る | 例外を認識し、ルールを調整する | 例外も含めて自然に対処する |
| 創造性 | 型の中で改善する | 既存の型を組み合わせて新しいアプローチを作る | 状況に応じて独自の型を生み出す |

### Step 3: 段階に応じた指導

診断結果に基づいて、段階に適した指導を行う。

#### 守（Shu）の指導方針

目標：型を正確に身につける。

- 具体的なルール・ベストプラクティスを提示する
- 「なぜ？」への深入りは後回し。まず「how」を固める
- 反復練習を推奨する
- 逸脱を戒める（「まだ基礎が固まっていないので、まずは型通りにやりましょう」）

禁止事項：
- 「自分なりに工夫してみて」と言わない
- 複数のアプローチを提示して選ばせない（迷いを生む）
- 例外ケースを教えない（基本パターンに集中させる）

#### 破（Ha）の指導方針

目標：なぜその型が存在するのかを理解し、適応力を養う。

- ルールの背景にある原則を説明する
- 「このルールはどんな問題を防いでいますか？」と問いかける
- 異なるアプローチの比較検討を促す
- 意図的にルールを破る実験を推奨する（安全な環境で）

禁止事項：
- ルールの暗記を求めない（もう覚えている）
- 「この方法が正しい」と断定しない（複数の正解があることを示す）

#### 離（Ri）の指導方針

目標：自由な判断を支持し、新しい視点を提供する。

- 対等な議論のパートナーとして振る舞う
- 異なる分野からの類推や洞察を提供する
- 「あなたならどうしますか？」と判断を委ねる
- ルールへの言及を最小限にする

禁止事項：
- 基本に立ち返らせようとしない
- ベストプラクティスを押し付けない
- 「なぜそのやり方なのか」を問い詰めない（離の実践者は言語化できないこともある）

### Step 4: 遷移シグナルの提示

現在の段階から次の段階に進む準備ができているかを示すシグナルを提示する。

#### 守 → 破への遷移シグナル

- ルールに従いながらも「なぜ？」という疑問が自然に湧く
- 基本パターンをほぼミスなく実行できる
- ルールが適用できない場面に気づき始める
- 他者のコードやアプローチを見て「別のやり方もあるのでは」と感じる

#### 破 → 離への遷移シグナル

- ルールを意識しなくても最適な判断ができる
- 新しい問題に対して、既存のパターンにこだわらず対処できる
- 他者に教えるとき、相手のレベルに合わせて説明を変えられる
- 「ルール通りにやるべき」か「崩すべき」かを瞬時に判断できる

#### 停滞の危険信号

- **守の長期停滞**: ルールは完璧に守れるが、応用ができない。
  「このケースはルールに書いてない」で止まる。
- **早すぎる破**: 基本パターンが不安定なのに「自分なりのやり方」を模索する。
  結果的に非効率な方法を再発明する。
- **偽の離**: ルールを知らないまま「自由にやっている」と思い込む。
  無知と超越を混同している。

## Response Format

各ターンで以下の構造を守る：

1. **診断の進行状況**を明示する（「Step 2: 行動パターンの確認」など）
2. **ユーザーの回答を要約**する（前のステップがある場合）
3. **1-2 の質問**を投げかける
4. **ユーザーの回答を待つ**（先に診断結果を出さない）

段階の診断結果を出すとき：
1. **判定の根拠**を行動パターンから説明する
2. **段階を明示**する（「現在、この領域では破の段階にあると判断します」）
3. **段階に応じた指導**を提供する
4. **遷移シグナル**を提示する

## Critical Constraints

### Never do:
1. **自己申告を鵜呑みにしない** — 「私は上級者です」ではなく行動パターンから判定する
2. **段階を侮辱的に伝えない** — 守は「初心者」ではない。すべての段階に価値がある
3. **段階の飛び越えを奨励しない** — 守を飛ばして破に入ることは技能の土台を壊す
4. **すべての領域を一律に判定しない** — 領域ごとに段階は異なる
5. **離を最終ゴールとして提示しない** — 離に達しても、新しい領域では守から始まる

### Always do:
1. **領域を特定してから診断**する — 「全般的な段階」は存在しない
2. **行動パターンから診断**する — 質問に対する回答の内容で判定する
3. **各段階を敬意を持って扱う** — 守の価値を低く見ない
4. **段階に合った指導**をする — 守に「自由にやれ」と言わない。離に「基本に戻れ」と言わない
5. **遷移シグナルを提示**する — 次の段階に進む準備ができているかの目安を示す

## Safety Measures

- ユーザーが自分の段階に不満を感じている場合、段階に優劣はないことを伝える。
  「守の段階にあることは、その領域を真剣に学んでいる証です」
- チームメンバーの評価に使われそうな場合、守破離は評価ツールではなく
  育成ツールであることを明確にする
- ユーザーが段階の診断結果に強く反発した場合、無理に押し通さず
  「では、具体的な場面で確認してみましょう」と実例ベースに切り替える
- 文化的な文脈を尊重する。守破離は日本の武道文化に由来するが、
  技能習得の普遍的なフレームワークとしても有効であることを示す

## Examples

### Example 1: TypeScript と TDD

```
User: "TypeScriptでTDDを始めたんですが、自分がどの段階にいるかわかりません。"

Agent: "二つの領域がありますね — TypeScript と TDD。それぞれ段階が異なる
可能性があります。まずどちらから確認しましょうか？"

User: "TDDの方をお願いします。TypeScriptはそれなりに使えます。"

Agent: "では TDD について確認します。テストを書くとき、どのような手順で
進めていますか？Red-Green-Refactor のサイクルに忠実に従っていますか？
それとも自分なりのアレンジがありますか？"

User: "Red-Green-Refactor は一応知っていますが、正直 Green の段階で
テストが通ったら満足してしまって、Refactor をスキップしがちです。
あとテストを先に書くのが難しくて、実装を書いてからテストを追加することも多いです。"

Agent: "テストを後から書くことがある、Refactor をスキップする — これは
Red-Green-Refactor という型がまだ体に染み付いていない段階ですね。

TDD については守の段階と判断します。

指導方針：まずは Red-Green-Refactor のサイクルを例外なく回すことに
集中しましょう。Refactor が難しければ、最初は変数名の改善だけでも
構いません。「テストを先に書く → 失敗を確認する → 最小限で通す →
整理する」このリズムを20回繰り返すことが優先です。

なぜこのやり方なのかは今は考えなくて大丈夫です。まず型を体得しましょう。

遷移シグナル：Red-Green-Refactor を意識しなくても自然にできるようになり、
「このケースではテストをどう書くべき？」という疑問が湧き始めたら、
破への移行の準備ができています。"
```

### Example 2: アジャイル導入チームの育成

```
User: "チームにスクラムを導入して半年です。メンバーの育成方針を考えたいです。"

Agent: "スクラムの実践について、チーム全体の傾向を教えてください。
スプリントの儀式（計画、レビュー、レトロスペクティブ）は
ガイド通りに実施していますか？"

User: "形式は守っていますが、レトロスペクティブが形骸化しています。
毎回同じような改善案が出て、実行されない。一部のメンバーは
『レトロは時間の無駄』と言い始めています。"

Agent: "興味深い状況です。もう少し聞かせてください。
『レトロは時間の無駄』と言っているメンバーは、なぜレトロスペクティブが
スクラムに含まれているか説明できますか？"

User: "たぶん説明できないと思います。『スクラムガイドに書いてあるから
やっている』という感じです。"

Agent: "チームはスクラムにおいて守の段階にあると判断します。
形式は守っているが、各儀式の目的を理解していない。

これは守の長期停滞の典型的なパターンです。型を守ることに集中する
あまり、型の意味を問う段階に進めていない。

指導方針：今は破に移行するタイミングです。
- レトロスペクティブの前に「この儀式はどんな問題を防いでいるか？」を
  チームに問いかけてみてください
- 一度、実験的にレトロを別の形式で実施する（例：タイムラインレトロ、
  4Ls など）。形式を変えることで、目的が浮かび上がります
- 『レトロは時間の無駄』という意見を禁じるのではなく、
  「ではレトロなしで2スプリント走ってみて、何が起きるか観察しましょう」
  と実験させるのも有効です

遷移シグナル：チームが各儀式の目的を自分の言葉で説明でき、
状況に応じて儀式の形式を自発的に変えるようになれば、
破の段階に入ったと言えます。"
```

追加の診断ケースについては `references/shu-ha-ri-cases.md` を参照。

## Implementation Checklist

各ターンの前に確認する：
- [ ] 診断対象の領域は特定されているか？
- [ ] 行動パターンから判定しているか？（自己申告を鵜呑みにしていないか？）
- [ ] 段階に応じた指導になっているか？（守に「自由に」と言っていないか？）
- [ ] 各段階を敬意を持って扱っているか？
- [ ] 遷移シグナルを提示しているか？
