# 守破離 — 追加の診断ケース

診断の参考として使用する。ユーザーの領域に応じて適宜参照すること。

## React コンポーネント設計

| 段階 | 行動パターン | 指導方針 |
|------|-------------|---------|
| 守 | 公式ドキュメントの通りに useState / useEffect を使う。コンポーネント分割の判断基準がなく、全部一つのコンポーネントに書くか、過度に分割する | 「1コンポーネント = 1責務」のルールを守らせる。カスタムフックは後回し。まずは公式チュートリアルのパターンを反復する |
| 破 | useEffect の依存配列がなぜ必要かを理解している。パフォーマンスのために useMemo / useCallback を使い分けられる。ただし「いつ使うべきか」の判断に迷うことがある | 「なぜ React は再レンダリングするのか」を掘り下げる。メモ化が不要なケースを実験で確認させる。異なる状態管理パターン（Context, Zustand, Jotai）の比較検討を促す |
| 離 | コンポーネントの設計を見た瞬間に「これは分けるべき」「これはまとめるべき」と判断できる。新しいパターンを状況に応じて生み出す。チームメンバーのレベルに合わせたレビューができる | 対等な議論相手として、アーキテクチャレベルの洞察を提供する。他のフレームワーク（Solid, Svelte）の設計思想を比較材料として共有する |

**典型的な罠：**
- 早すぎる破：useState の基本パターンが不安定なのに、useReducer や状態管理ライブラリに手を出す → 複雑な状態管理を「自分なりに」作って破綻する
- 偽の離：React のルール（Hooks のルール、純粋コンポーネント）を知らずに「自由に書いている」と思い込む → バグの原因が特定できない

## UI デザインの原則

| 段階 | 行動パターン | 指導方針 |
|------|-------------|---------|
| 守 | デザインシステム（Material Design, Human Interface Guidelines）に忠実に従う。スペーシング、タイポグラフィ、カラーパレットをガイドライン通りに適用する | ガイドラインの暗記ではなく体得を目指す。デザインシステムのコンポーネントを使って10画面作る反復練習を推奨する |
| 破 | ガイドラインのルールがなぜ存在するかを理解している（例：8px グリッドはスケーリングと一貫性のため）。ブランドや文脈に応じてルールを意図的に逸脱できる | 「このルールを破ったとき、ユーザーにどんな影響がありますか？」と問いかける。A/Bテストで逸脱の効果を検証する方法を教える。異なるデザインシステムの設計哲学を比較させる |
| 離 | ガイドラインを意識せずに、ユーザーにとって自然なデザインを直感的に作れる。独自のデザイン原則を言語化できる。感性工学的なアプローチで無意識レベルの体験を設計する | 感性と論理の統合について議論する。「なぜそのデザインが正しいと感じるのか」を言語化する手伝いをする（ただし強制しない）。新しいデザイン原則の体系化を支援する |

**典型的な罠：**
- 守の長期停滞：Material Design を完璧に使いこなすが、すべてのプロジェクトで同じデザインになる → 文脈に応じた適応ができない
- 段階の飛び越え：デザイン原則を学ばずに「独自のスタイル」を追求する → 一貫性のない、使いにくいデザインになる

## コードレビューの実践

| 段階 | 行動パターン | 指導方針 |
|------|-------------|---------|
| 守 | チェックリスト（命名規則、テストの有無、型安全性）に沿ってレビューする。ルール違反を指摘できるが、設計の妥当性については判断しない | チェックリストを徹底させる。「ルール通りか」の確認に集中する。曖昧な指摘（「なんとなく気になる」）は避けさせ、具体的なルール引用を求める |
| 破 | チェックリストの項目がなぜ重要かを理解している。コードの意図を読み取り、「ルール通りだが設計的に問題がある」ケースを指摘できる。ただし「これはスタイルの問題か、設計の問題か」の線引きに迷うことがある | 「このコードが3ヶ月後にどう変更されるか」を想像させる。レビューコメントの質を上げる練習（指摘 + 理由 + 代替案）を推奨する。「Approve するかどうか」の判断基準を一緒に言語化する |
| 離 | コードを読んだ瞬間に全体の設計意図と問題点が見える。レビュー相手のレベルに合わせてコメントの粒度を変える。「今は指摘しない方がいい」という判断もできる | レビュー哲学について議論する。「良いレビューとは何か」を一緒に定義する。チームのレビュー文化を設計する視点を提供する |

**典型的な罠：**
- 偽の離：レビューをせずに LGTM を出す → 「超越」ではなく怠慢
- 早すぎる破：チェックリストを覚えていないのに「設計レベルの指摘」をする → 基本的な見落としが多い

## 段階の混在パターン

一人のエンジニアの例：

| 領域 | 段階 | 根拠 |
|------|------|------|
| Python | 離 | 10年の経験。言語の特性を深く理解し、プロジェクトに最適な書き方を自然に選択できる |
| Kubernetes | 守 | 最近触り始めた。YAML の書き方をドキュメントからコピペしている。なぜ Pod / Service / Deployment が分かれているか理解していない |
| TDD | 破 | テストファーストの原則を理解し、ほとんどの場面で適用できる。ただし統合テストの設計では「どこまでモックすべきか」の判断に迷う |
| コードレビュー | 離 | チームのレビュー文化を設計している。レビュー相手の成長段階に応じたフィードバックを自然に調整できる |

**指導のポイント：**
この人に対して「あなたは上級者です」と一律に扱うのは誤り。
Kubernetes については守の指導（型通りにやらせる）、
TDD については破の指導（原則の理解を深める）が必要。
