# Shu-Ha-Ri Skill

[English](README.md) | [日本語](README.ja.md) | **繁體中文**

診斷精通階段，並針對任何 skill 領域提供階段相應的指引。

## 三個階段

| Stage | 漢字 | 含義 | 行為樣態 |
|-------|------|------|---------|
| Shu | 守 | 守 / 跟隨 | 嚴格遵循規則。學習形式。不偏離。 |
| Ha | 破 | 破 / 適應 | 理解規則「為何」存在。依情境調整。 |
| Ri | 離 | 離 / 超脫 | 規則內化。自由揮灑。創造自己的道。 |

## Method Type

Framework-driven 診斷（不像 Socratic method 是 dialogue-driven）。
agent 透過行為性提問診斷使用者目前的階段，
再提供階段相應的指引與轉換訊號。

關鍵原則：階段是領域專屬的。一位實踐者可在某領域為 Shu，
另一領域為 Ri。

## 診斷訊號

| 訊號 | Shu | Ha | Ri |
|--------|-----|----|----|
| 規則 | 忠實遵循 | 理解並調整 | 無意識地做出最佳判斷 |
| 解題 | 查文件 / 指南 | 將原則套用到新問題 | 直覺抵達最佳解 |
| 教學 | 傳達規則 / 程序 | 解釋原則與道理 | 依學習者階段調整教法 |
| 例外 | 規則不適用時卡住 | 認出並調整規則 | 自然處理 |
| 創造力 | 在形式內精進 | 將既有形式組合出新做法 | 為情境創造原創形式 |

## SKILL.md 中的範例

| 範例 | Domain | 關鍵洞見 |
|------|--------|----------|
| TypeScript + TDD | Software Development | 兩個領域分別評估；TDD 因略過 Refactor 步驟被診斷為 Shu |
| Scrum 團隊輔導 | Agile Methodology | 團隊卡在 Shu 停滯；ceremony 照形式做但失去本意 |

## 常見陷阱

| Trap | 描述 | 對策 |
|------|------|------|
| 跳階段 | 未掌握 Shu 就跳到 Ri | 診斷缺口，回到適合的階段 |
| Shu 停滯 | 規則做得完美卻無法應變 | 引入「為什麼」提問，鼓勵實驗 |
| 假 Ri | 把無知誤認為超脫 | 用診斷性問題揭露缺口 |
| 一致性評估 | 將所有領域視為同一階段 | 永遠按領域逐一指定與評估 |

## 適用領域

| Domain | Shu 範例 | Ha 範例 | Ri 範例 |
|--------|-------------|------------|------------|
| 程式語言 | 遵循 style guide、套用標準 pattern | 知道權衡，依情境選 pattern | 直覺寫出 idiomatic 的程式 |
| 方法論（Agile、TDD） | 嚴格依 ceremony / cycle 執行 | 依團隊需求調整流程 | 體現原則而不拘泥流程 |
| 設計（UI/UX） | 遵循 design system 指南 | 知道何時可破例 | 培養出設計品味與直覺 |
| Code Review | 用 checklist、套用規則 | 評估取捨，提供情境感知的回饋 | 整體性地感受品質 |

## 文化起源

Shu-Ha-Ri 起源於日本武道（合気道，由遠藤征四郎發揚），
與日本透過嚴謹練習達到精通的傳統深刻相連。
這個概念與 Dreyfus 的 skill acquisition 模型有相通之處，
但更強調實踐者與規則的關係，而非認知階段。
