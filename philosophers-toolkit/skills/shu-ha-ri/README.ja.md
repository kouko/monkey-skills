# Shu-Ha-Ri Skill

[English](README.md) | **日本語** | [繁體中文](README.zh-TW.md)

任意の skill 領域における習熟段階を診断し、段階に応じた指導を行う。

## 3 つの段階

| Stage | 漢字 | 意味 | 行動様式 |
|-------|------|------|---------|
| Shu | 守 | 守る / 従う | 規則を厳格に守る。形を学ぶ。逸脱しない。 |
| Ha | 破 | 破る / 適応 | 規則の「なぜ」を理解する。文脈に合わせる。 |
| Ri | 離 | 離れる / 脱する | 規則を内面化。自在に動く。自分の道を創る。 |

## Method Type

Framework-driven 診断（Socratic method のような dialogue-driven ではない）。
agent は行動的な問いかけでユーザーの現在の段階を診断し、
段階に応じた指導と移行のサインを提供する。

基本原則：段階は domain ごと。同一の実践者がある領域で Shu、
別の領域で Ri ということもある。

## 診断サイン

| サイン | Shu | Ha | Ri |
|--------|-----|----|----|
| 規則 | 忠実に従う | 理解し適応する | 意識せず最適に判断 |
| 問題解決 | docs/ガイドを参照 | 原則を新しい問題に適用 | 直感で最適解に到達 |
| 教える | 規則・手順を伝える | 原則と理由を説明 | 学習者の段階に合わせ指導 |
| 例外 | 規則が当てはまらないと止まる | 規則を認識し調整 | 自然に対応 |
| 創造性 | 形の内で改善 | 既存の形を組み合わせ新しい方法へ | 状況に応じた原型を創る |

## SKILL.md 中の例

| 例 | Domain | 主な洞察 |
|------|--------|----------|
| TypeScript + TDD | Software Development | 2 つの領域を別々に評価；TDD は Refactor step を省略するため Shu と診断 |
| Scrum チームコーチング | Agile Methodology | チームが Shu の停滞；ceremony は形式通りだが目的を失っている |

## よくある罠

| Trap | 説明 | 対処 |
|------|------|------|
| 段階飛ばし | Shu を習熟せず Ri に跳ぶ | ギャップを診断し適切な段階に戻る |
| Shu の停滞 | 規則を完璧に守るが応用できない | 「なぜ」の問いを導入、実験を促す |
| 偽 Ri | 無知を超脱と勘違い | 診断質問でギャップを露呈させる |
| 一律評価 | 全 domain を同一段階と扱う | 必ず domain ごとに指定し評価する |

## 適用 domain

| Domain | Shu の例 | Ha の例 | Ri の例 |
|--------|-------------|------------|------------|
| プログラミング言語 | style guide に従い、標準 pattern を使う | trade-off を知り、文脈で pattern を選ぶ | 慣用的なコードを直感で書く |
| 方法論（Agile、TDD） | ceremony / cycle を厳密に守る | チーム事情に合わせて適応 | 厳格な手続なしに原則を体現 |
| デザイン（UI/UX） | design system のガイドラインに従う | いつ破ってよいかを理解 | 美的感覚と直感を養う |
| Code Review | checklist を使い規則を適用 | trade-off を評価、文脈に応じた feedback | 品質を全体で感じ取る |

## 文化的起源

Shu-Ha-Ri は日本武道（合気道、遠藤征四郎を経由）に由来し、
規律ある稽古を通じた習熟という日本の伝統と深く結びついている。
この概念は Dreyfus の skill acquisition モデルと並行するが、
認知段階ではなく、実践者と規則との関係を強調する点が特徴。
