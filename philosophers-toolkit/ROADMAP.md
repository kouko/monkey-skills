# Philosophers Toolkit — Roadmap

哲學家/理論の索引。各項目は将来 skill として実装予定。
優先度は使用者の工作流（企画 → 規格 → 實作 → 迭代）との関連性で判断。

## 問題釐清 / 定義（What & Why）

### P1: 亞里斯多德 — 四因說 + 第一原理

- **四因說（Four Causes）**: 質料因・形式因・動力因・目的因
  - 適用：理解一個系統/產品「是什麼」「為什麼存在」
  - Skill 方向：引導使用者從四個角度分析問題

- **第一原理（First Principles）**: 分解到不可再分的基本真理，從那裡重建
  - 適用：從根本重新思考問題，避免類比思維的陷阱
  - Skill 方向：系統性地質疑現有假設，找到真正的基本事實

### P4: 笛卡兒 — 方法性懷疑（Methodical Doubt）

- 系統性質疑所有假設，只保留無法懷疑的部分
- 適用：驗證前提是否成立，避免建立在錯誤假設上
- 與蘇格拉底提問法的差異：笛卡兒更系統化（有步驟），蘇格拉底更對話式
- Skill 方向：提供結構化的懷疑清單

### 已整合：蘇格拉底 — 提問法

- 已部分整合在 `domain-teams` 的 `planning-brainstorming.md`
- 六類問題：澄清、假設、理由/證據、觀點、影響/後果、反思
- 不需要獨立 skill，但可以考慮做一個更完整的版本

## 分析 / 推論（How to Think）

### P2: 黑格爾 — 辯證法（Dialectics）

- 正題（Thesis）→ 反題（Antithesis）→ 合題（Synthesis）
- 適用：探索 trade-off、整合對立觀點、方案比較
- Skill 方向：強制使用者考慮反面論點，然後找到更高層次的整合

### P3: 波普 — 可否證性（Falsifiability）

- 好的假設必須可以被推翻；找不到反例才能暫時接受
- 適用：驗證產品假設、研究假說
- 與 Assumption Mapping 互補：AM 識別假設，波普教你怎麼測試它們
- Skill 方向：對每個關鍵假設設計「如何推翻它」的測試

### 奧卡姆剃刀（Occam's Razor）

- 最簡單的解釋通常是正確的；不必要的複雜性應該被削除
- 適用：工程設計中避免 over-engineering、debug 時找最簡單的原因
- 已隱含在 code-team 的 YAGNI 原則中，可考慮做更通用的版本
- Skill 方向：系統性地簡化方案

### 實用主義（Pragmatism）— 杜威 / 皮爾士

- 以實際後果評估想法：「如果這是真的，會有什麼實際差異？」
- 適用：技術決策、方案比較、避免純理論的空談
- Skill 方向：對每個選項列出具體的實際後果

## 決策 / 價值判斷（What Should We Do）

### 功利主義（Utilitarianism）— 彌爾

- 最大化整體效益：「哪個選擇對最多人產生最大好處？」
- 適用：功能取捨、優先排序、資源分配
- Skill 方向：量化每個選項的影響範圍和程度

### 康德 — 義務論（Deontological Ethics）

- 普遍化原則：「如果所有人都這樣做，世界會怎樣？」
- 適用：倫理決策、隱私/安全設計、API 設計原則
- Skill 方向：用普遍化測試檢驗設計決策

### 德行倫理（Virtue Ethics）— 亞里斯多德

- 「一個有德行的工程師/設計師會怎麼做？」
- 適用：品質標準、設計原則、code review 標準
- 與角色定義互補：team skill 的角色描繪就是一種德行倫理的應用

## 東洋哲學

### P5: 守破離（Shu-Ha-Ri）

- 守（遵守規則）→ 破（突破規則）→ 離（超越規則）
- 適用：學習曲線設計、技術成長路徑、skill 本身的使用方式
- 與 domain-teams 的設計哲學文化上契合（感性工学、無意識の設計）
- Skill 方向：評估使用者目前在哪個階段，給予對應的指導

### 三現主義（Genchi Genbutsu）

- 現場（去現場）・現物（看實物）・現実（面對現實）
- 適用：Context Discovery、使用者研究、debug
- 已隱含在 Context Discovery 的「explore what exists」中
- Skill 方向：更結構化的「去看、去摸、去確認」流程

### 道 — 無為（Wu Wei）

- 不強求、順應自然流向、最小干預
- 適用：系統設計中的最小干預原則、API 設計的慣例優先
- 與 code-team 的 YAGNI 和奧卡姆剃刀有精神上的共鳴
- Skill 方向：引導使用者找到「不需要做什麼」

## 優先實作順序

| 順序 | Skill | 理由 |
|------|-------|------|
| P1 | 亞里斯多德（四因說 + 第一原理） | 企画和問題定義的核心工具 |
| P2 | 黑格爾辯證法 | Trade-off 分析、方案比較 |
| P3 | 波普可否證性 | 假設驗證，跟 Assumption Mapping 互補 |
| P4 | 笛卡兒方法性懷疑 | 系統化質疑方法 |
| P5 | 守破離 | 文化契合、meta-level 的學習框架 |
| — | 其餘 | 視使用頻率和需求決定 |

## 設計原則（建 skill 時遵守）

- 每個 skill 應該是 **可操作的流程**，不是哲學課
- 輸出是「更好的問題定義」或「更全面的分析」，不是哲學論文
- 可以跟 domain-teams 的 brainstorming protocols 搭配使用
- 用使用者熟悉的語言，不預設對方懂哲學術語
