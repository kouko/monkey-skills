## Content Summary

#### 學習 AI Agent 的路線與時程

- 23 到 40 歲、有學習意願的人，一個月足以從零基礎逆袭成 AI 工程師
- 學習分三階段：打基礎（核心理論、框架模組）→ 強化學習（多智能體邏輯、Prompt 調整）→ 實戰（完成 2-3 個企業級專案）
- 課程涵蓋五大部分：API First、Agent 核心閉環、框架與提示詞、情境工程與多智能體、記憶管理與全鏈路調試

#### API First：從最簡單開始，拒絕過度設計

- 做 AI 應用最容易掉入的陷阱：一開始就堆砌複雜工具、寫幾千字提示詞、套用最複雜框架，導致系統失控
- 核心法則：拒絕過度設計——如果一個需求一次 API 呼叫就能解決，就絕對不要用 Agent
- 大模型本身具有隨機性與不確定性，在不穩定的基礎上過度堆疊功能，系統容易崩潰
- 反例：只需文字摘要的任務，卻裝上大腦讓它做計劃、選工具、再執行，屬於畫蛇添足

#### 真正的 Agent vs. 自動化工作流

- 確定性工作流（偽 Agent）：流程固定、自動執行，如視頻自動剪輯（轉錄→提取重點→剪輯），適合用拖拉式自動化工具即可
- 真正的 Agent：面對沒有標準答案的情況，能夠停下來主動提問，例如發現數據異常時反問確認，具有真正的「智能協同」能力
- Agent 最大價值：以自然語言作為入口解決「飛機駕駛艙困境」（軟體按鈕太多找不到），並處理難以用固定邏輯判斷的任務

#### Agent 核心閉環：規劃、記憶、工具

- Planning（規劃）：拆解複雜目標為步驟，並能自我反思後重新規劃
  - 例：接到「準備下週演講」，拆分為查資料 → 寫大綱 → 潤色 → 做 PPT
- Memory（記憶）：短期記憶（當前對話上下文）+ 長期記憶（使用者偏好，如老闆只看折線圖不看餅圖）
- Tools（工具）：賦予模型實際執行能力，如搜尋網頁、執行程式碼、讀取日曆
- 運作方式：理解需求 → 規劃步驟 → 調用工具 → 取得反饋 → 調整 → 再循環

#### 人機協同：人類絕不能缺席

- 真正的 Agent 不是全自動機器人，而是需要人類持續介入的協作者
- 人的責任：決策拍板、給予反饋、表達偏好
- Agent 的責任：執行、提出建議、生成初稿
- 設計 Agent 時必須留下讓人類介入的「氣口」，由人來承擔不確定性
- 很多需求一開始自己也描述不清，是在與 AI 反覆互動中才逐漸明確的

#### 框架選型陷阱：從 MVP 開始

- 常見誤區：不用重型框架（如 LangChain、LlamaIndex）就顯得不專業，實際上許多團隊為上框架花幾週配置環境，最簡單任務仍跑不通
- MVP 原則：早期只需最簡單 API + 最簡單狀態，跑通「用戶提問→執行→返回結果」的小閉環就足夠
  - 比喻：先學會騎自行車的平衡，不要一開始就裝 24 段變速器、GPS 和藍牙音箱

#### 提示詞陷阱：小步迭代，不要一次寫滿

- 常見誤區：把所有邊緣情況全塞進提示詞膨脹到幾千字，大模型注意力有限，核心指令反而被淹沒
- 比喻：在嘈雜菜市場講話，最重要的指令被噪音蓋掉
- 正確做法：從一句大白話開始 → 觀察輸出 → 發現問題 → 加一個約束 → 再觀察，逐步迭代
  - 例：「幫我寫個週報」→ 語氣太隨意 → 加「語氣要正式」→ 太長 → 加「字數限制在 500 字以內」

#### 情境工程與多智能體協作

- 情境工程本質：信息過濾漏斗，只提供 Agent 完成當前任務最必要的信息，濾掉噪音
- 複雜任務靠多智能體架構解決：拆成各司其職的子 Agent，而非造一個容易「精神分裂」的超級 Agent
  - 規劃者（Planner）：理解意圖、拆解任務、分配工作，不用自己執行
  - 程式碼 Agent：負責邏輯運算與 API 呼叫，完全不接觸設計需求
  - 設計 Agent：負責排版與視覺輸出
  - 搜尋 Agent：負責爬取最新網路資料
- 優勢：實現上下文物理隔離，各子 Agent 指令精簡、高度專注，是現今複雜業務應用的主流架構

#### 記憶管理：分層設計節省成本

- 錯誤做法：把所有歷史對話一股腦傳給模型，容易資訊過載且浪費 token
- 短期記憶：當前任務上下文，任務結束後清空，如便利貼草稿
- 長期記憶：存於資料庫，需要時只檢索文件 ID 或核心摘要，不重新讀取完整內容
- 分層設計可省下大量 token 費用，並讓系統響應速度提升數倍

#### 全鏈路調試：看過程，不只看結果

- 常見錯誤：只看最終輸出不對就罵模型笨或猛改提示詞，真正線索藏在中間過程
- 實例：發現某 Agent 連續調用同一搜尋工具 5 次，翻日誌才發現前一步結果根本沒被存下來
- 比喻：醫生要看病歷、血液檢測、X 光片，而不只看頭痛就開止痛藥

## Key Takeaways

- 拒絕過度設計：能用一次 API 呼叫解決的絕不上 Agent；在不確定基礎上過度堆疊，系統必然崩潰
- 真正 Agent 的特徵是「能應對變化、遇到不確定能主動提問」，而非固定流程的自動化工具
- Agent 核心三要素：Planning、Memory、Tools 構成不斷循環的閉環，缺一不可
- 人機協同是設計根本：必須在系統中留下人類介入的介面，由人承擔不確定性
- MVP 優先：框架與提示詞都從最小化開始，一步一步迭代，而非一次堆到最複雜
- 多智能體架構是解決複雜任務的關鍵，通過物理隔離各子 Agent 上下文，保持每個 Agent 精簡專注
- 記憶分層（短期 + 長期）大幅節省 token 成本，並提升系統響應速度
- 全鏈路調試比只看最終結果重要；問題根因往往藏在中間過程的執行日誌裡
