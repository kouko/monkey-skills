# Proposal — Merge two open tables into one check

**Seed:** "Let a restaurant server merge two open tables (each with in-progress
orders) into one check."

Generated by `spec-expansion` (GENERATE) → critiqued by `completeness-critic`.
This is a spec draft, not an implementation. Trust is earned by execution
(code-toolkit's VERIFY gate), not by a spec that looks finished.

## OOUX object model

Phase ② artifact — object inventory + per-object state machine. Fanned out
per object (parallel sub-agent fan-out, one per object).

### Object inventory

| Object | Role | Key relationships |
|--------|------|-------------------|
| **Table** | spatial/ownership unit; holds ≤1 active check | → Check, Seat, Server-of-record (section), Guest party, Reservation, Merge-group |
| **Check / Order** | financial+culinary container (the thing actually merged) | → Table (1→N after merge), Order Item, Seat, Payment, Discount, Server-of-record |
| **Order Item / Line** | one menu product at a qty, fired/held against seat+course | → Check, Seat, Course, Modifiers, Menu product, KDS ticket |
| **Seat** | the position namespace that **collides** on merge | → Table/Check, Order Items, Guest, per-seat Payment |
| **Payment / Tender** | money event; makes merge potentially **irreversible** | → Check, Seat/Item scope, Tender type, Processor txn, Refund |

### State machines (merge-focused)

**Table** — `AVAILABLE → OPEN → DIRTY → AVAILABLE`; `AVAILABLE → RESERVED → OPEN`;
`AVAILABLE/DIRTY → OUT_OF_SERVICE`. Only **OPEN** is mergeable. Ineligible:
AVAILABLE/RESERVED (no check), DIRTY (check already settled — reopening is a
void/comp risk), OUT_OF_SERVICE (not a real location), MERGED-source (check
already emptied → ghost double-merge).

**Check** — `open → has-fired-items → partially-paid → fully-paid → closed`;
`→ voided`; `comped` is a flag overlay; `closed → reopen` is manager-gated.
Eligible: open / has-fired-items (both ways). **partially-paid**: blocked as
source by default, allowed only as constrained target. **fully-paid / closed /
voided**: ineligible both ways (settled money; re-parenting corrupts the audit
trail).

**Order Item** — fire-state `HELD → FIRED/SENT → IN-PREP → DELIVERED`; `→ VOIDED`.
Merge is **orthogonal to cook state**: `ON_SOURCE_CHECK → ON_SURVIVING_CHECK`,
reversible only while `not SETTLED`. Merge **never re-fires** — it re-parents
the billing line; the KDS ticket is re-labeled to the survivor, never re-sent.

**Seat** — `EMPTY → ACTIVE`; on merge `ACTIVE → COLLISION_PENDING → RENUMBERED |
MERGED_INTO`. EMPTY seats drop silently. Collision strategies: (A) offset/
renumber (default, lossless), (B) merge-into (server asserts same guest),
(C) namespace by origin. Renumber must cascade to every child item's seat in
**one transaction**.

**Payment** — `AUTHORIZED → CAPTURED → (open batch: void/adjust) | (settled
batch: REFUND only)`; cash is captured-on-tender. Undo is cheap only while
every tender is pre-capture or open-batch; **one settled capture turns "undo"
into "refund"** (processor fee + asymmetric audit trail).

## Provenance

Every emitted item tagged `seeded` / `inferred` / `critic-found`.

**seeded** (in or directly entailed by the seed):
- Two open tables, each with in-progress orders, exist (the merge inputs).
- A server performs the merge.
- The outcome is **one** check.
- Both source tables are "open" (have active orders).

**inferred** (derived from OOUX/USM/lens priors, not in the seed):
- Source/target distinction; survivor keeps id/number; loser becomes a void tombstone.
- Fired kitchen items migrate without re-firing; KDS ticket re-labeled.
- Recalc order (subtotal → discount → service-charge/gratuity → tax) with single rounding.
- Seat-number collision resolution and per-item cascade.
- Eligibility gating on table state and check payment-state.
- Cross-section ownership / claim-or-override before merge.
- Auto-gratuity re-evaluation at the party-size threshold.
- Time-boxed undo of a merge.

**critic-found** (surfaced by the completeness-critic loop, re-seeded):
- Empty source check; off-by-one party at gratuity boundary; large-check latency.
- In-flight merge state; concurrent edit to a source mid-merge.
- Concurrency races (same table merged twice; chained-merge cycle); merge lock + idempotency key.
- KDS re-label network failure; crash mid-merge / partial write → atomic-or-rollback.
- PCI constraint on re-attaching captured card tenders.
- Immutable audit-log requirement for every merge.
- Fiscal/tax-law constraint on voiding a registered check (blind spot).
- Tip-attribution as a labor/wage policy (blind spot).

## Blind spots — needs human/field input

The critic can surface that these gaps **exist** but cannot manufacture the
missing business/domain fact. Each names the human/field source that closes
it. (`needs human/field input`)

1. **Fiscal/tax-law treatment of the voided source check** — In many
   jurisdictions a check that has been printed or registered with a fiscal
   device cannot be silently voided; it requires a credit note / regulated
   reissue. The true rule depends on the operating country and the certified
   POS. *Source: tax/fiscal-compliance review + local accountant.*
2. **Whether a partially-paid check may be merged at all** — Treated here as
   "target-only, default-blocked," but the real answer is an operator/finance
   policy and may be fiscally constrained. *Source: restaurant operations +
   finance/POS-vendor policy.*
3. **Tip-attribution rule when two servers own the two checks** — Single-owner
   reassign vs proportional tip-split map is a labor-law / wage / house tip-pool
   decision, not a technical one. *Source: management + payroll/labor-law review.*
4. **Auto-gratuity / mandatory-service-charge legality** — Whether a mandatory
   charge crossing the party-size threshold is treated as tip or wage, and
   whether it must be disclosed, is jurisdiction-regulated. *Source: legal +
   payroll review.*
5. **Currency rounding & tax-recompute rules** — The exact tax base, rounding
   direction, and discount-stacking policy are POS/jurisdiction-specific and
   must come from the certified tax engine, not LLM priors. *Source: POS tax
   engine spec + jurisdiction tax authority.*
6. **Undo time-window length & who may undo** — How long after confirm an undo
   is allowed, and whether it survives a settled payment batch, is a business
   policy. *Source: restaurant operations + payment-processor batch schedule.*
7. **Data-retention period for voided/merged check records** — Required
   retention for audit/tax is jurisdiction-specific. *Source: compliance +
   tax authority.*
8. **Real concurrency model of the deployed POS** — Whether terminals are
   online-authoritative, offline-tolerant with later sync, or hub-and-spoke
   determines the true race/partition behavior. *Source: POS-vendor
   architecture + field observation of the venue's network.*
