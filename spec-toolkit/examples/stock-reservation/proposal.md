# Stock reservation — spec-expansion proposal

**Seed:** "Let a warehouse reserve stock for a pending order before it ships."

Generated by the GENERATE layer (`spec-toolkit:spec-expansion` → `spec-toolkit:completeness-critic`). This is a high-recall spec draft, not a finished spec. Trust is earned by execution (code-toolkit's VERIFY gate), never by a draft that looks thorough. **Coverage is stated relative to the seed + the lenses applied — never "complete".**

---

## USM backbone

Phase ① artifact — the happy-path spine: ordered user-journey steps of who does what, in sequence.

| # | Actor | Step (CTA) | Object touched | Provenance |
|---|-------|-----------|----------------|------------|
| 1 | Order-management system (OMS) | Places / confirms a **pending order** | Order | `seeded` (order), `inferred` (OMS actor) |
| 2 | Reservation engine | Checks **available-to-promise** (ATP = on_hand − reserved) for each order line | StockItem | `inferred` |
| 3 | Reservation engine | **Reserves** the requested qty against a warehouse's stock (soft hold) | Reservation, StockItem | `seeded` |
| 4 | Reservation engine | Spills the unmet remainder to **backorder** if ATP < requested | Reservation, Order | `inferred` |
| 5 | Order / payment | **Confirms / hardens** the soft hold into a committed allocation at order-confirm or pick-start | Reservation | `inferred` |
| 6 | Warehouse picker | **Picks / packs** the committed units (draws the hold down) | Reservation, StockItem | `inferred` |
| 7 | Warehouse | **Ships** — the hold is consumed; on_hand and reserved both decrement | Order, StockItem | `seeded` ("before it ships" implies the ship boundary) |
| 8 | Scheduler / actors | Cross-cutting: a hold may **expire** (TTL), be **released** (cancel/payment-fail), or be **re-sourced** to another warehouse — any time before step 7 | Reservation | `inferred` |

The spine is **reserve → (backorder?) → commit → pick → ship**, with expire / release / re-source as off-spine exits before the ship boundary.

### State machines

**Reservation**

States: `requested`, `partially_reserved`, `reserved`, `backordered`, `committed` (hard), `picking`, `expiring` (grace), `consumed`/`shipped` (terminal), `released` (terminal), `expired` (terminal).

```
requested            -> reserved             (full qty available)
requested            -> partially_reserved   (subset available; remainder spills)
requested            -> backordered          (zero available; queued)
partially_reserved   -> reserved             (top-up: remainder replenished)
partially_reserved   -> backordered          (remainder spilled to backorder)
backordered          -> partially_reserved   (partial replenishment)
backordered          -> reserved             (full replenishment / promotion)
reserved             -> committed            (confirm/harden: soft -> hard)
reserved             -> reserved             (transfer/re-source; qty preserved)
committed            -> picking              (pick task started)
picking              -> consumed             (pick/pack/ship draws qty down) [TERMINAL]
committed            -> consumed             (direct ship, no discrete pick)  [TERMINAL]
* (pre-ship)         -> released             (cancel / payment-fail / manual) [TERMINAL]
requested|partially_reserved|reserved -> expired  (TTL lapse)                 [TERMINAL]
reserved|committed   -> expiring             (TTL hit while in active pick — grace, NOT auto-released)
```

Guarded / illegal (flagged edges):
- `consumed`/`shipped` is **absorbing** — no release/expire/reactivation; reversal is a new return/RMA object.
- `committed`/`picking` is **NOT expirable** — TTL applies only to soft holds; the expiry job MUST skip `hold_type = hard` (else it orphans a started pick).
- `expired` and `released` are **distinct terminals** — never collapsed (audit + re-promotion semantics differ).
- `reserved -> committed` requires **on_hand ≥ qty_reserved at commit time** unless an authorized oversell override is set.
- Concurrent `requested -> reserved` on the last unit must arbitrate via priority + optimistic-lock version — the canonical oversell race.

**Order** (header rollup over line states)

```
draft -> pending                       (place)
pending -> on_hold / on_hold -> pending (payment/credit hold ↔ release)
pending -> reserved                    (all lines fully allocated)
pending -> partially_reserved          (some lines short -> backorder)
partially_reserved -> reserved         (backorder filled)
reserved -> picking -> packed
packed -> partially_shipped            (split / partial ship)   <-- SHIP BOUNDARY
packed -> shipped                      (full ship)              <-- SHIP BOUNDARY
partially_shipped -> shipped
shipped -> closed ; partially_shipped -> closed (short-ship remainder cancelled)
pending|on_hold|partially_reserved -> cancelled (pre-ship; releases all reservations)
reserved|picking -> cancelled          (mid-reservation; releases held ATP, may need un-pick)
```

After the ship boundary there is **no whole-order cancel** — only remainder cancel (short-ship) or post-ship return.

**StockItem / Inventory** — two coupled lifecycles: bucket-flow + the contended available-counter.

```
(inbound) --receive--> AVAILABLE --reserve--> RESERVED --ship--> CONSUMED (out)
                          ^  |                    |
                  release |  | quarantine         | release (cancel/expire)
                          |  v                    v
                          QUARANTINE/HOLD       AVAILABLE
            scrap --> DAMAGED/EXPIRED --scrap--> WRITTEN-OFF (out)
   transfer: AVAILABLE@locA --> IN-TRANSIT --> AVAILABLE@locB
```

Boundary / hazard states: **zero-stock** (available=0 → reject/backorder), **exactly-enough** (available = requested → the last good reserve; next concurrent reserve must fail), **oversold/negative-available** (lost-update race or intentional policy), **adjustment-below-reserved** (a count sets on_hand < reserved → promised units physically gone → reservation invalidation/shortage workflow), **phantom/negative-on_hand** (distinct from oversold — recount vs backorder remediation).

Core invariants: `available = on_hand − reserved − quarantine − damaged`; `reserved ≥ 0`; `reserved ≤ on_hand` (violation = oversold); bucket transitions conserve units (move, never vanish except scrap/consume terminals).

**Warehouse / Location** — state gates whether a reservation can be **created** and **honored**.

```
OPEN  --freeze-for-count-->  FROZEN-FOR-COUNT  --unfreeze-->  OPEN
OPEN  --take-offline-->      OFFLINE           --recover-->   OPEN
OPEN | FROZEN | OFFLINE  --close-->  CLOSED  --reopen (if temporary)-->  OPEN
```

Gating: **OPEN** create ✅ / honor ✅ (subject to ATP + cutoff + capacity); **FROZEN-FOR-COUNT** create ❌ in frozen scope / honor ❌ in scope (existing holds wait or re-source); **OFFLINE** create ❌ / honor ❌ (open holds auto-re-source candidates); **CLOSED** create ❌ / honor ❌ (open holds MUST be released or re-sourced before close completes). Any state leaving OPEN converts open holds into a re-source decision, never a silent fail.

**`seeded`** (stated in or directly entailed by the seed):

- The reserve action; the ship boundary ("before it ships"); ATP draw-down on reserve; consume-on-ship.

## Blind spots — needs human/field input

The critic surfaced that these gaps *exist* but **cannot manufacture** the missing business-domain fact. Inventing a plausible value here is worse than naming the gap. Each names the human/field source that could close it.

1. **Real reserve-path latency / throughput SLA + peak contention target** (hot-SKU QPS during promotions) — *SRE/platform team + product owner.*
2. **Actual TTL / expiry duration(s)** and whether they differ by channel or order-type — *fulfillment ops + commerce product owner.*
3. **True oversell policy** — is oversell ever permitted, for which SKUs, by whom — *merchandising / ops leadership.*
4. **Reservation priority-tiering rules + re-arbitration policy** — which order classes preempt which, and whether a late high-priority order may claw back a reserved last unit — *business / commercial team.*
5. **FEFO vs FIFO vs LIFO lot-pick policy** and which SKU classes use which — *warehouse ops + QA/regulatory (perishables).*
6. **Paid-vs-unpaid reservation expiry rule** — may a paid order's hold ever expire — *finance / legal (exposure of taking payment then releasing stock).*
7. **Applicable legal jurisdiction(s) + consumer-protection regime** for promised-stock / misrepresentation — *legal counsel per market.*
8. **Restricted-SKU rules** — hazmat / age-restricted / controlled-substance / export-customs eligibility per destination — *compliance / trade-compliance team.*
9. **Data-retention period for released/expired reservations + PII classification** (GDPR/CCPA scope) — *data-governance / privacy officer.*
10. **Authorization matrix** — exact role → action mapping for reserve / override-oversell / release-another's-hold — *security/IAM owner + ops management.*
11. **Returns-restock routing policy** — restocked unit to free-pool vs head-of-backorder-queue — *fulfillment ops + customer-experience policy.*
12. **Audit / compliance obligations** — SOX/GxP-equivalent for override attribution and ledger immutability — *internal audit / compliance.*
13. **Authoritative clock / time-source for distributed TTL ordering** under node clock-skew — *platform / infra team.*
14. **Channel safety-stock partitioning rules** — how ATP is split across marketplace / DTC / wholesale pools — *merchandising / inventory-planning.*
