## Problem (Axis 1)

> **Job story** — "When my daily Claude Code session needs to perform a
> *write* op on any of the 6 OAuth-granted Google services (send mail /
> insert calendar event / write doc / append sheet / upload Drive file),
> I want the same first-party safety net I already get from
> [`safe-delete.sh`](../../gws-toolkit/scripts/gws/safe-delete.sh) +
> [`tag-create.sh`](../../gws-toolkit/scripts/gws/tag-create.sh) for
> Drive — dry-run preview + tiered confirm + provenance tag — not a 'go
> figure out the raw API params yourself' fallback."

Two upstream pressures converge:

1. **v0.7.0 only granted half the contract.** PR #318 (merged 2026-05-19)
   added `gmail` + `calendar` to OAuth grant (6 scopes total, all full
   r/w), but [`gws-toolkit/CHANGELOG.md`](../../gws-toolkit/CHANGELOG.md)
   §[0.7.0] §Notes openly states code paths stay read-only "until write
   JTBDs land in v0.7.x." v0.7.1 is that landing — the **first write
   JTBD lands now**, and the asymmetry between "OAuth grant covers send
   + insert" and "no first-party path to actually do it safely" must
   close.
2. **First-party write paths today only exist for Slides + Drive.**
   `safe-delete.sh` covers Drive delete; `tag-create.sh` covers Drive
   upload with provenance; `slides-builder/protocols/recipe-*` covers
   Slides batchUpdate. Gmail / Calendar / Docs / Sheets have **zero
   toolkit-original write code paths** despite their OAuth scope being
   full r/w since 2026-05-19. Closing that asymmetry is the load-
   bearing v0.7.1 job.

The job is **not** "build a Gmail/Calendar client" (that's what the
vendored gws-* skills cover read-side, what upstream's recipe-* skills
will cover composition-side in v0.8.0). It is **"give every write op the
same safety-net contract that v0.4.0 established for Drive."**

## Alternatives Considered (Axis 4 — research-grounded)

### Q1 — Scope split: one PR vs two-version vs per-service

Industry approaches found (via WebSearch 2026-05-20 + brainstorm):

1. **Single big v0.8.0** — Bundle all v0.7.x + v0.8.0 work in one PR
   - Pros: One coherent release; one onboarding spike
   - Cons: Repeats v0.7.0 PR #318's 14-commit / 2-round-review risk
     user flagged; whole-branch reviewer round count explodes; merge
     conflict surface large
   - Used by: monorepo "release train" pattern; rejected by user
     pushback ("強烈建議至少切兩半")

2. **Per-service patches** (v0.7.1-gmail, v0.7.2-cal, …)
   - Pros: Smallest atomic PRs; can pause mid-stream
   - Cons: 4× lock-step bump overhead; vendor expansion still has to
     wait for a separate v0.8.0; no clear "asymmetry closed" milestone
   - Used by: pip's pep-440 patch series for narrow bugfixes — but
     v0.7.1's surface is wider than a single bugfix

3. **Two-version split** — v0.7.1 = write asymmetry close-out; v0.8.0 =
   platform expansion (recipes/personas/services). ← **Chosen**
   - Pros: Each release scope is bounded (v0.7.1 ~13 atomic tasks;
     v0.8.0 ~15-20); clear separation between "close asymmetry" and
     "expand surface"; v0.7.1 ships fast (asymmetry has been carrying
     since 2026-05-19); v0.8.0 is purely additive (zero new toolkit
     logic — just vendor + router updates)
   - Cons: Two version bumps in close succession (acceptable — v0.6.0
     and v0.7.0 also shipped same-day 2026-05-19)
   - Used by: `gws-toolkit` v0.6.0 → v0.7.0 cadence

**My take**: #3 (Two-version split). **Decision accepted by user.**

### Q2 — v0.7.1 surface: which write helpers + wrappers

Industry approaches:

1. **5 helpers + 4 wrappers** (gmail-send + cal-insert + docs-write +
   sheets-append + drive-upload + 4 wrappers; Drive's tag-create.sh
   already covers Drive write — no new wrapper, but compose pattern
   needs documentation). ← **Chosen**
   - Pros: Covers all 5 write surfaces in one release; symmetric
     across services; validates safety pattern at scale (L1 / L2 / L3
     all exercised)
   - Cons: Larger v0.7.1 than minimum (~13 atomic tasks vs ~6 if
     calendar-only)
   - Used by: pattern-establishing release (matches v0.7.0's "do all
     of Gmail + Calendar at once" precedent)

2. **Calendar-only first** (validates L2 dry-run pattern; Gmail/Docs/
   Sheets slip to v0.7.2)
   - Pros: Smallest v0.7.1; L2 is the "median" tier — proves the
     contract before L3 (gmail) and L1 (sheets)
   - Cons: Multiple v0.7.x releases (lock-step bump overhead repeats);
     asymmetry remains for 3 services for weeks

3. **Gmail-only first** (highest user JTBD)
   - Pros: Validates L3 typed-recipient pattern; user gets the most-
     painful asymmetry closed first
   - Cons: L3 is the highest-risk tier — riskier to validate first;
     Calendar/Docs/Sheets asymmetry remains

4. **5 + reply/reply-all/forward** (8 helpers, 5 wrappers)
   - Pros: Covers more daily Gmail JTBD (reply is more common than
     compose-from-blank)
   - Cons: 3 more wrapper variants that compose `gmail-confirm-send.sh`;
     bloats v0.7.1; better to validate L3 contract with 1 wrapper
     first, then batch the variants in v0.8.0

**My take**: #1 (5 helpers + 4 wrappers). **Decision accepted by
user.** `gmail-reply` family goes to v0.8.0 as a `gmail-confirm-reply.sh`
that composes the v0.7.1 `gmail-confirm-send.sh`.

### Q3 — Safety tier mapping (per op)

Industry findings (WebSearch EN + JA 2026-05-20):

- **Slack API** — least-privilege scope per write op (`chat:write`,
  `files:write`); no specific typed-name confirm pattern documented,
  but emphasizes audit logging + idempotency. Source:
  [Slack OAuth scopes](https://api.slack.com/scopes).
- **Microsoft Graph CLI** (`mgc`) — **deprecating Sept 2025, retiring
  Aug 2026**. Not a strong reference for new design. Source:
  [MS Graph CLI retirement](https://devblogs.microsoft.com/microsoft365dev/microsoft-graph-cli-retirement/).
- **Salesforce CLI `sfdx`** — **deprecated** in favor of `sf`. Not a
  strong reference. Source: [Salesforce CLI command reference](https://developer.salesforce.com/docs/atlas.en-us.sfdx_cli_reference.meta/sfdx_cli_reference).
- **AWS CLI** — `aws s3 rm` requires `--recursive` + supports
  `--dryrun`; no typed-name pattern (relies on least-privilege IAM).
- **GitHub CLI** — `gh repo delete` requires typing `<owner>/<repo>`
  exactly (typed-name pattern, mirrors GitHub web UI).
- **Heroku CLI** — `heroku apps:destroy` requires `--confirm <app-name>`.
- **kubectl** — `kubectl delete --dry-run=client` standard.
- **JA (Claude Code 破壊的操作 best practice)** — auto-mode classifier
  blocks destructive ops by default; least-privilege scope at OS layer.
  Source: [Claude Code best practices JA](https://www.pnkts.net/2026/01/22/claude-code-best-practices-ja).

**My take** (decision table for v0.7.1):

| Op | Tier | Why |
|---|---|---|
| `gmail messages.send` | **L3 typed recipient list + subject** | Irreversible after Gmail's ~30s undo window; password-reset surface; AWS / GH / Heroku precedent for typed-confirm on irreversible ops |
| `calendar events.insert` with attendees | **L2 dry-run preview + --confirm** | Sends invite emails (visible to others); event itself editable / deletable; L2 matches the "destructive but recoverable" tier per `safe-delete.sh` §L2 |
| `calendar events.insert` solo | L1 dry-run | Just blocks own time; lowest blast radius |
| `sheets values.append` | L1 dry-run | Reversible via Sheets undo + version history |
| `sheets values.clear` / `update` overwrite | L2 preview + --confirm | Destructive but recoverable via version history (30-day window) |
| `docs append text` / `replaceAllText` | L1 dry-run + match-count warning when `replaceAll` match-count > 5 | Drive version history covers; broad-stroke replaceAllText flagged for review |
| `drive files.create` (upload) | L1 (existing `tag-create.sh`) | Already covered v0.4.0; v0.7.1 only adds documentation linking it to the new safety taxonomy |
| `drive files.delete` | L1/L2/L3 (existing `safe-delete.sh`) | Already covered v0.4.0 |

### Q4 — gws-drive-upload composition with existing tag-create.sh

Question: do we vendor `gws-drive-upload` (upstream's drive upload
helper skill) when `tag-create.sh` already wraps `drive files create`?

**Yes — vendor it; both serve different roles**:

- `gws-drive-upload` (vendored, upstream) — read-only **SKILL.md
  reference** documenting upload params + composition patterns. Claude
  consumes it for orchestration decisions.
- `tag-create.sh` (toolkit-original) — **executable shell wrapper** that
  injects `appProperties.created_by = "gws-toolkit"` provenance. Shell-
  invoked by `slides-builder`'s image-upload recipe and any future
  upload caller.

The two are orthogonal: SKILL.md guides Claude, shell wraps the API
call. v0.7.1 adds a 1-line note in `tag-create.sh` docstring + in
`skills/using-gws-toolkit/SKILL.md` routing table cross-referencing the
two so users understand "read `gws-drive-upload` for params; invoke
`tag-create.sh` for the provenance-tagged execution."

No new safety wrapper is needed for upload — `tag-create.sh` already is
L1 (no destructive surface; upload always creates a new file, never
overwrites; Drive de-dupes by ID not name).

### Q5 — 2026-05-01 Gmail / Calendar API quota change

Critical timing finding from Axis 4b WebSearch:

- **Gmail API + Calendar API quotas updated 2026-05-01** (active 19
  days as of 2026-05-20)
- Projects with usage Nov 2025 – Apr 2026 → **grandfathered** to old
  quotas
- Projects created on/after **2026-05-01** → new quota schedule + **
  billing later in 2026** ("at least 90 days' notice before changes
  take effect"; source [Calendar API release notes](https://developers.google.com/workspace/calendar/release-notes))
- Gmail per-user limits: 250 quota units/user/second moving average;
  1B/day project; 2,000 messages/day per Workspace user; 500/day per
  trial user; 500 recipients/message
- Calendar per-project limit: 1M queries/day; 403/429 rate-limit
  errors require exponential backoff (matches existing `gws-wrap.sh`
  retry policy 5s/10s/20s)

**Impact on v0.7.1**:

- kouko's GCP project was created during slides-toolkit Phase 1 (late
  April 2026; pre-2026-05-01) → **grandfathered**
- New marketplace users post-2026-05-01 → new quota schedule + billing
  later
- `skills/gws-setup/SKILL.md` MUST add a sub-section noting this for
