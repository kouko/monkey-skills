## Problem

**Job-To-Be-Done** (Klement format):

> When I'm at work and want to use `gws-toolkit` against my `@acme.com` Workspace account (not personal `@gmail.com`), I want `auto-setup` to pick the **Internal-app** path automatically, so I can skip the Test User step that doesn't apply to org-internal apps and avoid the External + Testing path's **7-day refresh token treadmill**.

**Why now**: `gws-toolkit` MVP locked itself to personal `@gmail.com` accounts (SKILL.md line 9-10: *"MVP scope is personal @gmail.com accounts on macOS. Workspace accounts, Linux, and CI are Phase 2+"*). User has live Workspace account (`kouko@acme.com` at Acme) and can't dogfood gws-toolkit against work data.

**Underlying Google OAuth facts** (verified via WebSearch 2026-05-19):
- External + Testing → 7-day refresh token expiry (Google policy, no override short of Production verification)
- **Internal** → no 7-day limit; scope review skipped; simpler consent UX
- Internal requires Google Workspace org (non-`@gmail.com` domain implies Workspace in ~99% of cases)

## Users

**Sole primary user**: kouko (macOS, Asia/Taipei).
- Has `kouko@example.com` (personal — current v0.5.2 supports)
- Has `kouko@acme.com` (Acme Workspace — currently blocked)

**Future expansion** (out of v0.6.0 scope): anyone with a Workspace org. Org admin features (shared projects, billing, scope review) → Phase 3+.

## Alternatives Considered

(WebSearch grounded — bilingual EN + JA per Axis 4 protocol)

**Approach 1: Auto-detect by email domain (CHOSEN)**
- `gcloud auth list` active account → email → if `@gmail.com` → personal/External path; else → Workspace/Internal path
- Pros: zero user-side input, matches the 99% case (Workspace = non-gmail domain in practice)
- Cons: edge cases (Workspace org using `@gmail.com` sub-domain delegation — extremely rare; personal Gmail used in Workspace org — same)
- Sources:
  - [Google: Configure OAuth consent and scopes (EN)](https://developers.google.com/workspace/guides/configure-oauth-consent) — confirms Internal requires Workspace org
  - [Google: Manage App Audience (EN)](https://support.google.com/cloud/answer/15549945?hl=en) — Internal vs External user-type semantics
  - [Google OAuth 2.0 docs §"Refresh token expiry" (EN)](https://developers.google.com/identity/protocols/oauth2) — confirms 7-day = External + Testing only

**Approach 2: `--audience=internal|external` CLI flag (RETAINED AS ESCAPE HATCH)**
- User explicitly passes audience; no domain inference
- Pros: precise; supports edge cases without re-detect logic
- Cons: extra cognitive load; mistakes possible
- **Decision**: KEEP as override on top of Approach 1 — auto-detect for default UX, flag for edge cases

**Approach 3: Interactive `gcloud auth list` picker at script start**
- Script enumerates all authed accounts, prompts user to select
- Pros: handles multi-account state; explicit per-run
- Cons: more interactivity; complicates idempotence; user already at consent screen later anyway
- **Rejected**: belongs to a separate `--switch-account` slash command if ever needed; v0.6.0 keeps the "active account = chosen account" semantics

**JA-search empty finding**:
> Searched 「Google Workspace 内部 OAuth アプリ 認証 設定 7日 リフレッシュトークン Qiita Zenn」 — **0 relevant native JA-language hits** on Qiita / Zenn / personal tech blogs for this specific topic. The JA-language web has Google's official help-center JA translations + a few Workspace-admin-blog entries, but no implementation-detail tutorials. This is a finding: the Internal-vs-External OAuth distinction is primarily English-documented; JA-side users likely rely on Google's official JA docs ([Workspace 管理者 ヘルプ](https://support.google.com/a/answer/14114704?hl=ja)) rather than community blogs.

**My take**: **Approach 1 (auto-detect) + Approach 2 (flag override)** as a composed default. Approach 1 handles the 99% case for zero-friction UX; Approach 2 is the escape hatch documented in failure-modes.
