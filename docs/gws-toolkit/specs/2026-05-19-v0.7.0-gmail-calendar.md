## Problem (Axis 1)

> **Job story** — "When my daily Claude Code session needs Gmail or
> Google Calendar context (read my inbox, find a free slot, check today's
> agenda), I want **one** OAuth client / one consent screen / one re-auth
> ritual that already covers it, so I don't burn a second GCP project
> + a second OAuth client + a duplicate `gws` binary install just to add
> 2 Google APIs to my workflow."

Two upstream pressures converge:

1. **collab-toolkit v0.2.0 left two orphan responsibilities.** The
   late-pivot 2026-05-19 (`collab-toolkit/CHANGELOG.md` §[0.2.0]) moved
   Gmail + GCal **out** of collab-toolkit on the explicit promise that
   `gws-toolkit` would absorb them — "the 4 Gmail hero protocols and 4
   Calendar hero protocols documented in v0.1.6 will re-emerge as
   `gws-toolkit` skills in a follow-up effort." This is that follow-up.
2. **gws-toolkit v0.6.0 just unlocked the Workspace-account path.** With
   `detect_account_type()` + Internal-audience walkthrough now live
   (`gws-toolkit/CHANGELOG.md` §[0.6.0]), Workspace users (kouko on
   `@acme.com`) get a Gmail/Calendar OAuth flow that does NOT
   require Google's restricted-scope security assessment (Internal-app
   exemption) and does NOT hit External+Testing's 7-day refresh expiry.
   Personal-Gmail users on External+Testing still get the same path
   v0.6.0 already paved for the 4 Slides/Drive/Docs/Sheets scopes.

The job is **not** "give me a full Gmail/Calendar client." It is
"close the orphan responsibility while preserving the v0.6.0 dual-path
contract and the read-only charter."

## Alternatives Considered (Axis 4 — research-grounded)

### Q1 — Scope tier: read-only vs read-write vs full

Industry approaches found (3, via WebSearch EN + JA 2026-05-19):

1. **Least-privilege read-only** — Google Calendar API docs §"Choose
   scopes" (EN). `calendar.readonly` is least-privileged and
   *sensitive* (not restricted); `gmail.readonly` is least-privileged
   for Gmail but Gmail tags all data-access scopes as *restricted*
   regardless of read/write.
   - Pros: ASVS V1 least-privilege at scope boundary; smallest blast
     radius.
   - Cons: **Departs from v0.4.0 precedent** — the toolkit deliberately
     enforces least-privilege at app layer, not scope boundary (see
     `CHANGELOG.md` §[0.4.0-strangler-fig-seed] "OAuth scope upgraded
     from `drive.file` to `drive`; safe-delete.sh replaces the implicit
     guarantee that drive.file provided"). Picking readonly for
     Gmail/Calendar would break the cross-service consistency that
     v0.4.0 established for the existing 4 APIs.
   - Used by: every read-only Gmail/Calendar integration on the Google
     "Choose scopes" recommendation page.

2. **Modify scope (`gmail.modify` + `calendar.events`)** — Google
   guidance + Japanese Qiita/Zenn integration writeups (JA).
   - Pros: Hedge against Workspace admins that block full Gmail OAuth;
     still unlocks future label/archive JTBD without v0.7.x re-consent.
   - Cons: Half-measure — neither matches v0.4.0 precedent (which is
     full) nor matches a true least-privilege stance (which is
     readonly). Inconsistent rationale.
   - Used by: Mercari/Cookpad-style internal tooling articles surfaced
     in the JA search.

3. **Full scope (`gmail` / `calendar`)** — Google "Choose scopes" docs.
   - Pros: **Matches v0.4.0 precedent exactly** — every existing
     `gws-toolkit` service holds full scope today (`presentations` /
     `drive` / `documents` / `spreadsheets`). Future v0.7.x write
     features (send mail, create event) require only a new vendored
     skill + app-layer safety wrapper — **no OAuth re-consent**. Same
     restricted-tier as existing `drive` (zero new tier exposure).
   - Cons: Larger blast radius if `~/.config/gws/client_secret.json`
     or token leaks — attacker would get full Gmail (password-reset
     surface) in addition to existing full Drive (already P0 if
     leaked). Risk asymmetric vs Drive (email = password reset), but
     leak event itself is already P0.
   - Used by: every existing scope in this toolkit since v0.4.0.

**My take (given your context):** Recommend **#3 (full read-write)**.

The v0.7.0 brief's "read-only default unless strong demand" phrasing
initially biased me toward #1; user pushback during brainstorming
surfaced the real principle: **v0.4.0 deliberately moved least-privilege
from scope boundary to app layer** (see `safe-delete.sh` three-tier
delete + `tag-create.sh` provenance tagging). Choosing readonly for
Gmail/Calendar would silently reverse that architectural decision
without justification. The "strong demand" that v0.7.0 brief was waiting
on is **the consistency principle itself**.

**Workspace admin policy probe (2026-05-19, real-machine, Acme account):**
User opened [myaccount.google.com/permissions](https://myaccount.google.com/permissions)
on `kouko@acme.com` and confirmed macOS Mail (system-level
OAuth) already holds `https://mail.google.com/` — the broadest possible
Gmail scope (read / compose / send / permanently delete). Since Acme's
Workspace admin allows that scope for a 3rd-party-style integration, the
narrower `https://www.googleapis.com/auth/gmail` (Gmail API full, what
v0.7.0 will request) will not be blocked. The previously-open admin-
policy concern is therefore resolved in favor of #3 — no hedging
needed.

**Conditional reversal:** if a future dogfood event surfaces a token-
leak or a Workspace-admin-policy change that walks back this stance,
the only edit is the 4 scope strings in `auto-setup.sh` + `refresh-auth.sh`
+ `gws-login.sh` + the 5 doc-narrative mentions of the scope set. Two
hours of work; no schema change.

(Note for brief consumers: `gmail` and `calendar` are both *restricted-
tier*. Same tier as existing `drive`. Tier matters only for
External+Production publishing — neither the personal-External-Testing
path nor the Workspace-Internal path triggers Google's restricted-scope
security assessment.)

### Q2 — Pure-vendor vs minimal toolkit-original wrappers

Industry approaches found (3):

1. **Pure-vendor only** — strict upstream-SKILL-only consumption (the
   v0.4.0 strangler-fig pattern already used for `gws-shared`/`gws-drive`/
   `gws-docs`/`gws-slides`/`gws-sheets`).
   - Pros: Zero-maintenance — upstream owns the SKILL surface; provenance
     auto-injected by `sync-upstream-skills.sh`; one-line UPSTREAM_GWS_VERSION
     bump rolls in upstream improvements; no risk of toolkit-original
     wrappers drifting from upstream API surface.
   - Cons: Less opinionated routing — user / Claude must understand
     which of the granular sibling skills to invoke (`gws-gmail-read` vs
     `gws-gmail-triage`).
   - Used by: gws-toolkit Phase 1 (current state).

2. **Pure-vendor + router-only wrapper** — vendor everything, add only
   a routing-table row in `using-gws-toolkit/SKILL.md`.
   - Pros: Same maintenance profile as #1 + closes the discoverability
     gap; matches how `gws-slides`/`gws-drive` are surfaced today.
   - Cons: None measurable for v0.7.0.
   - Used by: Implicit pattern from v0.4.0 onward (every vendored skill
     gets a routing-table row).

3. **Toolkit-original `gmail-automate` / `gcal-automate` wrappers** —
   port the 8 collab-toolkit v0.1.6 hero protocols verbatim into
   first-party SKILL.md files on top of the vendored skills.
   - Pros: Identical user-facing surface to retired v0.1.6.
   - Cons: ~16 new files to write + maintain, most of which (6/8) just
     re-frame an upstream skill that already exists. Direct violation
     of YAGNI + `dev-workflow:complexity-critique` deletion-first
     question 1 ("smallest end state?"). Compounds drift risk every
     upstream version bump.
   - Used by: Nobody yet — this is the costliest option v0.7.0 could
     pick. Documented to make the rejection visible.

**My take:** Recommend **#2 (pure-vendor + router-only)**. Matches
v0.4.0 precedent exactly; closes the discoverability gap; defers
wrapper work to v0.7.x only if dogfood proves it's needed (RIA YAGNI).

### Q3 — Workspace vs personal — does the v0.6.0 dual-path still hold?

Industry findings (no real "alternatives" — the question is "did Google
shift policy?"):

- **Confirmed unchanged for 2026-05-19** (EN search + JA search agree):
  External+Testing still issues 7-day refresh tokens unless scopes are
  the openid/email subset; Internal apps still exempt from the 7-day
  policy + restricted-scope security assessment.
- **One delta worth surfacing in v0.7.0 docs**: Gmail + Calendar
  restricted scopes are subject to Workspace **admin OAuth-app
  policies**. A Workspace org admin can block third-party OAuth clients
  from accessing Gmail/Calendar at the org level — independent of
  Internal-app exemption. v0.7.0 Workspace path SHOULD add a one-line
  note: "If you see `access_denied` after consenting, your Workspace
  admin may have OAuth-app restrictions on Gmail/Calendar scopes."
  Zero code change — pure SKILL.md narrative.

**My take:** Recommend **no auto-setup branch-logic changes**. v0.6.0's
`detect_account_type()` + `step_5a` dual-path already routes correctly
for the 2 new scopes. Add the admin-policy note in `skills/gws-setup/SKILL.md`
error table + `commands/gws-setup.md` troubleshooting block.
