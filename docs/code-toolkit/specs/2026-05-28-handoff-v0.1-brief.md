## Problem

**JTBD**: When I (the human user) am ending a Claude Code session — either voluntarily ("today's work is done, picking up tomorrow") or forcibly (context window near full, auto-compaction imminent) — and the NEXT session will need to pick up the work from a cold start (different machine, different tool, different time, different agent instance), I want the current session to write a **structured handoff file** that the new session can load to reconstruct enough context to continue without losing decisions, dropped paths, in-progress work, file paths, or my stated intent — and **without** the next agent making up answers to questions the previous session already settled.

The failure mode this skill exists to prevent: cross-session context loss producing one of three patterns — (a) **silent intent loss** (the new agent doesn't know about a constraint I stated 3 hours ago in the previous session), (b) **re-litigated decisions** (the new agent picks an approach we already rejected because it has no record of the rejection), (c) **wrong-state assumptions** (the new agent assumes the repo / branch / DB / external service is in a state it isn't, because the handoff doc went stale).

## Users

- **Primary (v0.1 dogfood)**: kouko himself, ending a Claude Code session that took >30 min and produced work the next session will continue. Typical scenarios:
  - End-of-day "wrap up before bed"
  - Switching from a feature branch to a production-bug fix; coming back later
  - Long research session that spans multiple days
  - Context window near 70%+, voluntary `/handoff` invocation before forced auto-compaction
- **Secondary (v0.2+)**: same user across different tools (Claude Code → Codex / Cursor / Gemini CLI) — the handoff markdown is cross-tool readable; per-tool init prompt mechanics differ
- **Tertiary (v0.3+)**: other monkey-skills users with similar long-running engineering work patterns
- **Reader of the HANDOFF.md file (CRITICAL)**: a **cold AI agent in a new session** — not the human. This is the L2 audience the synthesis research distinguishes from L3 (recap, human reader). Audience determines what verbosity wins:
  - L2 wants **maximum verbatim user intent** (no compression — verbatim user messages is the rule, not the exception, opposite of recap)
  - L2 wants **explicit state assertions** (git SHA, file mtimes, env hashes, test counts) so it can verify before acting
  - L2 wants **technical precision** over plain-language (the reader IS a tool; jargon is fine if accurate)
- **User non-goal**: NOT in-session re-orientation (that's recap, L3, already shipped). NOT context compression to save tokens (that's Claude Code's built-in `/compact`, L4). NOT agent-to-agent task delegation (that's OpenAI Agents SDK handoff / LangChain handoff_tool, different problem).

## Smallest End State

**One-paragraph summary**: User invokes the skill (typed `/handoff` OR natural-language triggers: *"wrap up"* / *"save state"* / *"今日はここまで"* / *"明天繼續"*) → SKILL.md instructs the agent to **use its built-in Bash + Write tools** to gather git state (`git rev-parse HEAD`, `git status`, `git log -5`), inspect tool inventory (`claude --version`), then **author a markdown file directly** at `.claude/handoffs/HANDOFF-YYYY-MM-DD-HHMMSS-<slug>.md` (Claude Code community de-facto path, shared by 5+ plugins) following a v0.1-trimmed schema (10 blocks, see below). NO Python scripts ship with v0.1 — agent does file I/O via its native Bash/Write tools per pure-prompt-template pattern (lesson from recap v0.1; sister skill is 0-LOC same shape). User starts next session in any tool, types `/handoff resume` (or natural-language "load the latest handoff" / "pick up where we left off") → SKILL.md instructs the agent to find latest `.claude/handoffs/*.md` (`ls -t | head -1`), Read the file, **run each Verification Command via Bash**, parse outputs against expected results in the file, then summarize back what it understood and ask user to confirm-or-redirect before acting (soft-gate Synthesis-check pattern, same as recap).

**v0.1 schema (10 blocks — trimmed from research's full 14)** — reuses recap bundle's L2 column where applicable:

| # | Block | Purpose | Source |
|---|---|---|---|
| 1 | Frontmatter (`handoff_id`, `created`, `trigger_type`, `chain.continues_from`, `git.{branch,head,dirty}`, `tools.{claude_code,etc}`) | Machine-parseable state snapshot | Research §Schema v1.1 frontmatter |
| 2 | Situation | What we are doing right now | Bundle L2 |
| 3 | Background | Key decisions + rejected options + critical file paths (verbatim quotes) | Bundle L2 |
| 4 | All user messages (verbatim) | Cold-reader anti-drift; L2 audience NEEDS this (opposite of recap L3) | Bundle L2 / Research §all-user-messages |
| 5 | Recent decisions + WHY | Why-we-chose-X-over-Y, surviving 6 months | Research §3 / §all-user-messages |
| 6 | Pending work | Tasks not yet done | Bundle L2 |
| 7 | Critical files | Path + why-it-matters + recently-modified | Research §7 |
| 8 | Do Not Touch | Explicit guardrails — "this looks like code smell but has external constraint" | Research §10 |
| 9 | Verification commands | Runnable assertions the new session executes before continuing (git log -3, pytest -k, ls -la) | Research §12 |
| 10 | Confidence flags | Per-block ✅⚠️❓ so cold reader knows which sections to trust vs question | Research §13 |

**Deferred to v0.2** (research v1.1 has these but v0.1 drops):
- Block 0 Interruption Snapshot (forced-only; v0.1 = voluntary-only)

**Success criteria**:
1. **Schema fires consistently** — every voluntary handoff produces the 10-block file; no free-form output
2. **Verbatim user messages preserved** — Block 4 lists every user turn verbatim, no LLM filtering (L2 audience rule)
3. **Verification commands runnable** — Block 9 commands are syntactically valid + runnable in a fresh shell
4. **Confidence flags meaningful** — Block 10 marks which sections are agent-inferred (⚠️) vs ground-truth-from-this-session (✅)
5. **Resume mode validates before acting** — `/handoff resume` runs Verification, surfaces mismatches, refuses to continue if mismatched (HARD-GATE)
6. **Cross-tool readable at v0.1** — markdown only; no Claude-Code-specific syntax; Codex/Cursor/Gemini can READ even if they can't yet WRITE via this skill

**Non-criteria** (explicitly NOT measured at v0.1):
- ❌ Forced-handoff auto-trigger (no context-pressure watcher)
- ❌ Per-tool init-prompt variants (Codex / Cursor / Gemini write paths defer to v0.2)
- ❌ Multi-handoff milestone rollup (chain has frontmatter pointer only; no rollup script)
- ❌ Plain-language principle (this is L2; technical precision wins; this is the major audience-inheritance flip from recap)

## Alternatives Considered

Industry approaches found (5, via WebSearch 2026-05-28):

1. **softaworks/agent-toolkit `session-handoff`** ([GitHub](https://github.com/softaworks/agent-toolkit/blob/main/skills/session-handoff/README.md)) — 10-block schema, CREATE MODE / RESUME MODE pattern, `.claude/handoffs/YYYY-MM-DD-HHMMSS-<slug>.md` path. **Closest prior art**. Used by: softaworks itself; widely starred. **Pros**: production-validated 10 blocks; chain link; resume verification baked-in; established the de-facto path convention `.claude/handoffs/` that 5+ community plugins now share. **Cons**: 10 blocks miss Confidence Flags + Do Not Touch.
2. **Don't Sleep On AI 8-Section Template** ([source](https://www.dontsleeponai.com/handoff-prompt)) — 8 blocks including unique Confidence Flags + Do Not Touch + How to Think About sections; static template (not a skill, just a prompt to copy-paste). **Pros**: confidence flags + guardrail blocks are best-in-class; clear scoring system. **Cons**: no machine-parseable frontmatter; no chain; not file-based.
3. **dotbabel `/handoff`** ([DEV.to article](https://dev.to/kaiohenricunha/dotclaude-handoff-portable-context-across-claude-code-codex-copilot-cli-and-gemini-cli-3733)) — **new since research v1.1**; explicit cross-tool portability angle (Claude Code / Codex / Copilot CLI / Gemini CLI all read the same `.dotbabel/handoff/*.md`). **Pros**: cross-tool first-class; validates the portability story in HANDOFF research. **Cons**: lighter schema (~6 blocks); no verification commands; no confidence flags.
4. **yan5xu/code-relay** ([GitHub](https://github.com/yan5xu/code-relay)) — **new since research v1.1**; Chinese-ecosystem framework, "零損失上下文傳遞", multi-repo scope. **Pros**: emphasizes "已做 / 進行中 / 下一步 / 注意事項" structured continuity; multi-repo aware. **Cons**: zh-CN docs primarily; heavier framework (full "工作協議"); v0.1 too narrow to absorb.
5. **OpenAI Agents SDK `handoff_<id>_tool`** ([docs](https://openai.github.io/openai-agents-python/handoffs/)) — agent-to-agent in-memory delegation (NOT cross-session). **Pros**: industry-standard term. **Cons**: **different problem** — agent A hands a task to agent B in the same session; we want session A to hand state to session B across time. Surfaced here only to disambiguate the term "handoff" in 2026 industry usage.

## Decision

**Build `dev-workflow:handoff` v0.1** as a single **pure-prompt-template skill** with two modes — **prepare** (SKILL.md instructs agent to gather git state + tool inventory via Bash and author `.claude/handoffs/<timestamp>-<slug>.md` directly via Write tool, with 10-block schema) and **resume** (SKILL.md instructs agent to Read latest `.claude/handoffs/*.md`, run each Verification Command via Bash, surface mismatches, refuse to continue if state drifted). Zero production Python at v0.1 — agent does all I/O via native Bash/Write/Read tools (lesson from recap v0.1 sister skill; same 4-task SDD shape). Voluntary-only at v0.1 (forced-handoff Interruption Snapshot defers to v0.2). Schema trimmed from research v1.1's 14 → 10 by deferring heavy sections (How to Think About / Patterns / Architecture overview). Ships its own self-contained `dev-workflow/skills/handoff/references/handoff-schema.md` per Anthropic skill-independence convention — NO file sharing with recap. The bundle contains: 10-block L2 schema (Frontmatter / Situation / Background / All user messages verbatim / Recent decisions + WHY / Pending / Critical files / Do Not Touch / Verification / Confidence Flags) + 4 shared 共通核心原則 (structured-schema / quote-not-paraphrase / all-user-messages / synthesis-check, with L2-tailored definitions) + 5th L2-specific principle `technical-precision` (mirror flip of recap's L3-specific `plain-language`) + good/bad-example pair. File path: `.claude/handoffs/YYYY-MM-DD-HHMMSS-<slug>.md` — adopts the Claude Code community de-facto convention shared by 5+ plugins (softaworks/agent-toolkit, ykdojo/claude-code-tips `dx:handoff`, qdhenry/Claude-Command-Suite, robertguss/claude-code-toolkit, willseltzer/claude-handoff). Same-folder sharing means multi-plugin handoff produces ARE cross-readable (different plugins write different timestamp files; folder shared). Verified 2026-05-28: NO Claude Code built-in `/handoff` exists (CHANGELOG only has "external-editor handoff Ctrl+G", an unrelated UI feature) and `.claude/handoffs/` is NOT a built-in convention — it's community-de-facto, which is exactly why adopting it is the integration choice rather than a built-in collision risk. Tri-language READMEs (EN/JA/zh-TW) like recap. Plugin manifest bump (`dev-workflow` 2.10.0 → 2.11.0) + `handoff` keyword. NO per-tool init-prompt variants at v0.1 (Codex / Cursor / Gemini deferred to v0.2 once Claude Code dogfooded). NO chain rollup script (frontmatter pointer only; multi-handoff history walks via filesystem ls + grep).

The plugin placement (`dev-workflow:`) mirrors recap; if/when forced-handoff + per-tool variants ship, evaluate whether to graduate the L1-L4 family to a dedicated `continuation-stack:` plugin per the synthesis research.

**Audience-aware design (lesson from recap, applied with skill independence)**: This skill's READER is a cold AI agent (L2), not a human (L3). Two mirror flips from recap: (1) where recap dropped Block 4 verbatim user messages at L3 (human remembers their own turns), HANDOFF KEEPS verbatim user messages as MANDATORY at L2 (cold AI has no other source); (2) where recap shipped `plain-language` as 5th L3-specific principle, HANDOFF ships `technical-precision` as 5th L2-specific principle (exact identifiers / full paths / complete error messages preferred over plain-language, because the AI reader is a tool that benefits from precision). Each skill's bundle encodes its own audience assumptions in its own file; no cross-skill file dependency.
