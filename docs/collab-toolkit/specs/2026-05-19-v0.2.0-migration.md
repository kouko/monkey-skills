# collab-toolkit v0.2.0 — Migrate from agent-browser to official MCP/CLI

**Status**: Brief — ACCEPTED 2026-05-19 → handoff to `writing-plans`
**Date**: 2026-05-19
**Owner**: kouko
**Predecessor**: v0.1.6 (`docs/superpowers/specs/2026-05-15-collab-toolkit-v0.1.0-design.md`)

## Problem

**Job-To-Be-Done** (Klement format):
> When I'm in Claude Code wanting visibility into work-in-flight or to search across my office tools, I want to ask "list Asana tasks due this week / search Slack for OKR" in natural language and get structured results, so I don't context-switch to 5 different browser tabs and don't maintain UI-snapshot scripts that break on each vendor's UI redesign.

**Pain with current agent-browser approach**:
- UI-snapshot pattern fragile to vendor reskin (v0.1.6 text-snapshot rewrite was a symptom — root cause is fighting browser UI as an interface)
- Per-locale label tables (EN / zh-TW / ja) manually maintained in every protocol — `## Localized labels` pollutes 20 protocol files
- Login coupling: shared mode dies on Chrome restart; dedicated mode forces a separate Chrome session + Chrome-for-Testing install
- "alpha-y" maintenance burden across 5 different vendor sites with independent UI changes

**Why now (2026)**:
- Asana official MCP V2 GA; V1 deprecated 2026-05-11
- Slack official MCP GA Feb 2026; Real-Time Search API live
- Notion remote MCP at `mcp.notion.com/mcp` Active since 2026-03-30
- Google released official Workspace CLI (`gws`) covering Gmail / GCal / Drive / Sheets + 40+ skills + MCP bundle
- Claude Code native OAuth pre-registration shipped — `/mcp add` becomes one-click for compliant servers

The vendor-side gap that justified agent-browser in v0.1.0 has closed across all 5 services.

## Decision

**Per-service official channel** (locked w/ user 2026-05-19):

| Service | Channel | Setup step |
|---|---|---|
| Asana | Official MCP V2 (`mcp.asana.com/v2/mcp`) | `/mcp add asana` — Claude Code native OAuth pre-reg |
| Slack | Official MCP (GA 2026-02) | `/mcp add slack` |
| Notion | Official remote MCP (`mcp.notion.com/mcp`) — **NOT** `ntn` CLI | `/mcp add notion` |
| Gmail | Google Workspace CLI `gws` | `gws auth` (shared OAuth w/ GCal) |
| GCal | Google Workspace CLI `gws` (same binary) | (same OAuth) |

**Will build**: 5 SKILL.md + 20 protocol rewrites; new `/collab-setup`; full agent-browser retirement.

**Will NOT build**: writes, `ntn` CLI integration, Cowork support, fallback layer, new protocols beyond v0.1.6 set.

## Alternatives Considered

Bilingual WebSearch (EN + JA) per Axis 4 protocol.

**Approach 1: Pragmatic mixed (CHOSEN)** — `gws` CLI for Google × 2 + official MCP for 3
- Pros: All-official; minimum steps (1 binary install + 1 Google OAuth + 3× `/mcp add`); single Google OAuth covers Gmail + GCal
- Cons: Two invocation models (CLI vs MCP); `gws` requires Google Cloud project ID; OAuth scope 25-item limit for unverified apps
- Sources: [Asana V2 MCP Connecting Coding Clients (EN)](https://developers.asana.com/docs/connecting-mcp-clients-to-asanas-v2-server) / [Google Workspace CLI 導入ガイド Qiita (JA)](https://qiita.com/supertask/items/5a37cda144b5c0f964dc)

**Approach 2: All-MCP uniformity** — even Google via `gmailmcp.googleapis.com` + `calendarmcp.googleapis.com`
- Pros: Single invocation model; future-proof if MCP dominates; Cowork-portable
- Cons: Each Google MCP needs Google Cloud project + per-API enablement + OAuth client config; loses 40+ skills bundled with `gws`; **higher setup friction**
- Rejected: Fails setup-simplicity criterion
- Source: [Configure Google Workspace MCP servers (EN)](https://developers.google.com/workspace/guides/configure-mcp-servers)

**Approach 3: Notion `ntn` CLI + MCP for others**
- Pros: `ntn` is officially Notion; uniform "CLI where possible" framing
- Cons: Uses Integration Token (manual paste + **per-page connect** — Notion API limitation, not CLI's fault); Notion themselves call it *"alpha-y so auth is a little wonky"*; designed for Workers deployment + cron scripts, not read-only office collab
- Rejected: Setup-simplicity fails (per-page connect ×N pages); `mcp.notion.com` OAuth materially simpler
- Sources: [Notion 3.5 Developer Platform release (EN)](https://www.notion.com/releases/2026-05-13) / [Sakasegawa 観察文 (EN)](https://nyosegawa.com/en/posts/notion-cli-for-coding-agent/)

**Approach 4: Keep agent-browser + add MCP as parallel layer**
- Pros: Risk-mitigation fallback
- Cons: Doubles maintenance; "rarely-used fallback" bit-rots; user explicitly chose full retirement
- Rejected: User decision (option A)

**JA-search migration risk findings**:
- MCP OAuth implementation priority per 2026-04 official doc: pre-registration > CIMD > DCR. Asana V2 uses pre-reg ✓ — [MCP OAuth 仕様通り実装が難しい3選 Zenn (JA)](https://zenn.dev/ncdc/articles/40a7a51a2a016d)
- `gws` browser-flow timeout — terminal `connection refused` if user too slow; document in setup UX — [gog 導入ガイド Zenn (JA)](https://zenn.dev/takna/articles/gog-cli-setup-guide)
- MCP confused-deputy risk: child agents inheriting parent token; surface in failure-modes — [MCP 認証実装事故 Zenn (JA)](https://zenn.dev/luoxi/articles/mcp-auth-oauth-from-scratch)

## Open Questions

1. **`gws` install path**: **DECIDED 2026-05-19 — brew preferred, npm fallback.** Setup.md walks brew first; if formula unavailable on user's `brew search`, fall back to `npm i -g`. Plan must verify brew formula availability before committing copy.
2. **Google Cloud project**: must kouko create a personal GCP project (one-time), or does `gws` ship with a default? → `gws auth` dry-run during plan stage
3. **Asana MCP scope coverage**: does default OAuth scope cover read-tasks / read-projects / read-portfolios / search? Need to verify against v0.1.6 hero protocols
4. **Slack MCP `find-user`**: does it support display-name lookup that v0.1.6 protocol relies on?
5. **Notion MCP `page-backlinks`**: Notion API has no native backlinks endpoint; v0.1.6 used UI scraping. **Likely need to drop this protocol** or document limitation; surface in plan
6. **Migration cadence**: **DECIDED 2026-05-19 — single atomic v0.2.0 PR** (all 5 services). Rationale: services are conceptually homogeneous (all "migrate to official"); CHANGELOG remains single coherent entry; cross-skill consistency easier to enforce in one review pass. Risk-mitigation: SDD's per-task review provides the slicing inside the PR.
