## Verdict (one line)

Loop *semantics* (numeric caps, exhaustion→escalate-to-human, upstream-not-cap fixes) match or beat industry norms and the self-correction literature; the deficit is **where the counter lives** — industry runtimes enforce caps as real exceptions while loom's interactive mode counts rounds in prose — plus one universal-principle violation: batch mode's `RUNNING` orphan can silently drop work while the batch reports done.

## 2. Gaps (ranked) × industry comparison

| 1 | **Batch `RUNNING` orphan → false success.** No code path transitions RUNNING without the dispatcher calling `mark`; a killed session leaves the entry skipped by every future `next` (`batch_queue.py:656-658`), invisible to the circuit breaker (counts only DONE/FAILED, `:546`), while the batch prints `{"done": true}`. ## 3. Designs the comparison *validates*

1. **The 3-round cap number is evidence-backed.** Self-Refine (arXiv:2303.17651): gains front-loaded in rounds 1–2, capped experiments at 4. Huang et al. (arXiv:2310.01798, ICLR'24): intrinsic self-correction without external feedback is unreliable and can regress. CRITIC (arXiv:2305.11738): reliable self-correction requires external grounding (tests, tools). Aider independently hardcodes 3. loom's 3 rounds *with* dual external reviewers sits in the supported zone.
2. **git-guard matches the industry's one hard rule and is stricter than most.** Devin documents a pre-push identity hook; Copilot coding agent is restricted to `copilot/*` branches and cannot approve/merge its own PR. "Agent always goes through a PR gate; never trust self-certification" is cross-industry consensus. loom's HEAD-sha/patch-id match + single-use fail-closed waiver is a strict implementation of the same shape.
3. **Seg1/seg2 "flag and proceed" degradation has industry precedent.** CrewAI `max_iter` exhaustion forces the best-available answer (graceful degradation, no exception); ADK `escalate` is a typed event. The soft-upstream/hard-downstream gradient is a defensible design choice — contingent on a human actually reading the ledger's Intervention section, which nothing currently guarantees.

4. **(round 3) The Gap-5 remediation pattern already exists in-repo beyond the pipeline drivers.** `.claude/workflows/principles-improve-loop.js` implements a hard `ROUND_CAP = 3` (`:428`), **schema-forced JSON verdicts** via a real CLI (`loom-product-principles/scripts/improve_loop_verdict.py`, exit codes consumed as `exitCode` fields, `:499-611`), a **fail-closed plateau brake** (courier failure ⇒ stop, never continue, `:606-610`), and two-stage accept + held-out smoke against oracle-overfitting. Gap 5's fix is a port of an existing in-house pattern, not new invention.

Also: **semantic no-progress detection is industry-wide unsolved.** No surveyed framework ships it; OpenHands' action-hash detector is the lone product attempt and has documented false positives (issue #5355). A documented incident: turn-cap-only agents looped 11 days, ~$47K. loom is not behind here — but it argues for cost/wall-clock hard budgets, which are the industry's actual defense.

## 4. Recommendations (minimal, industry-shaped; decisions pending)

1. **Gap 1** — heartbeat timestamp on RUNNING entries (written on cadence, not just at start) + independent watchdog scan force-failing stale entries + batch `done` derived from `terminal_count == total` + keep existing breaker. This quartet is the Airflow/Temporal/Prefect common denominator.
2. **Gaps 2/7** — write RALLY_CAP=2 semantics into the interactive-mode SKILL.md flows (few lines; qualifies as cheap hardening). Longer term: the #585 mechanize-enforcement arc is already the right direction.
3. **Gap 4** — add a step or wall-clock hard budget to systematic-debugging; keep the 2-falsifications→WebSearch brake as the qualitative layer.
4. **Gap 5** — schema-constrain interactive-mode verdicts (Workflow `schema` pattern already proven in-repo by the pipeline drivers).
5. **Gap 3** — wire `validate_discovery_artifacts.py` into both discovery SKILL.md flows, mirroring product-principles Step 8.
6. **Gap 6** — one kill-mid-segment → resume → assert-station-and-no-double-write test.
7. **(round 3) Gap 8** — state a NEEDS_CONTEXT re-dispatch cap in SDD (one line; same cheap-hardening class as rec 2).
8. **(round 3) Gap 9** — actually pass `budgets.run` into `checkRunFloor` (small code change), and document reliance on host Workflow caps.
9. **(round 3) Gap 10** — add a PreToolUse(Bash) git-guard entry to `.codex/hooks.json` and live-verify on Codex; longer term, consider a GitHub branch-protection backstop so the push gate is host-independent.
10. **(round 3) numeric drift** — reconcile continuous-mode's 2-round halt with SDD's 3-round cap (one-line cross-reference in whichever is intended).

### 4b. Red-team caveats on the proposed fix variants (round 3)

**Fix 1 as "lease-at-read" needs redesign before implementation:**
- No runId exists at RUNNING-write time — `_dispatch_entry` (`batch_queue.py:505-539`) records status/branch/worktree only; runId is written by `mark` *after* completion. A `mark-running --run-id` write step is required, and a crash window between RUNNING-write and runId-write remains undetectable.
- There is **no on-disk liveness oracle** for a Workflow runId readable by a stdlib Python CLI — "lease-at-read" is honestly a *timestamp-staleness heuristic* and must be named as such.
- A single start-timestamp threshold is unsafe: legitimate seg3 runtime is unbounded (sequential tasks × 30-min stations × ladder), so staleness must key on a **cadence heartbeat or per-station progress marker**, not dispatch time.
- Force-transition to FAILED without a `requeue` verb = permanent data loss on false positives; a requeue path must ship together with reconciliation.
- `save_state` (`batch_queue.py:154-172`) has a pre-existing read-modify-write race with no lock/CAS; reconciliation-on-read multiplies the window — add locking first.
- `status` must stay a pure query (its own docstring: first thing a fresh session reads); put reconciliation in `next` or a dedicated `reconcile` verb, and gate transitions on `status == RUNNING` for idempotency.

**Fix 4 as "verdict-file + validator" is schema-without-teeth unless three things ship together:**
- Bind the verdict to artifact state (`head_sha`/`patch_id` recomputed at check time) exactly as `gate-markers-spec.md:45-69` / `git-guard.py:325-344` do — schema alone re-creates the recorded stale-cross-branch-marker incident.
- Decide scoping + rotation explicitly (in-tree change-folder = branch-scoped, vs git-dir-shared + hash-bound; who overwrites/deletes between rounds).
- **Wiring is the teeth**: today nothing mechanically consumes a design→spec verdict (`design-critic/SKILL.md:249-257`), so a validated file that no next-station step invokes reproduces Gap 3's orphaned-validator pattern. Minimal real version: a `validate_critic_verdict.py` that exit-fails on schema/staleness + the next station's flow (or a hook) invoking it and refusing on NEEDS_REVISION.
- Residual either way: schema validates *shape*, not verdict correctness — a rubber-stamp PASS_WITH_NOTES stays undetectable by this layer (judge-reliability limits, §5 sources).

### 4c. Deep-dive designs resolving the 4b blockers (round 4, 2026-07-18)

Inputs: on-host archaeology of Workflow artifacts; claude-code-guide doc verification; requeue-verb + file-locking research; gate-marker port design study. Key facts established this round:

- **Workflow runs leave a per-run record at `~/.claude/projects/<project>/<session-id>/workflows/wf_<runId>.json`** containing `status` — 16 files observed on this machine, all terminal (`completed` 13 / `failed` 2 / `killed` 1); no in-progress file observed, so presence ⇒ definitively ended, absence ⇒ running OR hard-killed (ambiguous). Subagent transcripts (`<session>/subagents/agent-*.jsonl`) append during a run — mtime is a liveness proxy. **All of this is undocumented internals** (confirmed against code.claude.com/docs/en/workflows.md + sessions.md): usable only as an opportunistic layer that degrades gracefully, never as the sole oracle.
- Documented hook facts: PreToolUse exit 2 blocks with stderr fed to the model; structured `permissionDecision: deny` also documented. Whether a PreToolUse matcher can target the `Skill` tool is **undocumented — needs an empirical probe** before any design builds on it.
- Recovery-verb consensus (Airflow clear/mark-failed, Temporal reset/terminate, Celery revoke/acks_late): two verbs — **reset-to-queued** (attempt counter + audit line) and **force-fail** (no re-run + audit line); queue layer never promises idempotency (job author's duty, per Celery docs); never silently lose the audit trail of why a human intervened.
- Locking: the failure mode is lost-update, which atomic-rename alone does NOT fix; recipe = `fcntl.flock(LOCK_EX)` held across the full read-modify-write + `tempfile`+`os.replace()` for the write. macOS quirks: request LOCK_EX directly (shared→exclusive upgrade is not atomic on APFS); never open with a truncating mode before locking.

**Fix 1 revised design — "reconcile + two verbs" (implementable):**
1. `_dispatch_entry` additionally records `dispatched_at`; new subcommand `mark-running <id> --run-id <wf_...> --session-dir <path>` invoked by the dispatcher immediately after `Workflow()` returns (closes the no-runId-at-RUNNING-write blocker; the residual pre-`mark-running` crash window is covered by rule 3 below).
2. Reconciliation lives in `next` and a new explicit `reconcile` verb — **never in `status`** (stays a pure query, per its own contract). Idempotency: every forced transition gates on current `status == RUNNING`.
3. Auto-transition only on **definitive** evidence: recorded `wf_<runId>.json` exists with `failed`/`killed` → force-FAIL (counts toward breaker, audit line). `completed`-but-unmarked → flag `SUSPECT-COMPLETE`, human confirms via `mark` (workflow completion ≠ result adopted). Staleness alone (incl. missing runId beyond a short grace window) → loud SUSPECT listing only; the human decides via `reset` / `force-fail`. Rationale: legitimate runtime is unbounded (4b), so timers may only *report*, never kill.
4. New verbs `reset` (→QUEUED, `attempts+=1`, append-only per-entry `audit[]` line) and `force-fail` (→FAILED, audit line) — the industry-consensus minimal pair.
5. `done` verdict derived from `terminal_count == total`; RUNNING/SUSPECT entries enumerated loudly in the same output.
6. All subcommands wrap state I/O in `flock(LOCK_EX)` across read-modify-write + atomic replace (fixes the pre-existing lost-update race 4b found).
   Estimated one-time cost: 4-6 SDD tasks ≈ 1.5–2.5M tokens. Residual accepted: absence-of-record ambiguity for hard-killed hosts (human resolves via verbs); undocumented wf-record format may break on CC upgrades (reconcile must treat unreadable records as "no evidence", fail-safe to SUSPECT).

**Fix 4 revised design — in-tree, content-hash-bound verdict files (implementable):**
1. Location: **in-tree per change-folder** (`docs/loom/<change-id>/<critic>-verdict.json`) — NOT `.git/loom/` (single-slot design collides when many change-folders coexist; in-tree travels with the artifact through branches/archival).
2. Binding: sha256 over the exact artifact files covered (`DESIGN.md`+`ui-flows.md` / `proposal.md`+`specs/**`), computed at mint and re-checked at validate — content-identity for uncommitted drafts (no HEAD exists mid-flow); gate-markers' patch-id idea minus the merge-base machinery.
3. CLI mirrors `loom_gate_markers.py` (argparse skeleton + verdict-text schema checks reused): `mint`/`validate`; validate exits 0 = fresh PASS_WITH_NOTES, 2 = no verdict (critic never ran), 3 = fresh NEEDS_REVISION, 4 = stale hash. **Divergence from gate-markers: NEEDS_REVISION still mints** — the consumer must distinguish "ran and blocked" (exit 3) from "never ran" (exit 2); both stop, for different reported reasons.
4. Rotation: overwrite-in-place, latest-wins (round history stays in the critic's prose Round summary); lifecycle = the change-folder's.
5. Teeth (minimal consumer set, both extending an existing trust-the-exit-code idiom): `loom-spec/skills/spec-expansion/SKILL.md` §Consuming-a-ui-flows-seed gains a validate-before-fan-out step; `loom-code/skills/writing-plans/SKILL.md` §Who-runs-the-validator additionally requires critic-verdict validate exit 0 (structural-clean ≠ critic-fresh-and-passed).
   Estimated one-time cost: 2 scripts + 2 SKILL.md wirings + tests ≈ 0.8–1.5M tokens. Residual accepted (stated honestly): consumption is still prose-invoked — a weak executor can skip the paragraph; verdict files are forgeable by the same agent (content-hash ties to files, not to "a critic actually produced this"); correctness of the verdict remains ungated. Upgrade path if the `Skill`-tool PreToolUse matcher probe succeeds: hook-gate the `spec-expansion` invocation itself (would convert the consumer from prose to mechanical).

Open probes before implementation: (P1) PreToolUse matcher on `Skill` tool — empirical, ~minutes; (P2) whether an in-progress Workflow writes any `wf_*.json` (only terminal records observed; a live-run check settles the absence-ambiguity boundary); (P3) the pre-existing Codex Bash-hook probe (Gap 10).
