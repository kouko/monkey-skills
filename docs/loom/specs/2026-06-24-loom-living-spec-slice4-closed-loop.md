## Problem

(Axis 1 — JTBD) The loom spec pipeline is **single-cycle**: each feature starts from a
fresh seed. spec-expansion now *authors* a persistent intent layer (slice 3) and loom-code
generates an index (slice 1), but nothing tells the **next** change to *read* them — so a
second change to an existing capability re-derives everything from scratch instead of
building on / diffing against the current truth. The accumulating model exists but is
write-only.

Job: **"when spec-expansion starts a change against a capability that already has persisted
intent (a MID readme, TOP invariants/state-machines, an index of covered requirements),
make it READ that prior state as context and fan only NET-NEW behavior — so changes build
on current truth instead of restarting, without ever copying that truth into a second
place that drifts."**

## Users

(Axis 2) The spec author (human or agent) running spec-expansion on a change to an
*existing* capability — they have prior intent/requirements but currently no convention
that surfaces them. Job story: *When I expand a spec for a capability that already exists,
I want spec-expansion to point me at its persisted intent + already-covered requirements,
so I extend the model and fan net-new paths instead of re-specifying what's already there.*
(A net-new capability has no prior state — handled by the empty base case below.)

## Smallest End State

A new **doc-convention section** in `spec-expansion/SKILL.md`, e.g. **"Consuming the
persisted intent layer as prior-state"**, mirroring the existing §"Consuming a ui-flows.md
seed" template, carrying:
1. **Point-don't-copy + link-back** — spec-expansion **references** the persisted layers by
   path and links back; it **never copies** their content into the change-folder (a copy =
   a 2nd SSOT that drifts). This is B1's genuine point-don't-copy seam.
2. **A seam-mapping table** (like the ui-flows one) — which persisted prior-state feeds which
   of the three phases:
   | persisted prior-state | feeds |
   |---|---|
   | MID `<capability>/README.md` (intent / why / scope) | Phase ① seed-adequacy + USM backbone — the capability's existing intent frames the journey |
   | TOP `MODEL.md` `## Object state machines` | Phase ② OOUX object/state model — extend existing lifecycles, don't redefine |
   | TOP `MODEL.md` `## Invariants` | Phase ③ matrix guard-rule lenses — existing invariants are guards |
   | TOP `MODEL.md` `## Out of scope` | Phase ③ pruning — don't fan paths the system deliberately excludes |
   | the generated INDEX (capability→req→test) | the fan boundary — mark already-covered reqs; **fan NET-NEW only** (#406 semantics) |
3. **Empty base case (B1 `[audit2:F2]`)** — prior-state intake is **additive and may be
   empty**: a net-new capability (or a repo mid-adoption with no layer/index yet) reads
   whatever exists, possibly nothing; an empty/absent layer is **never authoritative** and
   produces no cold-start deadlock. Mirror the ui-flows "if no … exists, ignore this section
   and treat the input as a generic seed" stance.
4. **A behavioral test** (in `test_spec_expansion_skill.py`) asserting the section's
   load-bearing elements (point-don't-copy/link-back, the seam-mapping rows, the empty base
   case).

## Decision

Build **F = a prior-state reading convention** in `spec-expansion/SKILL.md`: a new section
mirroring the ui-flows seam — point-don't-copy + link-back, the phase-feed seam-mapping
table above, and the empty base case — plus a behavioral test. spec-expansion reads the
persisted TOP/MID + the index as **prior-state context** and fans **net-new only**; it never
copies them in.

- **INDEX reference is "when present"** — the committed index file + its repo location is
  capstone-G scope, so F's convention references it as read-when-present, which the empty
  base case already covers (no dependency on G; TOP/MID from slice 3 are concrete now).
- **No machine merge / parser** — this is a READING convention (point-don't-copy), not an
  auto-application of prior-state. Industry grounding inherited from the parent brief (Kiro
  reads prior state via git-ref; OpenSpec build-on-prior; the in-repo ui-flows seam is the
  proven template) — no new research needed.

## Alternatives Considered (Axis 4)

1. **Mirror the ui-flows seam as a new doc-convention section** (chosen) — proven in-repo
   template; point-don't-copy keeps the SSOT single; additive + empty-base-case safe.
2. **A machine prior-state merger/parser** (rejected) — out of scope: builds a second
   apply-path that can diverge from the persisted truth; the brief's #6 calls B1 a *pointing*
   seam, not a merge. Defer any automation until a real need appears.
3. **Fold prior-state into Phase ①'s seed-adequacy pre-flight inline** (rejected) — the
   ui-flows precedent is a standalone top-level section (`SKILL.md:53`); a sibling section is
   discoverable + symmetric, and the pre-flight just references it.

**My take:** Recommend #1. **Conditional reversal:** if real usage shows authors routinely
miss the prior state, escalate to a lightweight pre-flight checklist item (still
point-don't-copy), not a parser.

## What Becomes Obsolete (Axis 5)

Nothing is deleted — additive. Conceptually the "single-cycle pipeline" framing is retired
(the loop is now closed), but there is no code/doc enforcing single-cycle to remove. The
parent brief's "missing back half" is now closed end-to-end (E-core persisted it; F reads it).

## Out of Scope

- active/deferred requirement **status** — capstone G.
- **repo-wide index wiring** + the committed `docs/loom/INDEX.md` location + pre-merge
  required checks — capstone G (F references the index "when present").
- Any **machine merge / parser / auto-application** of prior-state — F is a point-don't-copy
  reading convention only.
- Editing the persisted layer from spec-expansion's read path (read-only; authoring is
  slice-3's §"Authoring the persistent intent layer").
