## Decision

**Recommend: E2 (light snapshot rewrite).** Reject E3 (A-shard, user's tentative plan) and E1 (delete entirely).

| End state | Pros | Cons | Verdict |
|---|---|---|---|
| **E0 — do nothing** (keep stale banner + 636-line accumulated content) | Zero work | Banner says "may drift" — user has no signal whether content is current; ugly when opened | ❌ — banner makes a promise the file can't keep |
| **E1 — delete `wiki/index.md` entirely** | Smallest, most honest | User said "偶爾掃視" — micro-but-nonzero snapshot need; would force them to mental-count from folder pane | ❌ — over-corrects |
| **E2 — auto-gen ≤30-line snapshot** (RECOMMEND) | Solves real JTBD (overview); ingest-time auto-sync (never stale); LLM ignores; Obsidian renders natively; ~50 LOC delta net negative (banner-prepend logic replaced + simpler) | Need to fix wiki-ingest STEP 4e + wiki-setup Step 3c | ✅ |
| **E3 — A-shard (`wiki/<type>/_index.md`)** (USER'S TENTATIVE) | Per-category browsing surface | Solves nav-by-category — but user navs via `/wiki-query`, not manual browse; adds 6× shard files + count-sync logic to wiki-ingest; Obsidian file explorer ALREADY shows per-folder file list; net ADDS maintenance surface to fix a JTBD user doesn't have | ❌ — solves wrong job |

**Why E2 over E3 specifically**:

1. **Obsidian's file pane is already the per-type `_index.md`** — clicking `entities/` shows all 44 names. Adding `_index.md` shards duplicates what Obsidian renders for free.
2. **User's nav workflow is `/wiki-query`, not markdown browsing** — confirmed in 2026-05-20 interview. The A-shard's value proposition (per-category navigation entry) is for a workflow kouko doesn't have.
3. **A-shard ADDS code paths to wiki-ingest** — STEP 4e would need: detect target type, update target shard, recompute top-level counts, ensure idempotency across 6+1 files. E2 needs: overwrite 1 file with 1 template. Maintenance surface ratio ≈ 7:1.
4. **Industry consensus aligns**: 2025/2026 trend is auto-gen via Bases/Dataview/Zoottelkeeper for live views, OR curated MOC for high-value entries; the "manually maintained exhaustive index per category" pattern is being retired. E2 is the auto-gen path adapted to pure-markdown toolchain (no plugin dep).

**Why E2 over E1**: User confirmed weak-but-nonzero snapshot need. ≤30 lines of always-fresh overview costs nothing to maintain and gives positive value (vault size at a glance, growth rate visible week-over-week). E1 over-corrects.

## Open Questions

1. **Should `## Source` for journal pages be shown in snapshot?** Journal has 2 entries — too few to matter; default to type-row only.
2. **"Latest add" definition** — newest mtime, or newest `created:` frontmatter? Mtime is easier (no frontmatter parse); created could lag if user manually backdates. Recommend: mtime (matches how `wiki-ingest` already detects new pages).
3. **Top-level `Total` row** — show or hide? Recommend: show, weak-positive signal of vault growth ("we crossed 300!").
4. **Migration mechanic** — wait for next natural `wiki-ingest` run to overwrite, or trigger manually post-merge? Recommend: natural run (no urgency; banner mode is safe).
