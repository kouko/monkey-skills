## Problem

**JTBD (Klement format)**: *When I dogfood distill-sessions and get 4 proposal files with 33 Memory Items mostly in §pending bucket, I want a single human-readable advisory report I can skim in 60 seconds — telling me (a) what I keep doing wrong, (b) what to edit in which SKILL.md, (c) what to add to CLAUDE.md, (d) whether any new-skill candidate emerged — so I don't have to manually read 33 items × 4 files and synthesize the recurring patterns in my head.*

The v0.4 first-dogfood validated v0.4 architecture (11/11 coverage, 553K case handled) but surfaced a UX gap: 4 proposal files × 51-172 lines each, all 33 items stuck in §pending bucket, are formatted for **apply.py machine consumption** not human review. The user needed me to hand-synthesize the advisory narrative — that synthesis should be automated.

## Users

Primary: **kouko after a distill-sessions dogfood run**, wanting a 60-second triage decision (which items to manually edit into SKILL.md / CLAUDE.md, which to ignore, whether any pattern justifies a new skill).

Secondary user-experience constraint discovered in this session: first version of advisory report was "有點難懂" (too technical: Memory Item terminology, KEEP/CONSIDER labels, session IDs). Second version ("Claude 用法檢討") was 好多了 ("much better"): plain Chinese, anti-pattern → concrete改法 mapping, dropped jargon. The plain-language register is part of the success criteria, not optional polish.

## Smallest End State

**One paragraph**: After v0.4.1 ships, running `python scripts/main.py ...` produces (a) the existing 4 proposal files (unchanged for apply.py compatibility) AND (b) a new advisory report at `docs/skill-mining/<date>-advisory-report.md` matching the exemplar shape — 「你最常重複的 N 個小毛病」, 「該改的 N 個 skill (共 M 處)」, 「該加進 CLAUDE.md 的 N 條規則」, 「新 skill 候選」, 「數字摘要」, 「你現在能做的事」sections. The report is generated by a new sibling script `scripts/report.py` that consumes the same `merged.json` Stage 4 output as `propose.py`, applies heuristic clustering (title-keyword cross-target detection for CLAUDE.md candidates), and renders the markdown.

## Decision

**Build `dev-workflow:distill-sessions` v0.4.1** adding `scripts/report.py` — a Stage 5c sibling to `propose.py` that consumes the same `merged.json` and renders a human-readable advisory report to `docs/skill-mining/<date>-advisory-report.md`. Plus title-keyword heuristic clustering for CLAUDE.md candidate detection (cross-target same-keyword surfacing). SKILL.md gains §Pipeline §"Step 5 — Stage 5c report.py" + version bump 0.4.0→0.4.1. Plugin bump 2.8.0→2.8.1.

**Lock**: L1 (post-processing report.py) + sliver of L2 (cross-target keyword heuristic for CLAUDE.md candidates). L3 (LLM-driven cross-pattern subagent) deferred to v1.0 broad-scope sibling. All four heuristics in v0.4.1:

1. **Top N anti-patterns** — group items by shared keyword tokens in title (≥2 items sharing 2+ tokens after stop-word strip = same pattern)
2. **SKILL.md modifications** — per-target render: bullet per item, 1-line summary derived from description
3. **CLAUDE.md candidates** — items whose title keywords appear across ≥2 target skills → flag (precision over recall; false-positives acceptable since user reviews)
4. **新 skill 候選** — placeholder: "0 — v1.0 broad-scope sibling territory" (truly automated detection needs LLM cross-pattern analysis)
5. **數字摘要** — read from merged.json structure: trajectory count, items count, target distribution, cost-estimate (re-derive from payload-size sum)

### Locked Q-decisions for v0.4.1

**Q-v0.4.1-1 — Report file location: `docs/skill-mining/` or `/tmp/`?**
- A. `/tmp/<run-id>/advisory-report.md` (ephemeral; matches current /tmp/v0.4-proposals/ pattern)
- **B. `docs/skill-mining/<date>-advisory-report.md` (chosen)** — same parent as proposals; survives `/tmp` clean; matches SKILL.md L8 description of `docs/skill-mining/<date>-<target>-proposals.md` convention
- C. Output path arg with `docs/skill-mining/` as default

**Why B**: matches existing SKILL.md output-location convention (proposal files default to `docs/skill-mining/`); persistent survival; operator can `git ignore` if wanted (current `docs/skill-mining/` is untracked anyway).

**Q-v0.4.1-2 — Output language: zh-TW only / EN only / both / detect from skill?**
- A. EN-only (matches Trace2Skill prompt language)
- **B. zh-TW only (chosen)** — matches user-confirmed exemplar; one-language render is simpler; user's working language per global CLAUDE.md
- C. Both (parallel sections) — over-engineering for one user
- D. Detect from facets or user preference — over-engineering

**Why B**: exemplar locks zh-TW; user's working-language preference; simplest render; one user.

**Q-v0.4.1-3 — Heuristic clustering: strict (≥2 keyword overlap) or loose (any keyword)?**
- A. Strict — 2+ token overlap after stop-word strip; high precision low recall
- **B. Loose with stop-word strip (chosen)** — 1+ non-stop-word keyword overlap; higher recall; false-positives surface but user reviews anyway
- C. Configurable via flag

**Why B**: Stage 4 already filtered N=1 items to §pending bucket; the advisory report's job is to lift the bucket-stuck patterns; high recall serves that goal. False-positives cost user 5 seconds of skim per misfire.

**Q-v0.4.1-4 — Generate alongside propose.py or as separate command?**
- A. `main.py` invokes both propose.py + report.py automatically
- **B. Separate command `python report.py --input merged.json --output ...` (chosen)** — same shape as propose.py; user runs at their discretion; can re-run report.py with different heuristics without re-running expensive Stage 3
- C. Add `--report` flag to propose.py — couples two unrelated outputs

**Why B**: report.py is post-processing; doesn't need to fire on every main.py run. Loose coupling.

**Q-v0.4.1-5 — Test strategy?**
- A. Unit tests against fixture merged.json (the one this session produced lives at `/tmp/v0.4-dispatch/merged.json` but `/tmp` is ephemeral)
- **B. Commit a snapshot fixture (chosen)** — bundle `tests/fixtures/v0.4-merged-snapshot.json` based on this session's real run; assert output structure (section presence, item counts) not byte-identity (template prose might evolve)
- C. Pure heuristic unit tests on `cluster_by_keyword` / `extract_anti_patterns` etc. without integration test

**Why B**: real-data snapshot grounds the integration test; structural assertions (not byte-identity) survive prose-template tweaks; matches existing test pattern (test_propose.py uses fixtures).
